## Deploy Tooling

All Firebase projects in this GitHub account use a standardized 1Password-backed deployment system. The default source credential — interactive and CI/headless — is the per-project Firebase-vault SA key (`op://Firebase/{project-id} — Firebase Deployer SA Key`), resolved via the canonical precedence in `DEPLOYMENT.md` § [Deploy credential precedence (canonical)](DEPLOYMENT.md#deploy-credential-precedence-canonical). The shared 1Password ADC remains a fallback for projects that haven't provisioned the SA key. Service-account impersonation is still used when the resolved source credential is the shared ADC; when it's the project SA key directly, no impersonation wrapper is needed.

### Scripts

| Script | Canonical source | Installed to | Purpose |
|---|---|---|---|
| `gcloud` | `scripts/gcloud/gcloud` in this repo | `~/.local/bin/` | Local wrapper that refreshes access tokens from a 1Password-backed or explicit source credential for normal `gcloud` commands |
| `op-firebase-deploy` | `scripts/firebase/op-firebase-deploy` in this repo | `~/.local/bin/` | Deploy any Firebase project. Uses the project Firebase-vault SA key directly when present; falls back to short-lived impersonation when only the shared 1Password ADC is available. See `DEPLOYMENT.md` § Deploy credential precedence (canonical) |
| `op-firebase-setup` | `scripts/firebase/op-firebase-setup` in this repo | `~/.local/bin/` | One-time setup: creates deployer service account, grants IAM roles, and configures impersonation as the fallback path for when the project SA key isn't yet provisioned |

Install on a new machine:
```bash
cp scripts/gcloud/gcloud ~/.local/bin/gcloud
cp scripts/firebase/op-firebase-deploy ~/.local/bin/
cp scripts/firebase/op-firebase-setup ~/.local/bin/
chmod +x ~/.local/bin/gcloud ~/.local/bin/op-firebase-deploy ~/.local/bin/op-firebase-setup
```

### Credential Model

- Each project has a dedicated GCP service account: `firebase-deployer@{project-id}.iam.gserviceaccount.com`
- The default source credential is the per-project Firebase-vault SA key in 1Password — see `DEPLOYMENT.md` § [Deploy credential precedence (canonical)](DEPLOYMENT.md#deploy-credential-precedence-canonical) for the full 5-step resolution order and the impersonation-vs-direct trade-off
- Deploy preflight (`scripts/op-preflight.sh --mode deploy|all`) follows that same deploy order in Firebase repos: it caches the project SA key identified by `.firebaserc` before trying the shared human ADC, avoiding the recurring RAPT/reauth probe when the durable key is provisioned
- The 1Password-first deploy-auth model is a deliberate default for repos created from this template and should not be reversed without explicit human approval
- The local `gcloud` wrapper resolves source credentials through a narrower 3-step chain (`GOOGLE_APPLICATION_CREDENTIALS` → shared 1Password ADC → local ADC file) so normal `gcloud` commands work without routine browser login prompts. The wrapper does NOT consult per-project Firebase-vault SA keys; that step is specific to `op-firebase-deploy`
- No deploy service-account key is committed to a repo (the SA key lives in 1Password and is materialized only as a tempfile during a single deploy invocation)
- Runtime application secrets follow a two-lane 1Password model, reconciled with 1Password docs as of 2026-05-21: attended local agents should use 1Password Environments and the client-specific adapter that applies (Codex MCP where available, the 1Password local `.env` validation hook for supported non-Codex agents); CI/headless workflows should use explicitly scoped service-account tokens only after the relevant decision ticket approves the scope. Shell scripts may continue to use `op://` references, `op run`, and `op inject` where a generated runtime file is genuinely required; Secure Note `notesPlain` whole-file bootstrap is retired.

### Usage

```bash
# One-time setup: create the deployer service account + impersonation fallback
op-firebase-setup {project-id}

# Provision the project SA key into the 1Password Firebase vault (the
# default day-to-day deploy credential — see DEPLOYMENT.md
# § Provisioning the Firebase-vault SA key)

# Deploy (auto-detects project from .firebaserc)
op-firebase-deploy

# Deploy explicit project
op-firebase-deploy {project-id}

# Deploy specific targets
op-firebase-deploy --only hosting
op-firebase-deploy {project-id} --only functions

# Optional: activate the project-specific gcloud config
gcloud config configurations activate {project-id}
```

If the shared `Private/GCP ADC` item does not exist yet, bootstrap it once with `gcloud auth application-default login`, then copy the resulting `application_default_credentials.json` into 1Password. After that one-time step, the normal flow returns to 1Password-backed auth.
