# Installable PWA: install prompt, iOS safe-area, and a manual Lighthouse-category audit (`src/components/InstallPrompt.tsx`)

`vite-plugin-pwa` (manifest + Workbox shell precache, ADR 0006 layer 1) and the `index.html` PWA meta (`viewport-fit=cover`, Apple meta, theme-color) were already wired pre-ticket. `index.html` also owns the pre-JavaScript boot surface: an inline animated three-dot status appears immediately while a newly deployed hashed bundle downloads or a service-worker update settles, then React replaces it when `createRoot` mounts. The inline styling is intentionalâ€”an external CSS asset cannot prevent the blank interval it is meant to coverâ€”and respects reduced-motion preferences. The installable-PWA story adds an install-prompt component that captures the browser's `beforeinstallprompt` event (Chromium/Android) and falls back to a manual "Add to Home Screen" hint on iOS (which never fires that event), fires the existing `install_pwa` GA4 event (catalogued by #38) when the native prompt is accepted or, for installs that skip this banner entirely (e.g. the browser's own address-bar/app-menu install UI), when `appinstalled` fires on its own â€” a ref-backed guard keeps the two paths from double-counting the same install â€” adds the `env(safe-area-inset-bottom)` CSS the new banner needs plus a body-class toggle that reserves `.app`'s bottom clearance only while the banner is actually on screen, and audits the PRD's "Lighthouse PWA + performance â‰Ą 90" metric against the actual build output since a scored live run was not obtainable in this environment (see "Lighthouse â‰Ą 90" below for exactly what that means and why). It is exercised by `src/components/w1-pwa.test.tsx` and `src/test/w0-loading-indications.test.tsx` (RTL-jsdom).

## Capturing `beforeinstallprompt` offers an in-app Install affordance; `install_pwa` fires on acceptance or a bare `appinstalled`, deduped either way

The captured event's single-use `prompt()` / `userChoice` pair is one path to `track('install_pwa')` (`analytics.ts`, unchanged by this ticket â€” the catalog entry already exists per #38); the window's `appinstalled` event, handled unconditionally whenever a listener is attached, is the other, catching installs that never touch this banner's button at all. Offering the prompt is not the same as installing it, and a single real install must count exactly once no matter which path (or both) fired.

- **Given** an install-eligible Chromium/Android browser **when** it fires `beforeinstallprompt` **then** `InstallPrompt` calls `preventDefault()` (suppressing the browser's own mini-infobar), renders an "Install" button, and â€” once tapped â€” fires `track('install_pwa')` only if the native `userChoice` resolves `'accepted'`. (Test: "captures beforeinstallprompt (suppressing the mini-infobar), shows Install, and fires install_pwa on acceptance".)
- **Given** the Install button is tapped **when** `userChoice` resolves `'dismissed'` instead **then** `track` is never called, and the banner still clears (the captured event is single-use either way). (Test: "does not fire install_pwa when the native prompt is dismissed".)
- **Given** no install signal has occurred yet, on a non-iOS browser **when** `InstallPrompt` mounts **then** it renders nothing â€” there is no native prompt to offer and no iOS hint to show. (Test: "renders nothing with no install signal (unsupported browser, not iOS)".)
- **Given** a Player installs through the browser's own UI (address-bar icon, Chromium app menu) rather than this banner's button, whether or not the banner happened to be showing **when** `appinstalled` fires with no prior `userChoice` resolution **then** `InstallPrompt` fires `track('install_pwa')` from that handler directly and hides itself, so the metric isn't undercounted just because the Player never touched the in-app button. (Tests: "fires install_pwa and hides the banner when appinstalled arrives without ever using this button (installed via the browser UI directly)" covers the banner-was-showing case end to end including the hide; "fires install_pwa when appinstalled arrives with no beforeinstallprompt signal at all" covers the no-signal case for the tracking call alone.)
- **Given** a Player already tapped Install and `userChoice` resolved `'accepted'` (which tracked once) **when** `appinstalled` subsequently fires for that same install, as it does in every real browser shortly after acceptance **then** a ref-backed guard (set by whichever path tracks first) stops the second `track('install_pwa')` call, so one install is never counted twice. (Test: "does not double-fire install_pwa when appinstalled follows a banner-accepted install".)

## iOS gets a manual "Add to Home Screen" hint; already-installed and dismissed states render nothing

iOS Safari has no `beforeinstallprompt` API at all, so the only way to guide an iOS Player toward installing is a static instructional hint (no programmatic install path exists to trigger from script).

- **Given** an iOS device (iPhone/iPad UA, including iPadOS 13+ reporting as `platform: 'MacIntel'` with `maxTouchPoints > 1`) **when** `InstallPrompt` mounts and no native prompt is available **then** it shows a "tap Share, then Add to Home Screen" hint instead of an Install button. (Test: "shows the iOS \"Add to Home Screen\" hint, since Safari never fires beforeinstallprompt".) It shares the same `dismiss()` path as the Chromium banner (persists to `localStorage` under `gcb.install.dismissedAt`, mirroring `ConsentNotice`) â€” verified by reading the implementation, not a second dedicated test.
- **Given** the app is already running standalone (`display-mode: standalone` or iOS's `navigator.standalone`) **when** `InstallPrompt` mounts, or the `appinstalled` event fires later (installed via a path other than this button, e.g. the browser's own address-bar icon) **then** it renders nothing / hides â€” the hide-on-mount branch is a single two-line check verified by code review rather than a dedicated test, while the hide-on-`appinstalled` branch is asserted directly by "fires install_pwa and hides the banner when appinstalled arrives without ever using this button" above.
- **Given** a device that dismissed the banner **when** `InstallPrompt` mounts again **then** it does not render, matching `ConsentNotice`'s persistence contract. (Test: "dismissing the banner hides it and persists the dismissal to localStorage".)

## Mount point and iOS safe-area CSS

`InstallPrompt` mounts at `src/main.tsx` alongside `ConsentNotice` â€” the same stable, non-frozen, outside-the-auth-gated-tree mount point (#17) â€” so a Player can install before ever signing in. `.install-prompt` (`src/index.css`) is fixed to the viewport bottom with `padding-bottom: calc(10px + env(safe-area-inset-bottom))`, so its Install/Not-now tap targets clear the iOS home indicator on notched devices, the same pattern `.tabs` and `.sheet` already use. This CSS-only claim is verified by code review, not an automated test â€” jsdom's layout engine is a no-op (same limitation noted in `specs/w0-app-shell.md` for `.tabs`), and `env()` resolution needs a real layout engine.

`.tabs` (the signed-in bottom nav, `src/components/TabBar.tsx`) is also `position: fixed; bottom: 0`, mounted deep inside the auth-gated tree while `.install-prompt` mounts as a `main.tsx`-level sibling outside it, so the two fixed-bottom bars have no built-in coordination and, before this rule, the higher-z-index banner silently covered the entire tab bar for any signed-in Player who had not yet dismissed or installed. `body:has(.tabs) .install-prompt { bottom: calc(61px + env(safe-area-inset-bottom)); }` (`src/index.css`) stacks the banner directly above the tab bar instead â€” 61px is `.tabs`' own border + padding + min-height (1 + 8 + 44 + 8), and `.tabs`' safe-area padding is additive beneath that mark, so the stacking holds on notched devices too; `.install-prompt` keeps its own full `env(safe-area-inset-bottom)` padding even while stacked (a deliberate size-budget trade-off, since `.tabs` already clears the same inset below it), which just costs a few extra px of empty bottom padding inside the banner on notched devices, not an overlap. Verified by rendering the real `src/index.css` + `src/theme/themes.css` against the exact DOM `TabBar.tsx` and `InstallPrompt.tsx` produce, in a mobile-viewport browser (screenshot + `getBoundingClientRect` â€” no overlap) â€” not an automated test, for the same jsdom-has-no-layout-engine reason as the paragraph above.

Stacking the banner above `.tabs` (or floating it alone when signed out) still leaves `.app`'s own scrollable content under-cleared: `.app`'s `padding-bottom: 112px` (`src/index.css`) is sized only for `.tabs`, so while the banner is up it is just another fixed bar on top, and the last row of a long list can sit underneath it until the Player dismisses or installs. Rather than growing `.app`'s base padding unconditionally (permanently wasting that space for every Player who has already dismissed or installed), `InstallPrompt` toggles an `install-prompt-visible` class on `<body>` in a `useEffect` keyed off the same visibility condition that decides whether it renders its banner markup at all (added/removed on every visibility change, and removed unconditionally on unmount); `body.install-prompt-visible .app { padding-bottom: 176px; }` (`src/index.css`) applies only while that class is present, adding roughly the banner's own height (~64px, covering its padding/border plus up to two wrapped lines of its copy on narrow phones) on top of the existing 112px. Unlike the `.tabs`-stacking rule above, the ~64px figure was not confirmed against a live rendered measurement (no screenshot / `getBoundingClientRect` pass) â€” it is reasoned from `.install-prompt`'s own rules (10px top padding + `10px + env(safe-area-inset-bottom)` bottom padding + 1px border, plus up to two wrapped lines of its 12px/1.4-line-height copy on a narrow phone, since the paragraph is a `flex: 1` item sharing the row with two non-wrapping buttons) rather than measured, so treat 176px as a deliberately generous, unverified estimate and confirm it against a real device or browser before relying on it precisely; jsdom has no layout engine, so it cannot be verified by an automated test either. What automated tests do cover is the toggle mechanism the CSS depends on: `src/components/w1-pwa.test.tsx` asserts `document.body.classList.contains('install-prompt-visible')` flips to `true` exactly when the banner is showing (both the Chromium and iOS variants) and back to `false` once it is dismissed, installed, or unmounted (Tests: "never sets install-prompt-visible with no install signal (unsupported browser, not iOS)", "sets install-prompt-visible while the Chromium banner is up, and clears it once dismissed", "sets install-prompt-visible for the iOS hint too", "clears install-prompt-visible once the banner is gone after an accepted install").

## Lighthouse PWA + performance â‰Ą 90 (PRD metric) â€” verified manually, not via a scored live run

A real Lighthouse run was attempted against a production build in this environment and did not produce a score; the audit below is a manual substitute against the same checks, with the attempt and its concrete failure documented rather than silently skipped, per this ticket's verification instructions.

**What was attempted:** `npm run build` â†’ `vite preview` (port 4173) â†’ `npx lighthouse` (v13.4.0, resolved live from the registry) against Playwright's already-installed Chromium (`chrome-mac-arm64`), across several flag permutations (`--headless=new` and legacy `--headless`, with and without `--disable-gpu`, `--no-sandbox`). Every run failed with `NO_FCP` ("The page did not paint any content") after the full 30s wait, including against a temporary `.env.local` (gitignored, deleted after this check â€” not part of this diff) with a well-formed-but-fake Firebase config, added only so `getAuth()` wouldn't throw `auth/invalid-api-key` synchronously and blank the whole app before render (confirmed via a scratch Playwright script that this exact error was crashing the shell with the repo's real, credential-less environment). With that fake config in place, the identical Playwright Chromium build **did** render the SignIn screen correctly when queried directly (`page.content()` showed the mounted `ConsentNotice` + `SignIn` DOM) â€” so the app itself paints; Lighthouse's own trace-based first-contentful-paint detection specifically did not register a paint event in this sandboxed headless session, a known class of issue in constrained/virtualized headless-Chrome environments. Separately, `--only-categories=pwa` itself errors as an "unrecognized category" on Lighthouse v13 â€” the scored "PWA" category the PRD's metric names was removed from Lighthouse's default config in v10 (2022) and is no longer computable by any current Lighthouse version, a tooling-version gap independent of this environment's sandboxing.

**What was verified instead** â€” the actual `dist/` build output against the checks that category (and Chrome DevTools' still-current "Installable" panel) used to grade:

- **Manifest + icons** â€” `dist/manifest.webmanifest` has `name`/`short_name`, `start_url: "/"`, `display: "standalone"`, `theme_color`/`background_color` both `#07060d`, and `icons` with a 192Ă—192 + 512Ă—512 PNG plus a `purpose: "maskable"` 512Ă—512 (Android adaptive icon). Pixel-verified via `sips`: `public/pwa-192.png` is exactly 192Ă—192 and `public/pwa-512.png` exactly 512Ă—512 (a size mismatch here is the most common real-world installability failure); `public/apple-touch-icon.png` is 180Ă—180 (the standard Apple size, linked via `<link rel="apple-touch-icon">`, not the manifest â€” Safari doesn't read the manifest for its icon).
- **Home-screen label â€” set per platform, not once (#364)** â€” the label under the installed icon comes from a *different* field on each platform, so the two are configured independently and must be changed together or deliberately apart. Android/Chrome reads the manifest's `short_name` (`vite.config.ts`), currently `Gay Bingo`; iOS/Safari reads `<meta name="apple-mobile-web-app-title">` (`index.html`), currently `Gay Cruise Bingo`, and ignores `short_name` â€” the same split already documented for `apple-touch-icon` above, where Safari likewise does not read the manifest. The manifest's `name` (`Gay Cruise Bingo`) is the roomier install-prompt/splash label and is deliberately not shortened. Both labels truncate at roughly 12 characters on a real home screen, which is why `short_name` is the shorter of the two: the constraint is what forces the split, so a future change that "unifies" these strings re-introduces the Android clipping #359/#360 traded away. The truncation point is conventional, not specified, and the rendered result is verifiable only on a physical device â€” not from `dist/` or CI.
- **SW registration** â€” at the time of this audit, `dist/index.html` had the plugin-injected `<script id="vite-plugin-pwa:register-sw" src="/registerSW.js">` and `dist/registerSW.js` registered `/sw.js` at `scope: '/'` on `window load`. Since #178 (`specs/app-update-reload-prompt.md`), registration lives in-bundle instead: `UpdatePrompt`'s `virtual:pwa-register/react` import makes `vite-plugin-pwa`'s `injectRegister: 'auto'` treat registration as app-owned, so no `registerSW.js` is emitted or injected and `useRegisterSW` registers `/sw.js` when `UpdatePrompt` mounts at `main.tsx`. The build still emits `dist/sw.js` + a Workbox runtime chunk (15 entries / ~833 KiB precached at audit time), with `navigateFallback: 'index.html'` configured (`vite.config.ts`) for offline navigations.
- **Runtime caching (proof media, #363)** â€” beyond the build-asset precache, the SW carries one Workbox `runtimeCaching` route: CacheFirst over proof-media download URLs (`PROOF_MEDIA_URL_PATTERN` in `src/data/proofMediaCache.ts`, imported by `vite.config.ts`), bounded by `maxEntries`/`maxAgeSeconds` + `purgeOnQuotaError` and accepting opaque responses (`statuses: [0, 200]` â€” Feed `<img>` loads are no-cors). The pattern deliberately matches only the immutable `proofs/` tree â€” never mutable avatars â€” and is anchored to the `firebasestorage.googleapis.com` origin start, per Workbox's cross-origin RegExp rule; the positive/negative URL matrix is `src/data/proofMediaCache.test.ts`. The upload-side half of the same fix (`cacheControl` stamped by `uploadProofMedia`) is specced in `specs/w2-proof-capture.md`.
- **theme-color + viewport** â€” `index.html`'s `<meta name="theme-color" content="#07060d">` matches the manifest's `theme_color` exactly; `<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover, maximum-scale=1">` has `width=device-width`/`initial-scale=1` (satisfies the has-viewport check) and `viewport-fit=cover` (needed for this ticket's own `env(safe-area-inset-*)` CSS to resolve non-zero). Related-but-out-of-scope observation: `maximum-scale=1` disables pinch-zoom, which a live Lighthouse run would flag under **Accessibility** (not the PWA/performance categories this ticket's metric targets) â€” pre-existing from `w0-app-shell` (#17), unchanged here.

**What remains genuinely unverified:** an actual numeric performance score (paint timing, TBT, CLS, bundle-weight scoring under real network/CPU throttling) and any real-device Lighthouse/PageSpeed run. Both need an environment where Chrome's compositor actually emits paint trace events (a real workstation, a non-sandboxed CI runner, or a hosted PageSpeed Insights run against a deployed URL) â€” none of which this environment provides. This is a manual follow-up, most naturally alongside `x-launch-checklist`'s real-device pass. **Out of scope**, separately: Firestore's `persistentLocalCache` (ADR 0006 layer 2) is `w0-offline-persistence` (#20), not this ticket â€” the Workbox shell precache above is layer 1 only. The later inline `index.html` boot indication changed only the pre-bundle paint â€” no `vite.config.ts` change â€” and the #363 proof-media route above is the only Workbox-configuration change since this audit; manifest and registration are untouched.
