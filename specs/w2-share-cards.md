# w2-share-cards — on-device BINGO + Leaderboard Share Cards, native share sheet, ADR 0005

A Share Card is a retina image a Player generates on their own device — for a BINGO/Blackout win or the Leaderboard — to drop into the group chat (ADR 0005). Before this ticket (issue #36), the only share affordance was `Celebration.tsx`'s "Share" button, which shared *text and a URL* (`navigator.share({ title, text, url })`, falling back to the clipboard), and the Leaderboard had no share affordance at all. This ticket replaces that with an on-device image — canvas-free DOM rasterization via `html-to-image` — handed to `navigator.share({ files })` so the native share sheet carries the picture itself into the chat, plus a matching secondary affordance on the Leaderboard. Per ADR 0005 this ships **BINGO + Leaderboard Share Cards only** (no per-square proof cards — a Proof already carries its own shareable media) and stays entirely **behind the auth wall**: no server render, no public URL.

## Carried over unchanged (verified, not rewritten)

- `track('share_click', { surface })` (`src/analytics.ts`) — the existing catalog entry. This ticket adds a second call site (`surface: 'leaderboard'`) alongside the existing `surface: 'celebration'`; `analytics.ts` itself is untouched (off-limits this wave — #33 owns it).
- `sortPlayers`/`comparePlayers` (`src/game/logic.ts`) and the pin computation's full-roster scope (`specs/w2-leaderboard.md`) — untouched. The Leaderboard Share Card's row list is built from `useLeaderboard()`'s already-ranked `players`, and its First to BINGO pin reuses the SAME `firstBingoUid` the live badge already computes — never a second, divergent calculation.

This section originally also listed `Board.tsx`'s `<Celebration kind={celebrate} onClose={...} />` render call as unchanged, on the reasoning that `Board.tsx` was off-limits this wave so `Celebration` had to resolve its own board data through second listeners rather than props. Those listeners were themselves the races Codex flagged post-review — the board cells in round 1 (§ Empty-card race below) and the resolved display name in round 2 (§ Stale-name race below) — so `Board.tsx` is no longer untouched, though the change there stays a single render call now carrying two data props (`cells`, `playerName`).

## The change

- **`src/components/ShareCard.tsx` (new)** — the on-device renderer. Two data-in/blob-out entry points:
  - `renderBingoShareCard({ kind, playerName, eventName, cells }): Promise<Blob>` — a BINGO/Blackout card: event name, big BINGO!/BLACKOUT title, Player name, the B-I-N-G-O header, and the Player's own 25-cell grid (free/marked cells styled distinctly, a `pending`-marked cell styled distinctly again — see § Pending squares below — cell text shown). Refuses to render anything but a real 25-cell board — see § Empty-card race below.
  - `renderLeaderboardShareCard({ eventName, rows }): Promise<Blob>` — a Leaderboard card: event name, a "LEADERBOARD" title, and the given rows (rank, name, bingo/square stats, a pinned "★ 1st BINGO" badge on whichever row is flagged `firstToBingo`). `rows` is already the exact list to render — shaping (which Players, how many, whether the pin needs to be force-included) is the caller's job, not the renderer's.
  - Both build a real, off-screen `<div class="share-card ...">` node with plain `document.createElement` calls (no second React root), mount it via a `.share-card-host` wrapper (real layout so html-to-image can measure it, but `position: fixed; left: -10000px` so it's never visible or part of normal flow), rasterize with `html-to-image`'s `toBlob(node, { pixelRatio: 3 })`, then tear the host down. Because the card is just another node under `document.body`, it automatically inherits whichever `[data-theme='…']` CSS variables `ThemeContext` currently has set on `<html>` — "themed via the CSS variables in play" comes for free from the cascade, not a second copy of the palette.
  - `shareCardBlob({ blob, filename, title, text, url })` — the shared native-share-sheet + fallback chain BOTH `Celebration.tsx` and `Leaderboard.tsx` call, so the degrade path lives once, not twice: hands the blob to `navigator.share({ files })` when `navigator.canShare({ files })` says yes; otherwise falls back, in order, through a text/URL `navigator.share`, the clipboard (`navigator.clipboard.writeText(url)`), then a direct file download (`<a download>` on an object URL). Never throws — every branch self-catches — so callers can await it without their own try/catch. `blob: null` (an on-device render failure, including the validity gate above) simply skips the file-share and download legs. A cancelled share is terminal on BOTH legs, but detected differently — see § Cancellation vs. failure below.
- **`src/components/Board.tsx`** — hands its own already-loaded `cells` AND its already-resolved display identity to `<Celebration>` as props (`<Celebration kind={celebrate} cells={cells} playerName={identityKnown ? displayName : null} onClose={...} />`). This single render call is all this ticket touches in a file otherwise off-limits this wave — see § Empty-card race and § Stale-name race below for why each prop exists.
- **`src/components/Celebration.tsx`** — eagerly pre-renders the BINGO/Blackout card at mount (§ Eager pre-render below) via `renderBingoShareCard` with the rejection caught into a cached `Promise<Blob | null>` (a render failure — including the validity gate rejecting a non-25-cell board — degrades gracefully to the text/URL share leg rather than throwing or sharing a bad image); the `share` handler awaits that cached promise and hands the blob to `shareCardBlob`. `track('share_click', { surface: 'celebration' })` fires from a `finally`, exactly once, on every path (image share, cancelled share, every fallback leg, or a render failure). Card content: the board `cells` and the resolved Player name come from Board.tsx's own props, not Celebration-local listeners (§ Empty-card race, § Stale-name race); only the Event name still resolves via `useEventDoc` — it is not identity (no per-user staleness to leak) and has the benign app-name fallback the "falls back to the app name" test pins. The Share button is disabled while `playerName` is null (identity not yet known).
- **`src/components/Leaderboard.tsx`** — a new "Share leaderboard" button (`.lb-actions`, above the existing filter row) calls `shareLeaderboard`, which shapes the current roster into `LeaderboardShareRow[]` via the new `buildShareStandings` helper, renders the card — reusing the warm-on-intent pre-render started by hover/focus/press when its inputs still match (§ Eager pre-render) — and hands it to `shareCardBlob`. `track('share_click', { surface: 'leaderboard' })` fires from a `finally`, mirroring Celebration's shape.
- **`src/index.css`** — two new blocks: `.lb-actions`/`.lb-actions .btn` (the Share button's own layout, distinct from `.lb-filters` so it doesn't read as a fourth filter), and `.share-card*`/`.share-card-host` (the off-screen card's own visual language — deliberately its own class family, not a reuse of `.grid`/`.cell`/`.row`, so this ticket's `index.css` ownership this wave can't collide with any other in-flight ticket's styling of the on-screen Board/Leaderboard). `.share-card-cell.pending` and `.share-card-title`'s ink-token fill are later additions — see § Pending squares and § Title contrast below.
- **`package.json`/`package-lock.json`** — adds `html-to-image` as a regular dependency (it ships in the app bundle, not a build-time-only tool).

## Design decisions

- **Plain DOM construction, not a second React root.** `html-to-image` needs one real, already-laid-out element to walk and inline computed styles from. Standing up a second `react-dom/client` root (or `react-dom/server`'s `renderToStaticMarkup`) just to produce one throwaway node would add React lifecycle/`act()` complexity (and, for the server-render path, a Node-oriented dependency this Vite SPA doesn't otherwise need) for no benefit over `document.createElement`. This also mirrors the rest of the app's own image work — `data/storage.ts`'s `downscaleImage` reaches for raw `<canvas>`/DOM APIs, not React, for its one-shot image transform.
- **The card inherits the live theme for free.** Because the off-screen host is mounted under `document.body` (not a detached, never-attached node), it sits inside the same `[data-theme='…']` cascade `ThemeContext.tsx` already applies to `<html>` (`src/theme/ThemeContext.tsx:62`). No explicit `ThemeId` needs to be threaded into `ShareCard.tsx` — the card simply uses `var(--primary)`/`var(--bg)`/etc. like every other themed surface, and it renders in whatever theme the Player is currently using.
- **The renderer takes shaped data; the caller does the shaping.** `ShareCard.tsx` knows nothing about `sortPlayers`, the presentational leaderboard filter, or how "top standings" is defined — it only knows how to lay out whatever `LeaderboardShareRow[]` it's given. `Leaderboard.tsx` owns `buildShareStandings` (top `MAX_SHARE_ROWS` by rank, 8, plus the First to BINGO Player appended if their rank falls outside that slice) because that's presentation policy specific to the Leaderboard, exactly like `matchesFilter` already is (`specs/w2-leaderboard.md`). This keeps the renderer reusable and keeps ranking/pin logic in one place (never duplicated into the renderer as a second, potentially-divergent calculation).
- **The pin can't silently fall off the card.** `buildShareStandings` appends the First to BINGO Player as an extra row when their rank is outside the top `MAX_SHARE_ROWS` — mirroring the live Leaderboard's own principle (`specs/w2-leaderboard.md`: bingo-count rank and earliest-`firstBingoAt` are independent signals, so the pin holder is not guaranteed to be a top-ranked Player). Without this, a fixed-size card could cut the honoured Player off entirely just because they stopped playing after their early win while others racked up more bingos later.
- **The Share Card always reflects the FULL roster, never the active filter.** `shareLeaderboard` reads `players` (the full, already-ranked list) and the same `firstBingoUid` the pin badge uses — not the filtered `visible` subset. Sharing while "Blackout" is selected still shares the real top standings, not just the Blackout achievers; this mirrors the pin's own full-roster scope (`specs/w2-leaderboard.md`) and avoids a confusing card whose contents silently shift with whatever filter a Player happened to have open.
- **A cancelled native file share does not cascade into more fallbacks.** `shareCardBlob` treats a caught error from `navigator.share({ files })` as `'cancelled'` and stops — it does not then also try a text share, the clipboard, or a download. A Player who explicitly dismissed the OS share sheet should not be handed a second, different share prompt (or a surprise clipboard write / file download) immediately after declining the first. The fallback chain (text → clipboard → download) is reserved for when file sharing wasn't *available* in the first place, not for when it was available and refused.
- **No per-square proof cards (ADR 0005).** Only `renderBingoShareCard` and `renderLeaderboardShareCard` exist; there is no third card kind, and `ProofSheet.tsx`/`ProofFeed.tsx` are untouched (off-limits this wave in any case).

## Empty-card race: `cells` is now a prop, not a second listener (Codex P2, PR #111 finding 1)

Post-review, Codex flagged that `Celebration.tsx` opened its OWN `useBoard(uid)` listener and read `board?.cells ?? []` for the card — a subscription that starts with `data: undefined` until its first snapshot arrives. `Board.tsx` already has the loaded Board by the time it ever renders `<Celebration>` (the `!board` early-return guarantees it), but Celebration's second listener does not inherit that guarantee — it starts fresh on mount. A Player who tapped Share fast enough to beat Celebration's own first snapshot rendered and shared a card with the free center only, or nothing at all: a ZERO-CELL card, not the honor-system win it claimed to depict.

**Fix.** `Board.tsx` hands its own `cells` — the exact array it just rendered the Board from — to `<Celebration>` as a `cells: Cell[]` prop; `Celebration.tsx` no longer calls `useBoard` at all. This removes the race structurally rather than papering over the timing: there is no second subscription left to lose. (Round 2 widened the same render call with the `playerName` identity prop — § Stale-name race below — so `Board.tsx`'s footprint is now that one `<Celebration kind cells playerName onClose />` call, still with no change to Board's own gating, hooks, or render logic.)

**Backstop.** `renderBingoShareCard` refuses to rasterize a board that is not exactly 25 cells (`dealBoard`'s own invariant in `game/logic.ts` — free center + 24 prompts), throwing instead. `Celebration.tsx` catches that rejection into the cached pre-render promise as `blob: null` and falls through to `shareCardBlob`'s text/URL fallback leg — so even a future bug that hands Celebration a malformed board degrades to a text share rather than ever producing or sharing an empty/partial image.

## Stale-name race: `playerName` is now a prop, not a second listener (Codex P2, PR #111 round 2 finding 1)

The identity twin of the empty-card race above, flagged by Codex in round 2. `Celebration.tsx` still ran its own `useMyPlayer(uid)` listener and resolved the card's name as `resolveDisplayName(player, user?.displayName)` — a subscription that starts `data: null` on mount, making the resolver fall back to the live AUTH name until the first snapshot arrives. An immediate Share tap therefore rendered and shared the STALE Google name for a returning Player with a saved custom name — the exact window in which `Board.tsx`'s `doMark` deliberately withholds the name from Tally markers (`displayName: identityKnown ? displayName : undefined`, Codex P2, PR #87) and the Moment broadcasts HOLD (PR #99 finding 1).

**Fix.** `Board.tsx` passes its own already-resolved identity down: `playerName={identityKnown ? displayName : null}` — the SAME single `resolveDisplayName(player, user?.displayName)` output the Tally marker, Proof attribution, and Moment broadcasts carry, gated by the SAME `identityKnown` tri-state. `Celebration.tsx` drops `useAuth`, `useMyPlayer`, and its own `resolveDisplayName` call entirely; there is no second identity resolution left to race. The card renders no avatar (`BingoShareCardData` has no photo field), so only the name is threaded.

**The `null` gate, and why it is reachable.** `playerName: null` means the saved identity is not yet KNOWN, and Celebration disables the Share button (and skips the pre-render) until it resolves — holding, like the Moment broadcasts, rather than ever stamping the auth fallback onto a card. In every online flow this gate is belt-and-braces: the player-row subscription resolves at app start, long before a Board can render and a win can be marked, so `identityKnown` is true by the time Celebration can open. The reachable-null window is the one Board.tsx already documents for `knownFirstBingoAt` (Codex P2, PR #75/#87/#99): an offline reload whose ADR 0006 persistent cache holds the board but not the player row — the celebration still opens (the win animation is local UI, never held), the Player can keep playing, and Share enables when the row resolves (on reconnect). Accepted residual: in that offline window the Share affordance stays disabled rather than degrading to an 'Anonymous' card or the stale auth name.

## Eager pre-render: rasterization starts before the tap (Codex P2, PR #111 round 2 finding 2)

`navigator.share` may only run under transient user activation, and the activation a tap grants EXPIRES after a browser-defined window (~seconds). Round 1's share handlers started rasterizing at tap time and `await`ed the result before calling `navigator.share` — on a slow device a long rasterization could outlive the activation window, making the share call reject `NotAllowedError`, which the round-1 cancellation classifier treats as terminal: the tap did NOTHING — no sheet, no fallback, no feedback.

**Fix (Celebration — mount-eager).** The card's data is fixed the moment the modal opens (the `kind`/`cells`/`playerName` props plus the resolved event name), so `Celebration.tsx` starts `renderBingoShareCard` in a mount effect and caches the promise in a ref; the tap awaits the usually-already-settled cache and `navigator.share` runs within the activation window. The effect re-keys on the card's inputs, so a late-arriving Event name re-renders the card once rather than baking the fallback in. The `.catch(() => null)` lives INSIDE the cached promise — a render failure resolves null (degrading to the text/URL leg at tap time) and can never surface as an unhandled rejection when the Player closes with "Keep playing" and the blob is never consumed. Deliberate consequence: the card is now rasterized even when never shared — it stays on-device and unobserved (ADR 0005), and the "Keep playing" test pins that closing shares nothing and fires no analytics.

**Fix (Leaderboard — warm-on-intent, deliberately NOT mount-eager).** `Leaderboard.tsx` re-renders on every roster snapshot (any Player's Mark updates a player row), so a mount-eager render re-keyed on live data would re-rasterize per snapshot — real CPU/battery burn on phones for a card that is rarely shared. Instead the Share button warms the render when the Player signals intent — `onPointerEnter` (mouse hover, a real head start), `onFocus` (keyboard), `onPointerDown` (touch press, a small head start) — caching the promise keyed by its inputs (the roster array's identity + the resolved event name). The tap reuses the warmed promise only while those inputs still match; a roster that moved between warm-up and tap renders fresh at tap time, so the card never shows stale standings. Stated residual: that cold/stale-tap path keeps round-1 timing, so a very slow device can still expire the activation window there — rare, and strictly no worse than before this fix.

**The `NotAllowedError` decision, revisited honestly.** `NotAllowedError` is ambiguous by platform: some browsers report a user dismissal / declined permission that way, while an expired activation window rejects with the same name. The round-1 classifier (`isUserCancelledShare`) treats it as a cancellation that STOPS the chain, and round 2 KEEPS that behavior, with this rationale: the eager pre-render removed the main non-dismissal cause (a full tap-time rasterization inside the activation window), so a `NotAllowedError` reaching the classifier is now predominantly a real dismissal — and the residual failure mode (a rare do-nothing tap on a slow device whose warmed render was cold or stale) is strictly better than the alternative (clobbering the Player's clipboard or downloading a file right after they declined to share). `AbortError` remains the unambiguous, spec-mandated cancellation and always stops. Both stops are pinned by test.

**The blob-ready gate completes the design (Codex P2, PR #111 round 3 finding 1).** Pre-starting the render narrowed the race but did not close it: a tap landing while the mount render was still UNSETTLED on a slow phone would await it inside the click handler and burn the activation window all the same — `NotAllowedError`, classified terminal, do-nothing tap. `Celebration.tsx` now tracks the cached render's settled state (`cardReady`, set when the mount promise resolves — with a promise-identity check so a STALE effect's settle cannot open the gate for a newer in-flight render) and keeps the Share button DISABLED until it settles, visually identical to the existing identity-gated disable. "Settled", deliberately not "blob exists": a failed or gate-refused render settles to `null` and still enables the button, so the text/URL fallback stays reachable — the gate must never dead-end the affordance. A tap can therefore only ever await an already-settled promise, keeping `navigator.share` inside its own activation window structurally. The Leaderboard deliberately gets NO equivalent gate (decided, stated): no render exists there until intent (warm-on-intent is the round-2 design), so disabled-until-settled would present a permanently disabled button that nothing warms — and a disable between `pointerdown` and `click` would swallow the very tap that warmed it. Its round-2 cold/stale-tap residual therefore stands, made rare by the warm-on-intent triggers.

## Pending squares render distinctly from confirmed marks (Codex P2, PR #111 finding 2)

In `admin_confirmed` claim mode a cell can be `marked: true` with `status: 'pending'` — the Player tapped it, but an Admin has not yet confirmed the Claim. `game/logic.ts`'s `markedMask` (and therefore `hasBingo`/`isBlackout`/`countMarked`) explicitly excludes `status: 'pending'` cells from counting as "on": `c.free || (c.marked && c.status !== 'pending')`. Before this fix, `buildBingoCardNode` rendered every `marked` cell identically (`.share-card-cell.marked`), so a Share Card could show more solid, confirmed-looking wins than the Player's stats actually credit — overstating unconfirmed squares exactly where the game logic itself withholds credit.

**Fix.** `buildBingoCardNode` now also applies a `.share-card-cell.pending` class when `c.status === 'pending'`, mirroring `Board.tsx`'s own on-screen treatment (`'cell' + ... + (c.status === 'pending' ? ' pending' : '')`) — a pending mark carries BOTH `.marked` and `.pending` together, never one instead of the other. `src/index.css`'s `.share-card-cell.pending { opacity: 0.7; border-style: dashed; }` is a direct, card-scoped mirror of `.cell.pending`'s existing on-page rule, so a pending square reads as faded/dashed rather than a solid win, consistent with what the Player already sees on their own Board.

## Cancellation vs. failure on the text/URL fallback leg (Codex P2, PR #111 finding 3)

`shareCardBlob`'s file-share leg (`navigator.share({ files })`) already treated ANY caught rejection as a Player cancellation and stopped the chain — deliberately: a Player who dismissed the native sheet should not be surprised with a second share prompt right after declining the first (the comment on that branch is unchanged and still applies). The text/URL leg (reached when file sharing is unsupported) did not make the same distinction: it caught every rejection — including a genuine `AbortError` cancellation — and unconditionally fell through to the clipboard/download legs. A Player who opened the OS share sheet on the text/URL leg and dismissed it would have their clipboard silently overwritten (or a file silently downloaded) immediately after declining to share.

**Fix.** A new `isUserCancelledShare` helper inspects the rejection's `name` for `AbortError` (the spec-mandated name for a user-cancelled Web Share API call) or `NotAllowedError` (some browsers report a declined permission/dismissal this way — an ambiguity round 2 revisits and resolves in § Eager pre-render above). On the text/URL leg specifically: a cancellation now returns `'cancelled'` and stops, matching the file leg's Player-respecting behavior; any OTHER rejection (a genuine failure — the API existed but something actually went wrong) still falls through to the clipboard, then the download, as before.

**`share_click` still fires on every path, cancelled or not.** Both call sites (`Celebration.tsx`, `Leaderboard.tsx`) fire `track('share_click', { surface })` from a `finally` around the whole `shareCardBlob` call — unconditionally, once per tap, regardless of `shareCardBlob`'s return value. That contract is unchanged by this fix and is the intended, documented behavior: `share_click` measures the Player's intent to share (they tapped Share), not the outcome, so a cancelled share still counts as a click. `shareCardBlob`'s return value (`'files' | 'text' | 'clipboard' | 'download' | 'cancelled' | 'none'`) is what distinguishes outcomes for any caller that cares; the analytics event deliberately does not fork on it.

## Title contrast: `.share-card-title` follows the theme ink, not hardcoded white (Codex P2, PR #111 finding 4)

`.share-card-title` filled its text with a hardcoded `color: #fff` — invisible against `summer-white`'s light `--bg` (`#f6f1e7`), the one Theme (of 8) with a light background. This is the same failure mode `specs/og-theme-parity.md`/issue #71 already found and fixed for the Cloud Run OG renderer's own `.title` rule (hardcoded `#fff` measured ~1.13:1 against `summer-white`, far under WCAG AA's 4.5:1), which now fills with the theme's `ink` token instead.

**Fix.** `.share-card-title` now sets `color: var(--ink)`, mirroring that precedent. The `text-shadow: 0 0 14px #fff, ...` glow is left as `#fff` deliberately — it is a bloom/halo effect layered around the fill, not the fill itself (the same choice the OG renderer's own fixed `.title` rule made, keeping its glow white while switching only the `color`), and does not reintroduce a contrast failure the way the flat fill did. `src/og-theme-parity.test.ts` is unaffected: it pins `cloud-run/og-renderer/src/template.ts` against `src/theme/themes.css` only and never reads `src/index.css`, so this fix — consuming an existing `themes.css` token from `index.css` — cannot regress it, and `themes.css` itself is untouched.

**Audited, not fixed: `.share-card-cell.marked`'s hardcoded white.** The same share-card CSS block has one other hardcoded-`#fff` pair — `.share-card-cell.marked { color: #fff; border-color: #fff; ... }`. This mirrors the app's own on-page `.cell.marked`, which `specs/w1-themes.md` already documents as a real, pre-existing, OUT-OF-SCOPE defect (alongside `.celebrate .big` and `.signin h1`) tracked as a follow-up (issue #72), not folded into that ticket's WCAG contract. Left alone here for the same reason, and it is not currently the SAME acute failure the title was: `summer-white`'s `--primary`/`--secondary` were deliberately deepened (see `themes.css`'s own comment on that block) specifically so they clear 4.5:1 as real text fills, which incidentally also means white text/borders on the `--primary`→`--secondary` gradient background still read on `summer-white` today. This fix does not add a third hardcoded-white spot to `index.css`.

## Long unbroken prompts wrap on the card instead of clipping (Codex P3, PR #111 round 3 finding 2)

`.share-card-cell` hid overflow (`overflow: hidden`) WITHOUT the wrapping rules the on-screen `.cell` carries (`word-break: break-word; hyphens: auto;`), so a long unbroken token — an 80-char URL-ish prompt — rendered as one unwrappable line and lost most of its text on the shared image, reading worse on the card than on the live Board. The fix copies exactly that `.cell` pair into `.share-card-cell` ahead of the overflow hide, so the text breaks and hyphenates first and the hide only trims genuine excess. Pinned the same way as the title fill (jsdom computes no layout): a source-parse test asserts both `.share-card-cell` and `.cell` carry the pair — so the card cannot silently drift from the Board's own wrapping semantics — and a DOM-side assertion hands the renderer a long unbroken token and confirms it reaches the cell node untrimmed (no renderer-side truncation layered on top of the CSS).

## Claim → test

Every claim below maps to a real assertion against the actual `ShareCard`/`Celebration`/`Leaderboard` — no vacuous coverage. `html-to-image`'s `toBlob` is mocked at the module boundary (the only jsdom-canvas-limited piece); `ShareCard.tsx` itself is never mocked when testing `Celebration`/`Leaderboard`, so the DOM node captured from the mock's call arguments is the renderer's REAL output.

Runner: `npm test` (Vitest, jsdom). Test: `src/components/w2-share-cards.test.tsx`.

- `renderBingoShareCard` produces a non-empty `Blob` and calls `html-to-image`'s `toBlob` with `{ pixelRatio: 3 }`; the DOM node handed to it contains the Player name, the Event name, exactly 25 `.share-card-cell` nodes, a `BLACKOUT`/`BINGO!` title matching `kind`, and free/marked cells carry their distinct classes.
- `renderBingoShareCard` mounts its host under `document.body` for the render and removes it afterward (nothing named `.share-card-host` survives in the document once the returned Promise resolves).
- `renderBingoShareCard` rejects when `html-to-image` resolves `null` (no image data) rather than silently returning an empty/invalid blob.
- `renderLeaderboardShareCard` produces a non-empty `Blob` whose DOM node contains the Event name, one `.share-card-row` per given row, each row's stats line (including a `BLACKOUT` suffix when set), and a `.share-card-row.pinned` row carrying "1st BINGO" exactly on the row flagged `firstToBingo`.
- `shareCardBlob` shares `{ files: [File], title, text }` via `navigator.share` when `navigator.canShare({ files })` reports true, and returns `'files'`.
- `shareCardBlob` returns `'cancelled'` (and does NOT fall through to the clipboard or a download) when the native file share throws.
- `shareCardBlob` falls back to a text/URL `navigator.share` (`'text'`) when `navigator.canShare` is absent/false; to the clipboard (`'clipboard'`) when there is no Web Share API at all; and to a direct download (`'download'`) as the last resort when neither Share nor Clipboard is available; returns `'none'` when nothing — including the blob itself — is available.
- `Celebration`'s Share button renders the real BINGO card (asserted via the `toBlob` DOM-node capture, carrying the resolved Player/Event name and the `cells` prop's fixture data) and invokes `navigator.share` with a `files` payload when `canShare` reports true.
- `Celebration`'s Share button falls back cleanly (text/URL share, verified via the mocked `navigator.share` call shape) when `navigator.canShare` is unsupported, and `share_click` (`surface: 'celebration'`) fires on both the file-share path and the fallback path.
- `Leaderboard` renders a "Share leaderboard" button; clicking it renders the real Leaderboard card (asserted via the `toBlob` DOM-node capture) built from the FULL roster (unaffected by the active filter), pins the same Player the on-screen "1st BINGO" badge pins even when that Player's rank falls outside the card's top-N slice, and fires `share_click` (`surface: 'leaderboard'`).
- `renderBingoShareCard` refuses to render — rejecting before ever calling `html-to-image`'s `toBlob` or touching `.share-card-host` — when `cells` is not exactly 25 entries (Codex P2, PR #111 finding 1: the empty-card validity gate).
- `Celebration`'s Share button renders the FULL 25-cell grid on an immediate tap even when the (now-unused) `useBoard` hook is stubbed to report no data at all — proving the card is built from the `cells` PROP `Board.tsx` hands it, never from a second listener (Codex P2, PR #111 finding 1).
- `Celebration`'s Share button never attempts an image share (`toBlob` is not called) and falls back to a text/URL `navigator.share` when handed an invalid (non-25-cell) board, even when `canShare` would otherwise greenlight a file share (Codex P2, PR #111 finding 1).
- `renderBingoShareCard`'s DOM node applies BOTH `.marked` and `.pending` to a cell that is `marked` with `status: 'pending'`, and `.marked` alone (no `.pending`) to a confirmed mark — the two states are structurally distinguishable on the card (Codex P2, PR #111 finding 2).
- `shareCardBlob`'s text/URL fallback leg returns `'cancelled'` and does NOT write to the clipboard when the rejection is an `AbortError` (Codex P2, PR #111 finding 3).
- `shareCardBlob`'s text/URL fallback leg falls through to the clipboard leg when the rejection is a genuine failure (any name other than `AbortError`/`NotAllowedError`) (Codex P2, PR #111 finding 3).
- `src/index.css`'s `.share-card-title` rule sets `color: var(--ink)`, not a hardcoded hex — parsed directly from the stylesheet source, the same technique `src/theme/w1-themes.test.tsx` and `src/og-theme-parity.test.ts` already use for CSS-token assertions (Codex P2, PR #111 finding 4).
- `Celebration`'s card carries the SAVED name from the `playerName` prop on an immediate tap — asserted on the `.share-card-player` node — and never the auth fallback, with `useMyPlayer` stubbed permanently empty and the auth mock carrying a poisoned stale name (Codex P2, PR #111 round 2 finding 1).
- `Celebration` disables the Share button, pre-renders nothing, shares nothing, and fires no analytics while `playerName` is `null` (identity not yet known) (Codex P2, PR #111 round 2 finding 1).
- `Celebration` starts rasterizing at MOUNT (before any tap) and the tap reuses the cached render — `toBlob` is called exactly once across mount + share (Codex P2, PR #111 round 2 finding 2).
- While the mount-time rasterization is UNSETTLED, the Share button is disabled and a premature tap is inert (no share, no `share_click`); the settle enables it, and the tap that lands then reuses the settled render — no second `toBlob` call (Codex P2, PR #111 round 3 finding 1).
- A settled-`null` render (failure or validity-gate refusal) still ENABLES the Share button — the ready gate never dead-ends the affordance; the tap degrades to the text/URL leg (Codex P2, PR #111 round 3 finding 1, asserted by the render-failure and invalid-board tests tapping through the ready gate).
- `Leaderboard` does NOT rasterize at mount; hovering the Share button warms exactly one render and the subsequent tap reuses it — `toBlob` called exactly once across hover + tap (Codex P2, PR #111 round 2 finding 2).
- `shareCardBlob`'s text/URL leg returns `'cancelled'` (no clipboard write) for a `NotAllowedError` rejection — the pinned round-2 decision that dismissal-vs-activation-expiry ambiguity resolves as a stop (Codex P2, PR #111 round 2 finding 2).
- `.share-card-cell` carries the same `word-break: break-word` + `hyphens: auto` pair as the on-screen `.cell` (both source-parsed, so they cannot silently diverge), and a long unbroken token handed to `renderBingoShareCard` reaches the cell node untrimmed (Codex P3, PR #111 round 3 finding 2).

## Acceptance criteria

- **Given** a Player hits BINGO, **when** they tap Share, **then** an on-device retina image is handed to the native share sheet (`navigator.share({ files })`) and `share_click` fires — `src/components/w2-share-cards.test.tsx` ("Celebration" describe block).
- **Given** the Leaderboard, **when** a Player shares it, **then** a Leaderboard Share Card image is generated on-device and shared — `src/components/w2-share-cards.test.tsx` ("Leaderboard" describe block).
- Only BINGO + Leaderboard Share Cards exist — no per-square proof cards: `ShareCard.tsx` exports exactly `renderBingoShareCard`/`renderLeaderboardShareCard`; `ProofSheet.tsx`/`ProofFeed.tsx` are unmodified.
- No public unauthenticated page is added; generation is on-device: `ShareCard.tsx` has no server call, no route, no public URL — it is a pure client-side DOM→blob renderer called from behind the auth-gated `Celebration`/`Leaderboard` surfaces.
- The text/URL-only share is replaced by the image path with a graceful fallback: `Celebration.tsx`'s `share` handler no longer calls `navigator.share({ title, text, url })` directly — that shape now lives only inside `shareCardBlob`'s fallback leg, reached only when the image path isn't available.
- **Given** a Player taps Share, **when** the board handed to the renderer is not a real 25-cell board (e.g. Celebration mounted with incomplete `cells`), **then** no image share is ever attempted and the flow falls back to the text/URL share leg — `src/components/w2-share-cards.test.tsx` ("ShareCard — renderBingoShareCard" and "Celebration" describe blocks).
- **Given** an `admin_confirmed` Event with a marked-but-unconfirmed Square, **when** a BINGO/Blackout Share Card is generated, **then** that Square renders distinctly (`.pending`) from a confirmed mark, never as an indistinguishable solid win square — `src/components/w2-share-cards.test.tsx` ("ShareCard — renderBingoShareCard" describe block).
- **Given** a Player dismisses the OS share sheet on the text/URL fallback leg (file sharing unavailable), **when** the share is cancelled, **then** the chain stops silently — no clipboard write, no download — while `share_click` still fires once for the tap; a genuine (non-cancellation) failure on that same leg still falls through to the clipboard/download legs — `src/components/w2-share-cards.test.tsx` ("shareCardBlob" describe block).
- **Given** the `summer-white` Theme (the one light-background Theme of 8), **when** a BINGO/Blackout Share Card is generated, **then** the card title is legible — `.share-card-title` fills from the theme's `--ink` token rather than a hardcoded white — `src/components/w2-share-cards.test.tsx` (CSS source-parity describe block).
- **Given** a returning Player whose saved custom name differs from their auth (Google) name, **when** they tap Share immediately on a win, **then** the card carries the SAVED name Board resolved — never the auth fallback — and while the saved identity is not yet known (`playerName` null, the offline-reload window) the Share affordance is disabled instead — `src/components/w2-share-cards.test.tsx` ("Celebration" describe block) (Codex P2, PR #111 round 2 finding 1).
- **Given** a slow device where rasterization outlives the tap, **when** the Player goes to share a win, **then** `navigator.share` always runs within the tap's own activation window — Celebration's Share button stays disabled until the mount-time render SETTLES (a tap can only ever share a ready blob), and the Leaderboard's tap reuses the hover/focus/press-warmed render, never starting a fresh tap-time rasterization on the warm path — `src/components/w2-share-cards.test.tsx` ("Celebration" and "Leaderboard" describe blocks) (Codex P2, PR #111 round 2 finding 2 + round 3 finding 1).
- **Given** a Prompt whose text is one long unbroken token, **when** a BINGO/Blackout Share Card is generated, **then** the cell text wraps/hyphenates on the card exactly as it does on the live Board instead of being clipped by the overflow hide — `src/components/w2-share-cards.test.tsx` (CSS source-parity describe block) (Codex P3, PR #111 round 3 finding 2).

## Out of scope

- No per-square proof cards (ADR 0005) — a Proof already carries its own shareable media.
- No server-rendered OG images or public share pages — the scaffolded `cloud-run/og-renderer/` and its removal are #39, not this ticket.
- No new GA4/PostHog event types — both new call sites reuse the existing `share_click` event with the existing `{ surface }` param shape; `analytics.ts` is unmodified (off-limits this wave).
- `useData.ts`, `firestore.rules`, `types.ts`, `auth/**`, `theme/**` are untouched — `Celebration`/`Leaderboard` read through hooks/functions those files already export; nothing about their own contracts changes. `Board.tsx` is the one exception (Codex P2, PR #111 finding 1 + round 2 finding 1): it hands `cells` and `playerName` to `<Celebration>` as props — one render call — with no other change to its gating, hooks, or render logic; see § Empty-card race and § Stale-name race above.
- The known, pre-existing hardcoded-white defects in `.cell.marked`, `.celebrate .big`, and `.signin h1` (documented in `specs/w1-themes.md`, tracked by issue #72) are NOT fixed here — only `.share-card-title` (this ticket's own surface) and the audit confirming `.share-card-cell.marked` isn't currently failing the same way (§ Title contrast above).
