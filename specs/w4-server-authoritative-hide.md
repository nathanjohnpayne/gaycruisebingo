# Phase 1 server-authoritative hide: flip status at the report threshold (#43)

Phase 0's community auto-hide is presentational only ([ADR 0004](../docs/adr/0004-reactive-moderation.md)): a Report increments `reportCount`, and every client independently filters out content whose count has reached the Event's `settings.reportHideThreshold` (`isReportHidden`, `src/data/moderation.ts`). It is bypassable by design. Phase 1 promotes that hide to **server-authoritative**: a Cloud Function flips the over-reported Proof or Prompt to `status: 'hidden'` so the removal is real for every reader, not just cooperating clients. The honor system is untouched ([ADR 0001](../docs/adr/0001-honor-system-trust-model.md)) — this hides over-reported content; it does not recompute stats, gate play, or make marks trustworthy. `recomputeStats` is neither re-added nor repurposed.

This ticket keeps `moderateProof` (sharp thumbnail + Cloud Vision extreme/violent-only flagging) as-is and adds the threshold-hide Functions plus the `firestore.rules` change that makes `status` server-authoritative for the community-hide transition — including gating item (Prompt) READS so the hide is authoritative, not merely presentational. It composes with the [#101 notifier](w4-email-resend-admin-notify.md): the `status → 'hidden'` transition these Functions write is exactly the one the notifier labels `(reports >= threshold)`, so the auto-hide admin email is the downstream reaction.

## The threshold-hide Functions

Two `onDocumentWritten` triggers in `functions/src/index.ts` — `hideItemAtThreshold` over `events/{eventId}/items/{itemId}` and `hideProofAtThreshold` over `events/{eventId}/proofs/{proofId}` — delegate to `applyThresholdHide` (`functions/src/autohide.ts`). The decision is a pure predicate, `shouldHideAtThreshold(before, after, threshold)`, with the Firestore threshold-read and status-write injected, so the whole flow is unit-testable without a Functions runtime (mirroring `notify.ts`). They inherit the module's global `us-central1` region (`setGlobalOptions`), alongside the #101 notifiers.

`applyThresholdHide` reads `events/{eventId}.settings.reportHideThreshold` via the admin SDK and, when the predicate holds, flips the doc to `'hidden'`. It is best-effort: a read/write failure is swallowed (`console.error`, return `false`), so the trigger never crashes the pipeline (ADR 0001, mirroring `moderateProof` and the notifier).

### Active-only, count-rose (crossing OR retry), fail-safe

`shouldHideAtThreshold` fires only when the doc is currently `'active'` AND `reportCount` **rose to at/over** the threshold on this write (`before.reportCount < after.reportCount` AND `after.reportCount >= threshold`). "Rose to at/over," rather than a strict below→over **crossing**, is deliberate: it covers both the initial crossing AND a **retry**. Because the write is best-effort and swallows failures, a first crossing attempt that hit a transient threshold-read or transaction failure leaves the doc `'active'` already at/over the threshold; a strict crossing (`before` already ≥ threshold) would never re-attempt on the next report, leaving an over-threshold doc readable (round 3 F2). The "rose" gate re-attempts on the next bump. That shape carries four properties:

- **Active-only (F2).** ONLY an `'active'` doc is auto-hidden. A `'flagged'` (Vision) or `'pending'` (admin_confirmed claim) doc that a stale/queued report bump carries over the threshold is **left alone**, so the stronger moderation state is never downgraded to a plain `'hidden'` — and an admin Restore can never expose a still-`visionFlag`ged proof. This is also the loop guard: the Function's own hide write makes the doc `'hidden'` (not active), so the re-fired `onDocumentWritten` no-ops. No infinite loop.
- **Rose, not a bare `count >= threshold`.** Admin restore is preserved: `restoreItem`/`restoreProof` (`src/data/admin.ts`) set `status → 'active'` but deliberately leave `reportCount` over the threshold. That write does **not raise** `reportCount`, so it is not a rise and is **not** re-hidden — the restore sticks (until the community reports the item again and the count rises, which is the intended over-hide-safe behavior). The broadened gate is retry-safe precisely because the actual write still passes through the transactional `hideIfQualifies` below.
- **Fail-safe.** The `threshold > 0` guard mirrors `isReportHidden`: an unset, zero, negative, or non-numeric threshold hides **nothing**, so one admin typo can never server-hide the whole pool.
- **Cheap.** `applyThresholdHide` short-circuits before the event-doc read whenever the doc is a delete, is non-active (flagged/pending/hidden), or `reportCount` did not rise — so the common no-op re-fires never pay for a threshold read.

### The write is conditional on LIVE state, in a transaction (round 2 F1)

The snapshot predicate above only decides whether to **attempt** a hide; the actual write is guarded so a delayed trigger can never act on the stale event snapshot. `applyThresholdHide` hands off to `hideIfQualifies(db, collection, eventId, docId)`, which inside a **Firestore transaction** re-reads the target doc AND the Event threshold, then writes `status: 'hidden'` (via `tx.update`, never a re-creating set) only if `stillQualifiesForHide` — the doc is currently `'active'` with `reportCount` at/over the current positive threshold. Otherwise it no-ops:

- an admin used the console's **Clear reports** on the row after the trigger fired (`reportCount` now below threshold) → the transaction sees the live low count and does **not** undo the admin's lift;
- the doc is already `'hidden'`/`'flagged'`/`'pending'`, was **deleted** since the snapshot (a missing doc never gets re-created — this subsumes the update-not-set guarantee), or the threshold was raised past the count → no-op.

Because the read-then-write is atomic against a concurrent clear, the hide reflects live state, not the queued event. The same `hideIfQualifies` guard is reused by the backfill and the rollout sweep, so every server hide in this module re-confirms before writing. `hideIfQualifies` takes the Firestore handle as a parameter so the transactional guard is unit-tested with a fake transaction.

- **Given** a report bump that raises `reportCount` to at/over `reportHideThreshold` on an `active` doc whose LIVE state still qualifies at write time — whether the initial crossing OR a retry after a swallowed earlier attempt (the doc stayed active at/over threshold and the count rose again) — **then** `applyThresholdHide` flips it to `'hidden'`; **given** a bump that stays below, a `flagged`/`pending`/`hidden` doc that crosses (F2), a write that does not raise `reportCount` (admin hide/restore, claim confirm, the Function's own re-fire), a delete, or an unset/zero/negative/`NaN` threshold **then** it is a no-op. (Tests: `tests/functions/w4-server-authoritative-hide.test.ts`, incl. the R3 F2 retry case.)
- **Given** the live doc no longer qualifies at write time — reports cleared below threshold, already hidden, or deleted — **then** `hideIfQualifies` writes nothing and returns `false`, so a delayed hide cannot undo an admin clear or resurrect a deleted doc. (Tests: the `hideIfQualifies` and `stillQualifiesForHide` cases.)
- **Given** the injected threshold-read or the transaction throws **then** `applyThresholdHide` swallows it and resolves `false` — never rejects. (Tests: the "never-throws" cases.)

## The threshold-decrease backfill (F3)

Because the per-write hides fire only on a fresh crossing, LOWERING `settings.reportHideThreshold` below reports that already exist would never server-hide those already-over-threshold docs (no new crossing; later bumps no-op because `beforeCount` already exceeds the new bar). For a server-authoritative hide that is a real hole, so a third trigger closes it: `backfillHideOnThresholdDecrease` over `events/{eventId}` delegates to `applyThresholdBackfill`. When `reportHideThreshold` **decreases** (or is enabled from unset — `backfillThreshold` treats a previously non-positive threshold as +Infinity), it sweeps the Event's own `items` + `proofs`, querying `reportCount >= newThreshold` (a single-field auto-index — no composite index) and hiding the `active` ones (reusing the F2 active-only gate and the F1 `update` writer).

The query + active-only + count checks are a cheap pre-filter; every actual write goes through the same transactional `hideIfQualifies` (round 2 F1), so a sweep that races an admin Clear-reports re-confirms live state and no-ops rather than undoing the lift. The sweep is bounded to the one Event's subcollections and best-effort: each per-doc write is independently try/caught so one failure never aborts the sweep, and the outer catch keeps it from ever throwing. It never writes the Event doc, so it never re-fires itself; its `status → 'hidden'` writes re-fire the per-write hides, which no-op (the doc is now `'hidden'`, not active). A threshold that is unchanged, raised, or disabled runs no query at all.

- **Given** the Event's `reportHideThreshold` decreases (10 → 4) or is enabled from unset **then** active items/proofs whose `reportCount` now meets the lower bar (and still qualify at write time) are hidden, while `flagged`/`pending` docs, docs below the bar, and docs an admin cleared mid-sweep are left alone; **given** the threshold is unchanged, raised, or disabled **then** nothing is swept (no query). (Tests: `backfillThreshold` + `applyThresholdBackfill` in `tests/functions/w4-server-authoritative-hide.test.ts`.)

## The one-time rollout sweep (round 2 F2)

The per-write hide and the decrease-backfill both fire on a **change**. On the Phase-1 functions **deploy**, content that already crossed the (unchanged) threshold under Phase 0 never crosses again and never triggers a decrease, so it would stay `'active'` and directly readable despite meeting the server-hide bar. A one-time, operator-invokable sweep closes that rollout gap — a **script**, not a deployed surface, so it fits the deferred-deploy model.

`scripts/backfill-hide.mjs` (run once after the first functions deploy) calls `runRolloutSweep(eventId?)` in `functions/src/autohide.ts`: for every Event (or one `eventId`) it reads the Event's current `reportHideThreshold` and reuses the backfill core with `before = null` — an enable-from-unset, so `backfillThreshold` sweeps at the current bar — hiding each Event's active over-threshold items + proofs through the SAME transactional guard. It is therefore DRY (one sweep implementation), idempotent (a second run finds nothing active over threshold — already-hidden docs no longer qualify), and race-safe (the transactional guard re-confirms per doc). The script imports the built `functions/lib/autohide.js`; every `firebase-admin` import there is lazy, so it needs only the built + installed functions package (no root install). For credentials it mirrors `scripts/seed.mjs` exactly (round 3 F1): it reads a gitignored repo-root `serviceAccountKey.json` and authenticates with `cert(...)` when present, else falls back to Application Default Credentials — the parsed key is passed to `ensureAdminApp`, which initializes the app INSIDE the functions package so `getFirestore()` sees the same credential. Its direct-run guard (`process.argv[1] && import.meta.url === pathToFileURL(process.argv[1]).href`) is also copied from `seed.mjs`, keeping the module import-safe when `argv[1]` is undefined (round 3 F3).

- **Given** the rollout sweep runs over an Event with pre-existing active over-threshold content **then** that content is hidden; **given** it runs again **then** nothing new is hidden (idempotent), and flagged/pending/below/cleared docs are always skipped. (Tests: `runRolloutSweep` + the enable-from-null and idempotency cases of `applyThresholdBackfill` / `hideIfQualifies`.)

## The firestore.rules change: status is server-authoritative, and hidden Prompts are read-gated

Phase 1 requires that only the server (or an admin) may set the community-hide `status`, and that a hidden Prompt actually disappears for players.

**Write side.** The rules already enforce the write half: a non-admin's only `update` on items/proofs is `hasOnly(['reportCount']) && reportCount == old + 1`, which permits changing `reportCount` alone and nothing else, so a non-admin can never set `status` (to `hidden` or anything) nor un-hide. The Function writes via the admin SDK, which **bypasses** rules. So the write change is surgical: the `items` and `proofs` update rules gain comments recording the server-authoritative invariant, pinned against regression.

**Read side (F4).** The item read rule is tightened to mirror proofs — `allow read: if signedIn() && (isAdmin(eventId) || resource.data.status == 'active')` — so a server-hidden Prompt's `text` is no longer directly fetchable by a non-admin; the hide is authoritative, not just presentational. Because Firestore rejects a collection query unless it is constrained to only match readable docs, the player-facing pool query is filtered: `useItems` now subscribes to `query(itemsCol(), where('status','==','active'))` (a single equality — `firestore.indexes.json` untouched), matching what the deal path (`joinAndDeal`, `src/data/api.ts`) already did. The admin console query (`useAllItems`) stays unconstrained and reads every status via the `isAdmin` arm. Admin manual hide/restore, proof creation (`status in ['active','pending']`), and the #106/#116/#110/#113 write paths are untouched.

- **Given** a non-admin **then** setting `status: 'hidden'` on an item or proof — alone or smuggled alongside a `reportCount` bump — is DENIED, and un-hiding (`status: 'active'`) is DENIED; **given** a bare `reportCount + 1` **then** it SUCCEEDS; **given** an admin **then** a manual hide/restore SUCCEED; **given** a valid proof create **then** it SUCCEEDS, while create-into-`hidden` is DENIED. (Tests: `tests/rules/w4-server-authoritative-hide.test.ts`.)
- **Given** F4 **then** a non-admin reads an `active` Prompt but NOT a `hidden` one, an unconstrained item collection query is DENIED while a `status=='active'`-filtered query is ALLOWED, and an admin reads every Prompt (single-doc and unconstrained). (Tests: the F4 read-gate cases.)

## Deferred deploy step (human / orchestrator)

Deploying the functions suite (`firebase deploy --only functions`) is a post-merge finishing step, not part of this code PR. The whole Phase-1 functions backend (`moderateProof` + these threshold-hides + the #101 notifiers) is Blaze-gated and lands in a separate, deliberate Phase-1-backend cutover the human owns. `moderateProof` is kept exactly as-is — this PR shows no diff on it.

Immediately **after** that first functions deploy, the operator runs the one-time rollout sweep — `node scripts/backfill-hide.mjs` — to hide already-over-threshold content that predates server-authoritative hiding. This is documented as a rollout step in `docs/app/phase-1-deploy.md` (§ 1b). It is safe to re-run (idempotent).

## Relationship to the Phase-0 client hide

The client presentational filter (`isReportHidden`, #107) stays as the Phase-0 fallback and is not removed; this promotes the same boundary to authoritative removal. Both apply the identical at/over/below-threshold and positive-threshold semantics, so the server hide and the client filter never disagree at the boundary.
