# w2-feed-moments — Feed = Proofs + Moments (first BINGO / Blackout / First to BINGO; bare Marks post nothing)

The Feed is the live, public stream the group watches together (ADR 0002) — the honor-system source of truth in place of server verification. Proofs already flow into it (`specs/w2-proof-capture.md`); this ticket adds Moments and merges the two into one newest-first stream. A **Moment** broadcasts a big social beat — a BINGO, a Blackout, or the First to BINGO — and, unlike a Proof, carries **no attached evidence**: it marks *that* something happened, not what it looked like. A **bare Mark posts nothing to the Feed** (ADR 0002): only a Proof or a Moment appears there; a plain honor Mark self-publishes to its per-Prompt Tally (`specs/w2-tally.md`) and nowhere on the Feed.

This ticket landed after #16/#18, so two contracts already shipped: the `MomentDoc` type (`src/types.ts` — `kind: 'bingo' | 'blackout' | 'first_bingo'`, plus `uid`, `displayName`, `photoURL`, `createdAt`; no media, no `proofId`) and the `events/{eventId}/moments/{momentId}` rules block (`firestore.rules`). The `MomentDoc` type is read, not modified. The moments rules block was designed strictly within as it shipped, then **tightened in the PR #99 review rounds** (Codex P2): the create rule now bounds `createdAt` near `request.time` (finding 3) and the admin update path was removed for full immutability (finding 4). A second review round hardened the client-side broadcast machinery (findings A–D: held-broadcast revalidation, the account-switch data race, cache-hydration baselines, and the regained-line witness). See § Review-round hardening (PR #99).

## Data model (merged-main type + rules, #16/#18)

A Moment is a top-level doc under the Event: `events/{EVENT_ID}/moments/{momentId}`. The shape is exactly `MomentDoc` minus its `id` (the `id` is the doc id, surfaced on read by `momentConverter`). The rules block allows:

- **create** if `isOwner(request.resource.data.uid)` (a forged uid is denied — you can only broadcast your own beat), the `kind` is one of the three `MomentKind`s, `displayName` is a non-empty ≤100-char string, and `createdAt` is a number **within +60s / -24h of `request.time`** (the same near-now window proofs uses — PR #99 finding 3) so a bad clock or a forged far-future stamp cannot pin a Moment above all Feed activity forever.
- **read** if signed in (public — the Feed everyone watches; no anonymity, ADR 0002).
- **update** — **none. A Moment is fully immutable** (PR #99 finding 4): the update path was removed. `MomentDoc` carries no moderation/status field, so an admin update could only rewrite content — and an admin-writable update defeated the create-only dedup (an admin-player's repeat broadcast became an authorized overwrite). Moment moderation is delete-only; a moderation/status field is deferred to #37/#41.
- **delete** if `isAdmin(eventId)` or the owner.

Two facts about that block are **load-bearing to this design and pinned honestly**:

1. **The doc id is caller-chosen** — no rule constrains `momentId`. Combined with **update being denied for everyone** (PR #99 finding 4), a deterministic id makes once-only *structural*: a second write to an already-written id is a doc-exists `update`, which no one — not even an admin — can do, so it is denied. This is the strongest dedup the rules allow, and it is what the writer relies on.
2. **The create rule has no `hasOnly()`/`keys()` constraint**, so it does **not** reject a Moment that carries extra `mediaURL`/`proofId` fields. The ADR 0002 "a Moment carries no evidence" guarantee is therefore a **writer + type** contract — `src/data/moments.ts` writes exactly the `MomentDoc` fields, and `MomentDoc` has no media/`proofId` field — **not** a rules-layer one. The rules test pins the actual (permissive) rule behaviour so no one mistakes the rule for enforcing it.

## The change

- `src/data/moments.ts` (new) — three broadcast functions, `broadcastBingo` / `broadcastBlackout` / `broadcastFirstBingo`, each writing one `MomentDoc` row via a raw (converter-free) ref in the mark path's offline-queueable, fire-and-forget style (`void setDoc(...).catch(console.error)`, mirroring `setMark`): the write pends durably in the persistent cache when offline (ADR 0006) and never blocks the celebration UI, and an online rejection is logged (never silently swallowed). The payload is `Omit<MomentDoc, 'id'>` — no media, no `proofId`, ever. `displayName` is bounded through the shared `markerDisplayName` helper (imported from `src/data/attribution.ts`, unchanged) to the rules' non-empty ≤100 contract, so a broadcast can never be rejected for a malformed name. Plus `hasPriorBingoWitness(uid)` (round 2 finding D): a cache-only, never-rejecting read of the player's own `${uid}-bingo` Moment doc — the durable witness of a prior win Board consults before queueing a ceremonial First-to-BINGO candidate (see § Review-round hardening (PR #99)).
- `src/data/converters.ts` — a `momentConverter` mirroring `proofConverter`/`claimConverter` (pins `id` to `snap.id`). Genuinely required so `paths.ts` can attach a converter to its refs, exactly as the tally/proofs helpers do.
- `src/data/paths.ts` — `momentsCol()` / `momentRef(id)` converter-attached refs under `events/{EVENT_ID}/moments`, mirroring `proofsCol`/`proofRef`.
- `src/hooks/useData.ts` — `useMoments()` subscribes to the moments collection through the SAME `useColSub` latch pattern the proof stream uses (`{ includeMetadataChanges: true }`, `hasServerData` latched on the first server-backed snapshot), newest-first, capped. `mergeFeed(proofs, moments, max)` is a pure function that folds both into one newest-first stream capped at `max`. `useFeed()` composes `useProofFeed` + `useMoments` through `mergeFeed`; `loading` stays true until both halves have delivered so the empty state never flashes. `useTally`/`useLeaderboard`/`useProofFeed` are untouched beyond being composed.
- `src/components/ProofFeed.tsx` — now renders the merged Feed from `useFeed()`. A Proof renders exactly as before (report ⚑, owner-delete 🗑, flagged badge, media by type, scheme-guarded via `safeMediaUrl`). A Moment renders distinctly: a celebratory per-kind line with an icon, **no media and no report/delete** (there is no evidence to dispute). The empty copy now reflects the whole Feed.
- `src/components/Board.tsx` — broadcasts at the EXISTING BINGO/Blackout edge detection (the `wasBingo`/`wasBlackout` refs + celebrate `useEffect`), leaving `setMark` and the #87 Tally UI untouched. It also subscribes to the roster via `useLeaderboard()` — the SAME data the Leaderboard's First-to-BINGO pin reads. The identity + roster (and their gate signals) plus the CURRENT attributable cells are stored in a latest-value ref so the edge effect fires only on a genuine board transition, never when the resolved name or roster merely re-renders. A crossed edge does not write directly: it enqueues the matching kind(s) into a `pending` ref (the ceremonial first_bingo candidate through the durable-witness check) and calls `flushPending`, which writes only once each gate is open AND the win still stands at fire time (see § Review-round hardening (PR #99) for the identity gate, the roster gate, the per-uid reset, held-flag revalidation, the attribution + confirmation baselines, and the witness — findings 1, 2, 5 and round 2 A–D). The celebration UI still fires on the edge itself and is never held.
- `src/index.css` — a `.moment` card style (festive tint, distinct from `.proof`; the First-to-BINGO card gets the Leaderboard pin's glow).

## Once-only semantics (deterministic ids + the create-only rule)

The design picks the **strongest dedup the rules allow** and layers it under the Board edge refs:

- **First BINGO — once per Player.** Broadcast on the TRANSITION into having a bingo (`bingo && !wasBingo.current`), never on every completed line or every render. Doc id `${uid}-bingo`. A re-fire (a lose→regain in one session, a reload with a bingo already standing, or a second tab) either never crosses the edge (the refs) or, if it does, hits the deny-all `update` rule on the existing id and is denied (PR #99 finding 4: denied for everyone, admins included).
- **Blackout — once per Player.** Broadcast on `black && !wasBlackout.current`. Doc id `${uid}-blackout`. On a real 5×5 board the first-line (BINGO) edge always precedes the full-card (Blackout) edge across renders, so a Blackout never suppresses the earlier first-BINGO broadcast.
- **First to BINGO — once per Event, ceremonial and self-reported (ADR 0001).** Doc id is the Event-**singleton** `first_bingo` (it can never collide with a per-Player id: those use a hyphen before the kind and a uid prefix). Board derives it client-side from the known-players roster: it claims First to BINGO only when the roster is **server-confirmed** AND, as far as that confirmed view shows, **no other Player has a `firstBingoAt` yet** (while the roster is unconfirmed the claim is HELD, not guessed — PR #99 finding 2). The singleton id makes it structurally once-per-Event: the first create wins; a later create becomes an `update` and is denied.

  **The regained-line witness (round 2 finding D).** `firstBingoAt` is BY DESIGN volatile — `computeMark` clears it when the last line falls — and the schema deliberately carries no durable first-win field. So the roster check alone would let a Player who had a line, unmarked it, and regained one later mint the event singleton despite not being first EVER (their own history is invisible to `othersBingoed`, which only inspects OTHER players). The fix uses **the Moment collection as its own memory**: the Player's own `${uid}-bingo` Moment is create-only and immutable, so its presence is a durable witness that a prior win existed even after the line was unmarked. Before queueing a ceremonial candidate on a bingo edge, Board calls `hasPriorBingoWitness(uid)` — a cache-only `getDocFromCache` read (instant, offline-safe, and same-session writes are visible via latency compensation): witness present → the ceremonial candidate is suppressed (the regained line still posts the plain bingo Moment, itself a denied duplicate create — fine); cache miss (fresh device) → fall back to the roster check. **Narrowed residual (accepted, ADR 0001):** a fresh-device, cache-cold, roster-blind regained line — the Player's prior win is neither in this device's cache nor visible as another player's `firstBingoAt` — can still claim the singleton. At friend-group scale this is acceptable for a self-reported ceremony; the honour is recorded, never adjudicated.

**The race, documented honestly.** First to BINGO is a ceremony, not a verified fact (ADR 0001). Under an offline/latency race two Players may each briefly believe they are first — each optimistically shows their own local `first_bingo` Moment from the persistent cache — but exactly one create reaches the server (whichever syncs first), the other is rolled back to the winner, and both clients converge. At friend-group scale this is acceptable; the Feed's `first_bingo` reflects who broadcast first, which under a bad offline race can differ from the Leaderboard pin's earliest-`firstBingoAt`. The honour is recorded, never adjudicated.

## Review-round hardening (PR #99)

The PR #99 Codex review ran two rounds. Round 1 surfaced six P2 findings; five are fixed here (the sixth is deferred — see § Deferred: admin-confirmed off-route wins): three client-side (two gates on the broadcast + a per-account reset), two `firestore.rules` boundary defects. Round 2 surfaced four more P2s in the round-1 broadcast machinery (A–D below), all fixed client-side.

**Held broadcasts — gated, revalidated, per-account (round 1 findings 1 + 2 + 5, round 2 findings A + B).** A crossed BINGO/Blackout edge no longer writes directly. It enqueues the matching kind(s) into a per-uid `pending` ref; `flushPending` writes each only once its gate is open — and only if the win STILL STANDS at fire time (a held flag is a candidate, not a promise). The deterministic doc id keeps a held-then-fired broadcast idempotent, and the celebration UI still fires on the edge itself (never held).

- **Identity gate (finding 1).** All three broadcasts require a KNOWN saved identity — the same `identityKnown` tri-state `setMark` gates on. During the player-row loading window `displayName` has fallen back to the auth/Google name; stamping that into an IMMUTABLE Moment would publish a returning Player's stale name forever. So an edge crossed while identity is unknown holds; when identity resolves, `flushPending` fires it once with the saved name.
- **Roster gate (finding 2).** First-to-BINGO additionally requires a **server-confirmed** roster (`useLeaderboard().hasServerData`). An initial empty `players` from a still-loading (or cache-only) subscription is not proof nobody has bingoed; claiming the ceremony off it would falsely pin a non-first Player. So the first_bingo candidate holds until the roster is server-confirmed, then re-decides `othersBingoed` against it. The per-Player BINGO does not wait on the roster — only the ceremonial claim does.
- **Held flags follow the board — dropped when the win falls (round 2 finding A).** A flag queued while a gate was closed could go stale: the Player unmarks and LOSES the win before the gate opens, and round 1 would then post a Moment for a board that no longer had one. Two complementary fixes, both required: the **falling edge** (bingo/blackout going true→false) clears the corresponding pending flag (a bingo fall also clears the ceremonial candidate), handling the common case; and `flushPending` **revalidates at fire time** — it recomputes `hasBingo`/`isBlackout` from the cells the component CURRENTLY holds (carried in the same latest-value ref as the identity/roster) and drops, without firing, any flag whose win no longer stands (first_bingo included: no standing bingo → no ceremonial claim). The revalidation is the invariant; a lose→regain crosses a fresh rising edge and legitimately re-queues, yielding exactly one write when the gate opens.
- **Per-uid reset, synchronous in render + attribution-gated (finding 5, hardened by round 2 finding B).** The edge refs (`initialized`/`wasBingo`/`wasBlackout`) and the `pending` ref are per-account. Round 1 reset them in a `useEffect` keyed on `uid` — but on a uid switch while Board stays mounted, `useBoard`/`useMyPlayer` still return the PREVIOUS uid's rows for the render(s) before their keyed effects clear, so the same commit's cells effect could seed the new account's baseline from the OLD board and make the new user's first real snapshot read as a fresh edge. Two fixes, both required: the reset now runs **synchronously during render** (React's adjust-during-render pattern — a ref tracks the uid the edge state belongs to; a mismatch restores the uninitialized baseline before ANY effect runs, idempotently); and the cells effect **seeds from attributable data only** — `BoardDoc` carries its owner's uid, and a board whose `uid` differs from the current account is ignored entirely (it also contributes NO cells to fire-time revalidation). The stale-render board is therefore never seeded, and the new uid's own first snapshot re-establishes init-without-firing.

**Cache hydration is baseline, not a transition (round 2 finding C).** Under the ADR 0006 persistent cache the first snapshot(s) after mount can be cache-only, and round 1 seeded the edge baseline from the FIRST snapshot regardless — so a stale cache lacking a BINGO the server already had made the server confirmation read as a live transition and post a Moment for a win that already stood. Now, while the board is NOT server-confirmed (`useBoard().hasServerData` false), EVERY snapshot re-seeds `wasBingo`/`wasBlackout` and never fires an edge (`initialized` stays false); the FIRST server-confirmed snapshot is baseline too (init-without-firing — any standing win it reveals already stood server-side); edge DETECTION runs only after it. Live play is unaffected: `hasServerData` is a latch that never un-sets, and a local optimistic Mark under a confirmed board arrives as the next snapshot via latency compensation and fires normally.

**createdAt bounded near request.time (finding 3).** The moments create rule now requires `createdAt` within the same +60s / -24h window of `request.time` that proofs use, so a bad clock or a forged far-future stamp can no longer pin a Moment above all Feed activity forever. The client keeps writing `Date.now()` — in bounds by construction for an online write. **Offline-drain residual (accepted, ADR 0001):** a Moment queued offline drains with its ORIGINAL `Date.now()` stamp, not the reconnect time, so one created and then held offline for **more than 24h** would trip the lower bound and be rejected on drain. At friend-group cruise scale a >24h offline gap before a BINGO/Blackout Moment reaches the server is implausible, and the honest alternative — an unbounded stamp — is the defect finding 3 closes; the bound wins. (Proofs cannot queue offline at all — `attachProof` rides a `runTransaction` that rejects offline — so they never hit this drain case; Moments, which ride an offline-queueable `setDoc`, do, hence this note.)

**Full immutability — no admin overwrite (finding 4).** The admin `update` allowance was removed: a Moment is now immutable for everyone. The create-only dedup is only sound if a re-broadcast to an existing id is denied for ALL callers; the prior admin-writable update let an admin-player's repeat broadcast become an authorized overwrite — including clobbering the `first_bingo` singleton under a stale-roster race. `MomentDoc` has no moderation/status field (only `kind`/`uid`/`displayName`/`photoURL`/`createdAt`), so an admin update could only rewrite content — there is nothing to moderate in place. Moment moderation stays delete-only; introducing a status field + a moderation-only update diff (`request.resource.data.diff(resource.data).affectedKeys().hasOnly([...])`) is deferred to #37/#41, which own that schema evolution.

**The regained-line witness (round 2 finding D)** is specified with the once-only semantics — see § Once-only semantics, "The regained-line witness", including the narrowed fresh-device residual.

## Deferred: admin-confirmed off-route wins (→ #41)

**The gap (finding 6).** In `admin_confirmed` claim mode a Mark starts pending and only becomes a win when an Admin resolves the Claim. If the winning Player is NOT on the Card route when the Admin confirms, no Moment is emitted: the BINGO/Blackout edge that broadcasts lives in `Board.tsx`'s cells effect, which only runs while the board is mounted. A confirm-time win by an off-route Player therefore lands on the leaderboard/stats without ever posting the celebratory Moment the live edge would have.

**Why it is not fixed here.** Emitting from the confirm write-path needs one of two things outside this ticket's file set and rules scope: (a) rules that let an Admin create a Moment for ANOTHER uid — but the moments create rule is own-uid-only by design, and finding 4 just made the collection strictly own-create + immutable, so widening it to cross-uid admin creates is exactly the change that reopens the dedup/forgery surface; or (b) a player-side listener on the Claim outcome that lives OUTSIDE the Card route so it fires wherever the Player is. Both are claim-lifecycle design owned by #41 (the confirm-path/claims ticket), not the Feed-Moments surface.

**Invariant #41 must satisfy.** A confirmed win must produce the SAME Moment the live edge would have — the same kind at the same deterministic id, attributed to the WINNER (not the confirming Admin) — exactly once. The deterministic-id, create-only, fully-immutable dedup this ticket ships must remain the backstop: whichever path emits (live edge or confirm-path), a duplicate is denied, so the two paths cannot double-post.

## Claim → test

Basename-aligned to this spec (the checker matches `specs/w2-feed-moments.md` → a `w2-feed-moments.test.*` under `tests/**` or `src/**`); every claim maps to a real assertion.

### Rules — a Player broadcasts their OWN Moment; forgery denied; fully immutable; createdAt bounded; extra fields not rejected

Runner: `npm run test:rules` (Firestore emulator). Test: `tests/rules/w2-feed-moments.test.ts`.

- A signed-in Player may create their own Moment (each of the three kinds) at a caller-chosen id; a forged uid (another Player's beat) is denied.
- The shape contract: an invalid `kind`, an empty `displayName`, an over-100-char `displayName`, and a non-numeric `createdAt` are each denied; exactly-100 is allowed.
- The once-only backstop (PR #99 finding 4): a second write to an already-written deterministic id is a doc-exists `update` and is denied for **everyone** — an owner re-broadcast, an admin repeat-broadcast, and an admin content rewrite all fail. This is what makes the deterministic id structurally once-only, and it holds for admin-players too.
- `createdAt` is bounded near `request.time` (PR #99 finding 3): a near-now stamp is accepted, a far-future stamp (+1h) and a far-past stamp (-2d) are each denied — the same +60s / -24h window proofs use.
- **Honesty pin:** a create carrying extra `mediaURL`/`proofId` fields is **accepted** — the rules do not reject it; the no-evidence guarantee is a writer/type contract (see the unit test), not a rules-layer one.
- Moment reads are public; an owner may delete their own Moment and an admin may delete any; a non-admin peer may not.

### Unit — the broadcast writer + the Feed merge

Runner: `npm test` (Vitest, jsdom). Test: `src/data/w2-feed-moments.test.ts`.

- `broadcastBingo` writes exactly ONE Moment at `${uid}-bingo` with `kind: 'bingo'` and EXACTLY the `MomentDoc` fields — no `mediaURL`, `storagePath`, or `proofId` (ADR 0002).
- `broadcastBlackout` writes one at `${uid}-blackout` (`kind: 'blackout'`); `broadcastFirstBingo` writes one at the Event-singleton `first_bingo` (`kind: 'first_bingo'`).
- The attributed name is bounded to the rules' ≤100 cap.
- Each broadcast is a single offline-queueable `setDoc` — never `addDoc`, never `runTransaction`.
- `hasPriorBingoWitness` (round 2 finding D) reads the player's own `${uid}-bingo` doc from the LOCAL cache only (`getDocFromCache`), resolves `true` when the cached witness exists, `false` when it does not, and `false` — never rejecting — on a cache miss (the fresh-device fallback path).
- `mergeFeed` interleaves Proofs + Moments strictly newest-first, keeps the `slice` cap, and returns an empty stream for `([], [])` — a bare Mark (neither a Proof nor a Moment) contributes nothing.

### Component — Board broadcasts on the edge; the merged Feed renders

Runner: `npm test` (Vitest, jsdom). Test: `src/components/w2-feed-moments.test.tsx`.

- Board broadcasts a first BINGO Moment exactly once on the TRANSITION into having a bingo — not on initialisation (a returning Player whose bingo already stood must not re-announce it) and not on a further Mark while the bingo still stands (the edge, not every render).
- A bare Mark that completes no line crosses no edge and broadcasts nothing.
- Board claims First to BINGO only when the roster shows no other Player has bingoed yet; when another Player already has a `firstBingoAt`, the Player's own first BINGO still posts but the ceremonial First-to-BINGO does not.
- A Blackout broadcasts one blackout Moment on the full-card edge without re-announcing the BINGO.
- **Identity gate (PR #99 finding 1):** a BINGO edge crossed while identity is UNKNOWN (the player row still loading) does NOT broadcast — the stale auth name is never written to an immutable Moment; when identity resolves, the held broadcast fires exactly once with the SAVED player-row name.
- **Roster gate (PR #99 finding 2):** a BINGO edge crossed while the roster is UNCONFIRMED holds the ceremonial First-to-BINGO (the own BINGO still posts); once the roster is server-confirmed, First-to-BINGO fires only if no other Player has an earlier `firstBingoAt`, and stays held-then-dropped if one does.
- **Per-uid reset (PR #99 finding 5):** switching accounts while Board stays mounted, onto the new account's OWN board with a standing bingo, fires NO spurious Moment for the new uid — the render-time reset re-establishes the init-without-firing baseline.
- **Held-flag drop + regain (round 2 finding A):** a BINGO queued while identity was unknown, then UNMARKED before identity resolves, fires NOTHING when the gate opens (the falling edge cleared it; fire-time revalidation is the backstop); a queue → unmark → remark → resolve sequence fires exactly ONE write (the remark edge re-queues legitimately).
- **Stale-board attribution (round 2 finding B):** on an account switch where the previous uid's board still renders once, that board is neither seeded as baseline nor read as an edge — the new uid's own first snapshot (even with a standing bingo) establishes init-without-firing, and no Moment posts.
- **Cache-hydration baseline (round 2 finding C):** a cache-only board snapshot without a bingo followed by the server-confirmed snapshot WITH one broadcasts nothing (confirmation is baseline, not a transition); a live Mark completing a line AFTER the board is server-confirmed broadcasts exactly once.
- **Regained-line witness (round 2 finding D):** when the durable `${uid}-bingo` witness is in the local cache, a regained line still attempts the plain bingo Moment write but does NOT re-queue the ceremonial First-to-BINGO candidate.
- ProofFeed renders Moments and Proofs merged newest-first (fed an already-merged list; the sort itself is `mergeFeed`'s unit test), each distinctly; a Moment carries no media and no report/delete affordances, a Proof keeps them; an empty Feed shows the empty state.

## Acceptance criteria

- Given a Player scores their first BINGO, when it commits, then a BINGO Moment appears newest-first in the Feed with no attached evidence — `src/components/w2-feed-moments.test.tsx` (Board edge broadcast) + `src/data/w2-feed-moments.test.ts` (one write, no media) + `tests/rules/w2-feed-moments.test.ts` (own-create allowed).
- Given a Player only marks a Square (no Proof), when the Feed refreshes, then nothing about that Mark appears in the Feed — `src/components/w2-feed-moments.test.tsx` (bare Mark: no broadcast + empty Feed) + `src/data/w2-feed-moments.test.ts` (`mergeFeed([], []) === []`) + the offline suite's moments-empty assertion.
