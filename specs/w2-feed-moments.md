# w2-feed-moments — Feed = Proofs + Moments (first BINGO / Blackout / First to BINGO; bare Marks post nothing)

The Feed is the live, public stream the group watches together (ADR 0002) — the honor-system source of truth in place of server verification. Proofs already flow into it (`specs/w2-proof-capture.md`); this ticket adds Moments and merges the two into one newest-first stream. A **Moment** broadcasts a big social beat — a BINGO, a Blackout, or the First to BINGO — and, unlike a Proof, carries **no attached evidence**: it marks *that* something happened, not what it looked like. A **bare Mark posts nothing to the Feed** (ADR 0002): only a Proof or a Moment appears there; a plain honor Mark self-publishes to its per-Prompt Tally (`specs/w2-tally.md`) and nowhere on the Feed.

This ticket landed after #16/#18, so two contracts already shipped: the `MomentDoc` type (`src/types.ts` — `kind: 'bingo' | 'blackout' | 'first_bingo'`, plus `uid`, `displayName`, `photoURL`, `createdAt`; no media, no `proofId`) and the `events/{eventId}/moments/{momentId}` rules block (`firestore.rules`). The `MomentDoc` type is read, not modified. The moments rules block was designed strictly within as it shipped, then **tightened in the PR #99 review rounds** (Codex P2): the create rule now bounds `createdAt` near `request.time` (finding 3) and the admin update path was removed for full immutability (finding 4). A second review round hardened the client-side broadcast machinery (findings A–D: held-broadcast revalidation, the account-switch data race, cache-hydration baselines, and the regained-line witness). See § Review-round hardening (PR #99).

**The broadcast source moved to the action path (issue #104).** PR #99's broadcast machinery drove the public Moment write off Board's SNAPSHOT DIFFING (the `wasBingo`/`wasBlackout` edge refs + a component-held `pending` ref). That coupling was the root of all five review rounds, and it had a residual bug: a win crossed while a gate was still closed was HELD in a Board-lifetime ref, so a Player who left the Card route before the gate opened lost the Moment forever (the ref died on unmount; the returning board baselined the already-standing win). Issue #104 moves the broadcast decision OFF snapshot diffing and ONTO the mark that completed the win — `setMark`'s synchronous transition verdict, consumed in `doMark`, enqueued into a MODULE-scope pending queue that survives unmounts. The snapshot machinery now serves ONLY the cosmetic Celebration animation. See § The action-driven broadcast model (issue #104), which also states which PR #99 hardening invariants are PRESERVED (on the action path), which became MOOT (they were artifacts of snapshot diffing), and which are NARROWED.

## Data model (merged-main type + rules, #16/#18)

A Moment is a top-level doc under the Event: `events/{EVENT_ID}/moments/{momentId}`. The shape is exactly `MomentDoc` minus its `id` (the `id` is the doc id, surfaced on read by `momentConverter`). The rules block allows:

- **create** if `isOwner(request.resource.data.uid)` (a forged uid is denied — you can only broadcast your own beat), the `kind` is one of the three `MomentKind`s, `displayName` is a non-empty ≤100-char string, and `createdAt` is a number **within +60s / -24h of `request.time`** (the same near-now window proofs uses — PR #99 finding 3) so a bad clock or a forged far-future stamp cannot pin a Moment above all Feed activity forever.
- **read** if signed in (public — the Feed everyone watches; no anonymity, ADR 0002).
- **update** — **none. A Moment is fully immutable** (PR #99 finding 4): the update path was removed. `MomentDoc` carries no moderation/status field, so an admin update could only rewrite content — and an admin-writable update defeated the create-only dedup (an admin-player's repeat broadcast became an authorized overwrite). Moment moderation is delete-only; a moderation/status field is deferred to #37/#41.
- **delete** if `isAdmin(eventId)` or the owner.

Two facts about that block are **load-bearing to this design and pinned honestly**:

1. **The doc id is bound to the payload kind on create (#103)** — the event singleton `first_bingo` must carry `kind: 'first_bingo'`, and every other id must equal `${uid}-${kind}`, mirroring the writer's deterministic scheme (`src/data/moments.ts`); see § Once-only semantics. Combined with **update being denied for everyone** (PR #99 finding 4), a second write to an already-written deterministic id is a doc-exists `update`, which no one — not even an admin — can do, so it is denied. This is the strongest dedup the rules allow, and it is what the writer relies on.
2. **The create rule has no `hasOnly()`/`keys()` constraint**, so it does **not** reject a Moment that carries extra `mediaURL`/`proofId` fields. The ADR 0002 "a Moment carries no evidence" guarantee is therefore a **writer + type** contract — `src/data/moments.ts` writes exactly the `MomentDoc` fields, and `MomentDoc` has no media/`proofId` field — **not** a rules-layer one. The rules test pins the actual (permissive) rule behaviour so no one mistakes the rule for enforcing it.

## The change

- `src/data/moments.ts` (new) — three broadcast functions, `broadcastBingo` / `broadcastBlackout` / `broadcastFirstBingo`, each writing one `MomentDoc` row via a raw (converter-free) ref in the mark path's offline-queueable, fire-and-forget style (mirroring `setMark`): the write pends durably in the persistent cache when offline (ADR 0006) and never blocks the celebration UI, and an online rejection is logged (never silently swallowed). All three share ONE write path, `writeMomentOnce` (round 3 finding C): a `getDocFromCache` pre-check skips the write entirely when the Moment already exists locally — otherwise a duplicate deterministic-id `setDoc` would optimistically overwrite the cached copy and its refreshed `createdAt` would pin the old Moment to the Feed top until the server's denial (indefinitely while offline). The payload is `Omit<MomentDoc, 'id'>` — no media, no `proofId`, ever. `displayName` is bounded through the shared `markerDisplayName` helper (imported from `src/data/attribution.ts`, unchanged) to the rules' non-empty ≤100 contract, so a broadcast can never be rejected for a malformed name. Plus `hasPriorBingoWitness(uid)` (round 2 finding D): a cache-only, never-rejecting read of the player's own `${uid}-bingo` Moment doc — the durable witness of a prior win Board consults before queueing a ceremonial First-to-BINGO candidate (see § Review-round hardening (PR #99)). Plus (issue #104) the **module-scope pending-Moment queue** — `enqueueWinMoments`, `enqueueFirstBingoMoment`, `peekPendingMoments`, `clearPendingMoment`, `dropPendingWins`, `pendingActionGeneration` (+ a test-only `resetPendingMoments`) — a per-app+uid `Map` of `{ bingo, blackout, firstBingo }` flags, keyed the same durable way `markChains` is in `api.ts`, plus a SEPARATE per-uid monotonic **action generation** map (PR #110 finding 1: bumped by every `dropPendingWins` — every unmark verdict and every observed passive fall — and never reset by an emptied flags entry, so a stale async continuation can be detected). `dropPendingWins` is the fall-observer clear (clears the fallen kinds — a bingo fall also drops the ceremonial candidate — and always bumps the generation); `clearPendingMoment` is the drain fire-clear (never bumps). A win a mark COMPLETES is enqueued here (off `setMark`'s verdict, in `doMark`); the flags survive Board unmounts and route changes, so the drain that fires them can run from a later mount (see § The action-driven broadcast model).
- `src/data/converters.ts` — a `momentConverter` mirroring `proofConverter`/`claimConverter` (pins `id` to `snap.id`). Genuinely required so `paths.ts` can attach a converter to its refs, exactly as the tally/proofs helpers do.
- `src/data/paths.ts` — `momentsCol()` / `momentRef(id)` converter-attached refs under `events/{EVENT_ID}/moments`, mirroring `proofsCol`/`proofRef`.
- `src/hooks/useData.ts` — `useMoments()` subscribes to the moments collection through the SAME `useColSub` latch pattern the proof stream uses (`{ includeMetadataChanges: true }`, `hasServerData` latched on the first server-backed snapshot), filters out non-canonical Moment ids before render (round 4 finding B), then sorts newest-first and caps. `mergeFeed(proofs, moments, max)` is a pure function that folds both into one newest-first stream capped at `max`. `useFeed()` composes `useProofFeed` + `useMoments` through `mergeFeed`; `loading` stays true until both halves have delivered so the empty state never flashes. `useTally`/`useLeaderboard`/`useProofFeed` are untouched beyond being composed.
- `src/components/ProofFeed.tsx` — now renders the merged Feed from `useFeed()`. A Proof renders exactly as before (report ⚑, owner-delete 🗑, flagged badge, media by type, scheme-guarded via `safeMediaUrl`). A Moment renders distinctly: a celebratory per-kind line with an icon, **no media and no report/delete** (there is no evidence to dispute). The empty copy now reflects the whole Feed.
- `src/data/api.ts` (issue #104, return-shape change ONLY — the write set, batch, serialization chain, and marker logic are untouched) — `computeMark`/`setMark` now also return `bingoTransition` / `blackoutTransition`: the rising EDGE this mark crossed (`bingo`/`blackout` are the STANDING state; the transitions are `previousBingoCount === 0 && bingoCount > 0` and `isBlackout(next) && !isBlackout(cells)`). Computed inside the folded next-state, so the write path is the single source of truth for "this mark won" — the synchronous verdict `doMark` broadcasts the Feed Moment off.
- `src/components/Board.tsx` — broadcasts on the ACTION path (issue #104): `doMark` — behind the shared `cellsAttributable` guard (PR #110 finding 2: a tap on a stale previous-uid render is a no-op, no write and no enqueue) — reads `setMark`'s transition verdict and, on a rising edge, enqueues the matching kind(s) into the module-scope pending queue (the ceremonial first_bingo candidate after the durable-witness check, generation-token-guarded — PR #110 finding 1), then drains with the action's folded `res.cells`. The snapshot machinery (`wasBingo`/`wasBlackout` + the celebrate `useEffect`) serves the cosmetic Celebration animation plus the passive fall observer (PR #110 finding 3) — no direct public writes. Board still subscribes to the roster via `useLeaderboard()` (the SAME data the Leaderboard's First-to-BINGO pin reads) and stores identity + roster + gate signals + the CURRENT attributable `cells` in a latest-value ref feeding `drainMoments` (a stable callback) — the cells are back for fire-time revalidation (PR #110 finding 3): a drain never publishes without re-checking the win against the freshest attributable board, and never adjudicates an empty one. `drainMoments` fires each held Moment once its gate opens: identity for all three (never stamp a stale name into an immutable Moment), a server-confirmed roster additionally for first_bingo. Drains re-attempt from the gate-change `useEffect` (`identityKnown`/`rosterConfirmed`) and from every attributable board snapshot; the queue holds the state across unmounts (see § The action-driven broadcast model). `setMark` and the #87 Tally UI are untouched beyond the return-shape read.
- `src/index.css` — a `.moment` card style (festive tint, distinct from `.proof`; the First-to-BINGO card gets the Leaderboard pin's glow).

## Once-only semantics (deterministic ids + the create-only rule)

The design picks the **strongest dedup the rules allow** and layers it under the mark's transition verdict + the module-scope pending queue (issue #104; the same guarantees held under PR #99's snapshot edge refs, which the action path replaced):

- **First BINGO — once per Player.** Broadcast on the TRANSITION into having a bingo — `setMark`'s `bingoTransition` verdict (`previousBingoCount === 0 && bingoCount > 0`), the synchronous fact that THIS mark completed the first line — never on every completed line or every render. Doc id `${uid}-bingo`. A re-fire (a lose→regain in one session, a reload with a bingo already standing, or a second tab) either never crosses the transition (an unmark can only reduce the mask, and a further mark while a line stands reports `bingoTransition: false`), is skipped by the writer's write-once cache pre-check (round 3 finding C — the Moment is already in the local cache), or hits the deny-all `update` rule on the existing id and is denied (PR #99 finding 4: denied for everyone, admins included).
- **Blackout — once per Player.** Broadcast on `setMark`'s `blackoutTransition` verdict (`isBlackout(next) && !isBlackout(cells)`). Doc id `${uid}-blackout`. On a real 5×5 board the first-line (BINGO) transition always precedes the full-card (Blackout) transition across marks, so a Blackout never suppresses the earlier first-BINGO broadcast.
- **First to BINGO — once per Event, ceremonial and self-reported (ADR 0001).** Doc id is the Event-**singleton** `first_bingo` (it can never collide with a per-Player id: those use a hyphen before the kind and a uid prefix). Board derives it client-side from the known-players roster: it claims First to BINGO only when the roster is **server-confirmed** AND, as far as that confirmed view shows, **no other Player has a `firstBingoAt` yet** (while the roster is unconfirmed the claim is HELD, not guessed — PR #99 finding 2). The singleton id makes it structurally once-per-Event: the first create wins; a later create becomes an `update` and is denied.

  **The regained-line witness (round 2 finding D).** `firstBingoAt` is BY DESIGN volatile — `computeMark` clears it when the last line falls — and the schema deliberately carries no durable first-win field. So the roster check alone would let a Player who had a line, unmarked it, and regained one later mint the event singleton despite not being first EVER (their own history is invisible to `othersBingoed`, which only inspects OTHER players). The fix uses **the Moment collection as its own memory**: the Player's own `${uid}-bingo` Moment is create-only and immutable, so its presence is a durable witness that a prior win existed even after the line was unmarked. Before enqueuing a ceremonial candidate for a bingo-completing mark, `doMark` calls `hasPriorBingoWitness(uid)` — a cache-only `getDocFromCache` read (instant, offline-safe, and same-session writes are visible via latency compensation): witness present → the ceremonial candidate is NOT enqueued (the regained line still enqueues + attempts the plain bingo Moment, which the write-once cache pre-check skips, or the create-only rule denies on a cold cache — fine); cache miss (fresh device) → fall back to the roster check at drain time. **Narrowed residual (accepted, ADR 0001):** a fresh-device, cache-cold, roster-blind regained line — the Player's prior win is neither in this device's cache nor visible as another player's `firstBingoAt` — can still claim the singleton. At friend-group scale this is acceptable for a self-reported ceremony; the honour is recorded, never adjudicated.

**The id↔kind binding makes the deterministic ids enforceable (create-side, issue #103).** The three ids above are a real dedup only if the rules hold a caller to them. The moments create rule binds the doc id to the payload kind: the event singleton `first_bingo` must carry `kind: 'first_bingo'`, and every other id must equal `${uid}-${kind}` — the writer's scheme exactly. Before #103 the id was fully caller-chosen, so any signed-in Player could create `moments/first_bingo` with a *mismatched* kind (e.g. `kind: 'bingo'`). The read-side filter (`hasCanonicalMomentId`) hid such a doc from every Feed, but because a Moment is **fully immutable** (finding 4 — no update path) the squatted singleton could never be overwritten, so the legitimate First-to-BINGO create — which targets that same id — was denied **forever** as a doc-exists update: a denial-of-ceremony squat. (Only `first_bingo` was exposed: the per-Player ids embed the writer's own uid and create already requires `isOwner`, so a peer cannot squat another Player's `${uid}-bingo` / `${uid}-blackout`.) Binding the id to the kind on create closes the hole; the read-side filter (§ Review-round hardening, "Canonical Moment ids before render") **remains as defense-in-depth**, enforcing the same deterministic ids on the render path so read and write never diverge.

**The race, documented honestly.** First to BINGO is a ceremony, not a verified fact (ADR 0001). Under an offline/latency race two Players may each briefly believe they are first — each optimistically shows their own local `first_bingo` Moment from the persistent cache — but exactly one create reaches the server (whichever syncs first), the other is rolled back to the winner, and both clients converge. At friend-group scale this is acceptable; the Feed's `first_bingo` reflects who broadcast first, which under a bad offline race can differ from the Leaderboard pin's earliest-`firstBingoAt`. The honour is recorded, never adjudicated.

## The action-driven broadcast model (issue #104)

**Broadcast source = the mark that completed the win.** PR #99 drove the public Moment write off Board's snapshot diffing: a `wasBingo`/`wasBlackout` edge ref detected a transition across renders, a component-held `pending` ref held it while a gate was closed, and `flushPending` fired it once the gate opened (revalidating the win against the current cells). Five review rounds all traced back to that coupling, and it left one residual bug (issue #104): the `pending` ref was **component-lifetime**, so a win crossed while a gate was still closed was lost if the Player left the Card route before the gate opened — Board unmounted, the ref was discarded, and the returning board baselined the already-standing win.

Issue #104 re-homes the broadcast on the **action path** (hardened by the PR #110 review — see § PR #110 hardening below):

1. **The write path returns the verdict.** `computeMark`/`setMark` (api.ts, return-shape change only) now also return `bingoTransition` / `blackoutTransition` — the rising edge THIS mark crossed, computed inside the folded next-state. `doMark` already consumed `res.bingo` for analytics; it now also reads the transition.
2. **The completing mark enqueues — behind the shared attribution guard.** `doMark` bails outright (no `setMark` write, no enqueue) unless the rendered board is attributable to the active uid — the SAME `cellsAttributable` derivation (`board.uid === uid`) the Celebration baseline uses (PR #110 finding 2: during an account switch a tap can land on a render still showing the previous uid's board). On a rising edge it enqueues the matching kind(s) into a **module-scope pending queue** (`src/data/moments.ts`, keyed by app+uid like `markChains`); the ceremonial first_bingo candidate is enqueued after `hasPriorBingoWitness` — an ASYNC read guarded by the **action generation token** (PR #110 finding 1, P1): the continuation enqueues only if the per-uid generation captured before the await is unchanged (every unmark and observed fall bumps it) AND the pending bingo flag still stands. The queue is module state, so it **survives Board unmounts and route changes** — the headline fix.
3. **The queue drains when the gates open — revalidating at fire time.** `drainMoments` (Board) reads the queue and fires each held Moment whose gate is open: identity-known for all three, a server-confirmed roster additionally for first_bingo (re-deciding `othersBingoed` against it). Before ANY publish it **revalidates the win against the freshest attributable cells** (PR #110 finding 3, restoring the PR #99 round-3 invariant on the queue path): `hasBingo`/`isBlackout` recomputed, a NON-EMPTY attributable board required (`isBlackout([])` is vacuously true — the empty-board guard is load-bearing again). The completing action passes its own folded `res.cells` (the render-current snapshot has usually not echoed yet); gate-open and snapshot drains use the render-current attributable cells. On revalidation failure the drain **HOLDS, never drops** — a drop would race the latency-compensation echo and re-lose held wins, the #104 bug class; the CLEARS for fallen wins belong to the fall's observers (step 4). Drain attempts run from `doMark`, from the gate-change `useEffect` (`identityKnown`/`rosterConfirmed`), and from **every attributable board snapshot** (the cells effect) — which is how a win held across an unmount posts on a later mount, once its board arrives to revalidate against.
4. **Snapshot diffing serves the Celebration UI + the fall observers.** The `wasBingo`/`wasBlackout` refs + celebrate `useEffect` fire the local animation (no direct public writes), and the same effect now also OBSERVES passive falls (PR #110 finding 3): a listener snapshot showing a previously-standing win gone (bingo/blackout true→false — another tab unmarked, or a rules rollback) clears the corresponding queued flag via `dropPendingWins`, which also bumps the action generation. `doMark`'s unmark verdict is the other fall observer. `localActionsPending` and the component-held `pending` ref remain removed.

### PR #110 hardening (Codex review of the action path)

The PR #110 review surfaced one P1 and two P2s in the queue machinery, all fixed:

- **The stale witness continuation (P1).** Between `await hasPriorBingoWitness()` and its continuation the pending win can change: the player unmarks and loses the bingo, the unmark verdict drops the queued flags — and a continuation that re-enqueued on the uid check alone would let a later drain publish the IMMUTABLE event singleton for a win that no longer stands. Fixed with the **action generation token**: `pendingActionGeneration(uid)` (moments.ts) is monotonically increased by every `dropPendingWins` call — every unmark verdict and every observed passive fall — and kept in a map separate from the flags so an emptied-then-recreated flags entry cannot reset it. `doMark` captures the generation immediately before the await; the continuation enqueues only if the generation is unchanged AND the pending bingo flag still stands (the token catches interleaved actions; the flag recheck catches clears the token cannot see). **Fail-closed residual (accepted):** a concurrent gate-open drain that FIRES the plain bingo while the witness read is in flight clears the flag without bumping the generation, so the recheck forfeits the ceremonial candidate for that action — the bingo itself published, nothing false ever posts, and the window is a local cache read.
- **The stale-board tap (P2).** During an account switch the board subscription can still expose the previous uid's board for a render; the snapshot path always guarded with `cellsAttributable`, but the action path initially trusted the setMark verdict without it — a tap in that stale render could fold the previous account's cells into the current uid's board write and broadcast a Moment for the current uid off the previous account's card. Fixed by sharing the guard: `doMark` bails first thing (no `setMark` call at all — the write itself would be wrong) unless `board.uid === uid`.
- **The unrevalidated drain (P2).** A win held behind the identity gate could drain later with no check that it still stood: the module queue is per-tab, but the BOARD changes via the listener — another tab unmarks, or a rollback/deletion snapshot lands — producing no local unmark verdict, so the stale flag survived and the drain published. Fixed on both prongs: `drainMoments` **revalidates at fire time** against the freshest attributable cells (non-empty board required — the round-3 vacuous-`isBlackout([])` guard restored), and the cells effect **observes passive falling edges** (win true→false in listener data) and clears the queued flag via `dropPendingWins`. The drain holds (never drops) on revalidation failure — see the table row above for why. **Zombie residual (accepted):** a flag whose fall this tab never observes (e.g. the win fell while unmounted, and no later snapshot ever shows it standing) idles un-fireable until reload — revalidation blocks it from ever publishing, so the failure mode is silence, not a false Moment.

### The reload residual (documented, accepted)

The pending queue is module state, so it is per-page-load: a queued-but-undrained broadcast **still dies on a full page reload** (not merely a route change — the narrower loss). This is honest and bounded: the deterministic id + the returning board baselining the standing win (the Celebration baseline) mean a lost-on-reload win produces **no duplicate and no spurious fire** — at worst one possibly-lost Moment for a win that straddled a reload. That is the SAME class of loss as the prior unmount bug, strictly narrower (only a full reload, not any unmount / route change). The cross-device / admin-confirmed off-route path stays deferred to #41 (below).

### Non-honor write-path emission (narrowed)

Broadcasts are now driven by a mark's completing verdict on the **honor** path (`doMark` → `setMark`). Two consequences of removing snapshot-broadcasting, stated honestly:

- **proof_required** wins (which complete via `attachProof`, a separate write path) no longer broadcast a Moment. Under the old snapshot machinery they did, incidentally, because the confirmed board snapshot crossed the edge. The Celebration animation still fires for them (snapshot-driven). Emitting the Moment from the proof write path is the same shape of change the § Deferred section defers for admin-confirmed — a write-path emission outside the honor `doMark` path.
- **admin_confirmed** on-route wins likewise no longer broadcast (the confirm materializes via the admin's write, not the player's mark). This ALIGNS on-route with off-route (which never emitted), handing the whole admin-confirmed emission cleanly to #41 rather than leaving an on/off-route inconsistency.

## Review-round hardening (PR #99)

> **History.** This section documents PR #99's SNAPSHOT-DIFFING broadcast machinery. Issue #104 replaced that machinery with the action path above; read § The action-driven broadcast model § "disposition" for which of these invariants are PRESERVED (re-homed), MOOT, or NARROWED. The rules-layer findings (3, 4) and the writer/read-side findings (round 3 C, round 4 B) are UNCHANGED and still current; the client snapshot-machinery findings (1, 2, 5, round 2 A–D, round 3 A–B, round 4 A) describe the retired mechanism.

The PR #99 Codex review ran three rounds. Round 1 surfaced six P2 findings; five are fixed here (the sixth is deferred — see § Deferred: admin-confirmed off-route wins): three client-side (two gates on the broadcast + a per-account reset), two `firestore.rules` boundary defects. Round 2 surfaced four more P2s in the round-1 broadcast machinery (A–D below), all fixed client-side. Round 3 surfaced three consequences of the round-2 fixes — the swallowed pre-confirmation offline win (folded into the cache-hydration rule below), the vacuously-true empty-board revalidation (folded into the held-flags rule), and the optimistic duplicate overwrite (the write-once pre-check) — all fixed client-side.

**Held broadcasts — gated, revalidated, per-account (round 1 findings 1 + 2 + 5, round 2 findings A + B).** A crossed BINGO/Blackout edge no longer writes directly. It enqueues the matching kind(s) into a per-uid `pending` ref; `flushPending` writes each only once its gate is open — and only if the win STILL STANDS at fire time (a held flag is a candidate, not a promise). The deterministic doc id keeps a held-then-fired broadcast idempotent, and the celebration UI still fires on the edge itself (never held).

- **Identity gate (finding 1).** All three broadcasts require a KNOWN saved identity — the same `identityKnown` tri-state `setMark` gates on. During the player-row loading window `displayName` has fallen back to the auth/Google name; stamping that into an IMMUTABLE Moment would publish a returning Player's stale name forever. So an edge crossed while identity is unknown holds; when identity resolves, `flushPending` fires it once with the saved name.
- **Roster gate (finding 2).** First-to-BINGO additionally requires a **server-confirmed** roster (`useLeaderboard().hasServerData`). An initial empty `players` from a still-loading (or cache-only) subscription is not proof nobody has bingoed; claiming the ceremony off it would falsely pin a non-first Player. So the first_bingo candidate holds until the roster is server-confirmed, then re-decides `othersBingoed` against it. The per-Player BINGO does not wait on the roster — only the ceremonial claim does.
- **Held flags follow the board — dropped when the win falls (round 2 finding A, hardened by round 3 finding B).** A flag queued while a gate was closed could go stale: the Player unmarks and LOSES the win before the gate opens, and round 1 would then post a Moment for a board that no longer had one. Two complementary fixes, both required: the **falling edge** (bingo/blackout going true→false) clears the corresponding pending flag (a bingo fall also clears the ceremonial candidate), handling the common case; and `flushPending` **revalidates at fire time** — it recomputes `hasBingo`/`isBlackout` from the cells the component CURRENTLY holds (carried in the same latest-value ref as the identity/roster) and drops, without firing, any flag whose win no longer stands (first_bingo included: no standing bingo → no ceremonial claim). The revalidation itself requires a **real, non-empty, attributable board** (round 3 finding B): `isBlackout([])` is vacuously TRUE (`[].every(Boolean)`), so without a `cellsNow.length > 0` gate a held blackout would fire against a board that was deleted or de-attributed between the hold and the flush; `hasBingo([])` is already false, but the guard is uniform on both paths so neither rests on a vacuous-truth accident. The revalidation is the invariant; a lose→regain crosses a fresh rising edge and legitimately re-queues, yielding exactly one write when the gate opens.
- **Per-uid reset, synchronous in render + attribution-gated (finding 5, hardened by round 2 finding B).** The edge refs (`initialized`/`wasBingo`/`wasBlackout`) and the `pending` ref are per-account. Round 1 reset them in a `useEffect` keyed on `uid` — but on a uid switch while Board stays mounted, `useBoard`/`useMyPlayer` still return the PREVIOUS uid's rows for the render(s) before their keyed effects clear, so the same commit's cells effect could seed the new account's baseline from the OLD board and make the new user's first real snapshot read as a fresh edge. Two fixes, both required: the reset now runs **synchronously during render** (React's adjust-during-render pattern — a ref tracks the uid the edge state belongs to; a mismatch restores the uninitialized baseline before ANY effect runs, idempotently); and the cells effect **seeds from attributable data only** — `BoardDoc` carries its owner's uid, and a board whose `uid` differs from the current account is ignored entirely (it also contributes NO cells to fire-time revalidation). The stale-render board is therefore never seeded, and the new uid's own first snapshot re-establishes init-without-firing.

**Cache hydration is baseline, not a transition — except the player's own action (round 2 finding C, bounded by round 3 finding A).** Under the ADR 0006 persistent cache the first snapshot(s) after mount can be cache-only, and round 1 seeded the edge baseline from the FIRST snapshot regardless — so a stale cache lacking a BINGO the server already had made the server confirmation read as a live transition and post a Moment for a win that already stood. Now, while the board is NOT server-confirmed (`useBoard().hasServerData` false), every PASSIVE snapshot re-seeds `wasBingo`/`wasBlackout` and never fires an edge (`initialized` stays false); the FIRST server-confirmed snapshot is baseline too (init-without-firing — any standing win it reveals already stood server-side); edge DETECTION runs only after it. Live play under a confirmed board is unaffected: `hasServerData` is a latch that never un-sets, and a local optimistic Mark arrives as the next snapshot via latency compensation and fires normally.

**The exception (round 3 finding A, hardened by round 4 finding A): the player's OWN Mark on a still-unconfirmed board is a live edge, not hydration.** The round-2 rule as stated swallowed a win completed OFFLINE: on an offline reload the board never confirms until reconnect, so the player's own just-completed line was baselined — the edge never fired, even after reconnect (the first confirmed snapshot baselines it as already standing) — violating the offline-queueable Moments AC. Board already knows when a snapshot is the player's own doing: `doMark` increments a `localActionsPending` ref (decremented on the synchronous-throw path where no write happened), and the cells effect consumes one marker per snapshot. This is deliberately a count, not a boolean: two fast offline taps can produce two local snapshots before server confirmation, and the second snapshot may be the one that completes BINGO. A snapshot marked local while the board is unconfirmed falls through to LIVE edge detection against the pre-action baseline (by the time a tap is possible the cells have rendered, so a pre-action baseline always exists), and the queued Moment rides the offline queue exactly as designed — `setDoc` pends and drains on reconnect. Passive unconfirmed snapshots keep the baseline behavior, and the later server confirmation re-baselines the same standing state without firing a second edge (`initialized` is still false until confirmation). Scope honestly stated: the local-action marker covers `doMark` — the honor-mode path, which is the only Mark that can commit offline (`attachProof`/claim resolution ride transactions that reject offline), so the proof-required and admin-confirmed paths cannot reach this window.

**Canonical Moment ids before render (round 4 finding B; create-side counterpart added in #103).** The create rule now binds the doc id to the payload kind (§ Once-only semantics, #103), so a forged singleton like `{ kind: 'first_bingo' }` at `moments/spoof` is rejected at write. The read side keeps `hasCanonicalMomentId` as **defense-in-depth**: `useMoments()` filters before sorting/capping so `first_bingo` renders only at the event singleton id `first_bingo` and per-Player moments only at `${uid}-bingo` / `${uid}-blackout` — hiding, e.g., the harmless `${uid}-first_bingo` orphan the create rule still permits (its id matches `${uid}-${kind}`). Read and write now honor the SAME deterministic-id invariant the writer depends on.

**createdAt bounded near request.time (finding 3).** The moments create rule now requires `createdAt` within the same +60s / -24h window of `request.time` that proofs use, so a bad clock or a forged far-future stamp can no longer pin a Moment above all Feed activity forever. The client keeps writing `Date.now()` — in bounds by construction for an online write. **Offline-drain residual (accepted, ADR 0001):** a Moment queued offline drains with its ORIGINAL `Date.now()` stamp, not the reconnect time, so one created and then held offline for **more than 24h** would trip the lower bound and be rejected on drain. At friend-group cruise scale a >24h offline gap before a BINGO/Blackout Moment reaches the server is implausible, and the honest alternative — an unbounded stamp — is the defect finding 3 closes; the bound wins. (Proofs cannot queue offline at all — `attachProof` rides a `runTransaction` that rejects offline — so they never hit this drain case; Moments, which ride an offline-queueable `setDoc`, do, hence this note.)

**Full immutability — no admin overwrite (finding 4).** The admin `update` allowance was removed: a Moment is now immutable for everyone. The create-only dedup is only sound if a re-broadcast to an existing id is denied for ALL callers; the prior admin-writable update let an admin-player's repeat broadcast become an authorized overwrite — including clobbering the `first_bingo` singleton under a stale-roster race. `MomentDoc` has no moderation/status field (only `kind`/`uid`/`displayName`/`photoURL`/`createdAt`), so an admin update could only rewrite content — there is nothing to moderate in place. Moment moderation stays delete-only; introducing a status field + a moderation-only update diff (`request.resource.data.diff(resource.data).affectedKeys().hasOnly([...])`) is deferred to #37/#41, which own that schema evolution.

**The write-once cache pre-check (round 3 finding C).** The create-only rules deny a duplicate deterministic-id write server-side — but Firestore applies **latency compensation first**: the duplicate `setDoc` briefly overwrites the LOCAL cache copy, and its refreshed `createdAt` pins the old Moment to the top of the Feed until the denial rolls it back — indefinitely while offline, where no denial ever arrives. So the writer extends the witness pattern to ALL broadcasts: `writeMomentOnce` (the single write path behind `broadcastBingo`/`broadcastBlackout`/`broadcastFirstBingo`) first reads the deterministic ref with `getDocFromCache`; if the Moment already exists locally the write is **skipped entirely** (a debug log — this is the designed once-only path, not an error). On a cache miss the write proceeds: a cold-cache duplicate still resolves server-side via the create-only rule, and the visible-flash window only exists when the doc IS cached — exactly the case the pre-check catches.

**The regained-line witness (round 2 finding D)** is specified with the once-only semantics — see § Once-only semantics, "The regained-line witness", including the narrowed fresh-device residual.

## Deferred: admin-confirmed off-route wins (→ #41)

**The gap (finding 6).** In `admin_confirmed` claim mode a Mark starts pending and only becomes a win when an Admin resolves the Claim. No Moment is emitted for it: the broadcast is now driven by the WINNER's own mark (`doMark`'s transition verdict, issue #104), but an admin-confirmed win materializes via the ADMIN's write, not the player's mark — so the player's `doMark` never sees the transition. Under this ticket a confirm-time win therefore lands on the leaderboard/stats without ever posting the celebratory Moment. (Before #104 the on-route case emitted incidentally, via the mounted board's snapshot edge; #104 removed that snapshot-broadcast, so on-route and off-route are now consistent — the whole admin-confirmed emission is #41's.)

**Why it is not fixed here.** Emitting from the confirm write-path needs one of two things outside this ticket's file set and rules scope: (a) rules that let an Admin create a Moment for ANOTHER uid — but the moments create rule is own-uid-only by design, and finding 4 just made the collection strictly own-create + immutable, so widening it to cross-uid admin creates is exactly the change that reopens the dedup/forgery surface; or (b) a player-side listener on the Claim outcome that lives OUTSIDE the Card route so it fires wherever the Player is. Both are claim-lifecycle design owned by #41 (the confirm-path/claims ticket), not the Feed-Moments surface.

**Invariant #41 must satisfy.** A confirmed win must produce the SAME Moment the live edge would have — the same kind at the same deterministic id, attributed to the WINNER (not the confirming Admin) — exactly once. The deterministic-id, create-only, fully-immutable dedup this ticket ships must remain the backstop: whichever path emits (live edge or confirm-path), a duplicate is denied, so the two paths cannot double-post.

## Claim → test

Basename-aligned to this spec (the checker matches `specs/w2-feed-moments.md` → a `w2-feed-moments.test.*` under `tests/**` or `src/**`); every claim maps to a real assertion.

### Rules — a Player broadcasts their OWN Moment; forgery denied; fully immutable; createdAt bounded; extra fields not rejected

Runner: `npm run test:rules` (Firestore emulator). Test: `tests/rules/w2-feed-moments.test.ts`.

- A signed-in Player may create their own Moment (each of the three kinds) at a caller-chosen id; a forged uid (another Player's beat) is denied.
- The shape contract: an invalid `kind`, an empty `displayName`, an over-100-char `displayName`, and a non-numeric `createdAt` are each denied; exactly-100 is allowed.
- The once-only backstop (PR #99 finding 4): a second write to an already-written deterministic id is a doc-exists `update` and is denied for **everyone** — an owner re-broadcast, an admin repeat-broadcast, and an admin content rewrite all fail. This is what makes the deterministic id structurally once-only, and it holds for admin-players too.
- `createdAt` is bounded near `request.time` (PR #99 finding 3): a near-now stamp is accepted, a far-future stamp (+1h) and a far-past stamp (-2d) are each denied — the same +60s / -24h window proofs use.
- **Honesty pin:** a create carrying extra `mediaURL`/`proofId` fields is **accepted** — the rules do not reject it; the no-evidence guarantee is a writer/type contract (see the unit test), not a rules-layer one.
- Moment reads are public; an owner may delete their own Moment and an admin may delete any; a non-admin peer may not.

### Unit — the broadcast writer + the Feed merge

Runner: `npm test` (Vitest, jsdom). Test: `src/data/w2-feed-moments.test.ts`.

- `broadcastBingo` writes exactly ONE Moment at `${uid}-bingo` with `kind: 'bingo'` and EXACTLY the `MomentDoc` fields — no `mediaURL`, `storagePath`, or `proofId` (ADR 0002).
- `broadcastBlackout` writes one at `${uid}-blackout` (`kind: 'blackout'`); `broadcastFirstBingo` writes one at the Event-singleton `first_bingo` (`kind: 'first_bingo'`).
- The attributed name is bounded to the rules' ≤100 cap.
- Each broadcast is a single offline-queueable `setDoc` — never `addDoc`, never `runTransaction`.
- The write-once cache pre-check (round 3 finding C): a broadcast whose Moment is already in the local cache performs NO `setDoc` at all; a cached exists-false snapshot and a cache miss (rejection) both let the write proceed — the cold-cache duplicate still resolves server-side via the create-only rule.
- `hasPriorBingoWitness` (round 2 finding D) reads the player's own `${uid}-bingo` doc from the LOCAL cache only (`getDocFromCache`), resolves `true` when the cached witness exists, `false` when it does not, and `false` — never rejecting — on a cache miss (the fresh-device fallback path).
- **The pending-Moment queue (issue #104):** `enqueueWinMoments` sets the per-uid `{ bingo, blackout }` flags off a mark's transition verdict (a no-op enqueue with neither transition never resurrects a drained flag); `enqueueFirstBingoMoment` sets the ceremonial flag separately (Board gates it on the witness first); the queue is per-uid isolated and SURVIVES across calls (module state — an unmount between enqueue and drain cannot lose it); `clearPendingMoment` clears one drained kind and drops the entry once empty; `resetPendingMoments` clears all (test-support).
- **The action generation + fall drops (PR #110 finding 1):** `pendingActionGeneration` starts at 0 per uid and is bumped by EVERY `dropPendingWins` call (an unmark that dropped nothing still bumps) but never by a fire-clear (`clearPendingMoment`); `dropPendingWins` clears the fallen kinds (a bingo fall also drops the ceremonial candidate); the generation survives the flags entry being emptied and recreated (a stale continuation cannot false-match); generations are per-uid and cleared by `resetPendingMoments`.
- **The transition verdict (issue #104)** that drives all of the above is unit-tested where it is computed — `src/data/w1-board-mark-win.test.ts`: `computeMark`/`setMark` report `bingoTransition` only on the mark that completes the first line (not a further mark while a line stands, not an unmark), `blackoutTransition` only on the mark that fills the card, and an `admin_confirmed` pending mark crosses neither (→ the confirm-path emission is #41's).
- `mergeFeed` interleaves Proofs + Moments strictly newest-first, keeps the `slice` cap, and returns an empty stream for `([], [])` — a bare Mark (neither a Proof nor a Moment) contributes nothing.

### Component — Board broadcasts on the action path; the merged Feed renders

Runner: `npm test` (Vitest, jsdom). Test: `src/components/w2-feed-moments.test.tsx`. Under the action model (issue #104) wins are driven by CLICKING a Square (the mocked `setMark` returns the win-transition verdict), and a passive board re-render is asserted to broadcast NOTHING — the prior snapshot-diff cases are replaced by click-driven equivalents, or by the structural invariant that survived.

- Board broadcasts a first BINGO Moment exactly once on the completing MARK (a click whose `setMark` verdict has `bingoTransition`) — not on initialisation (a returning Player whose bingo already stood must not re-announce it) and not on a passive re-render reflecting the mark.
- A bare Mark (a click) that completes no line broadcasts nothing.
- Board claims First to BINGO only when the roster shows no other Player has bingoed yet; when another Player already has a `firstBingoAt`, the Player's own first BINGO still posts but the ceremonial First-to-BINGO does not.
- A Blackout broadcasts one blackout Moment on the full-card MARK without re-announcing the BINGO.
- **Identity gate (PR #99 finding 1):** a win completed while identity is UNKNOWN (the player row still loading) does NOT broadcast — the stale auth name is never written to an immutable Moment; when identity resolves, the gate-open drain revalidates against the echoed standing win and fires it exactly once with the SAVED player-row name (the held-win-still-standing → drains-once case, PR #110 finding 3).
- **Held broadcast survives an unmount (the headline #104 fix):** a win held while identity is unknown, then a Board UNMOUNT, then a fresh MOUNT with identity resolved → the module-queued win drains on mount exactly once with the saved name. This is the case the old component-ref queue lost.
- **Roster gate (PR #99 finding 2):** a win completed while the roster is UNCONFIRMED holds the ceremonial First-to-BINGO (the own BINGO still posts); once the roster is server-confirmed, First-to-BINGO fires only if no other Player has an earlier `firstBingoAt`, and stays held-then-dropped if one does.
- **Action-driven falling edge (round 2 finding A, PRESERVED):** a win completed while identity was unknown, then UNMARKED before identity resolves, fires NOTHING when the gate opens (the unmark's verdict cleared the pending flag — the queue is verifiably empty); a complete → unmark → remark → resolve sequence fires exactly ONE write (the remark re-enqueues on its fresh verdict).
- **Stale witness continuation (PR #110 finding 1, P1):** a win whose durable-witness read is IN FLIGHT when the player unmarks and loses it enqueues NO ceremonial candidate when the read resolves — the action generation changed and the pending flag is gone — and nothing publishes for the fallen win; the clean path (no interleaved action) enqueues exactly once.
- **Stale-board tap (PR #110 finding 2):** a tap landing on a render that still shows the PREVIOUS uid's board mid-account-switch is a no-op — no `setMark` write at all, both uids' queues stay empty, nothing broadcasts.
- **Cross-source fall (PR #110 finding 3):** a win held behind the identity gate, then unmarked by ANOTHER SOURCE (the fall arrives as a passive falling-edge snapshot, no local verdict) never publishes — the observer clears the queued flag, and the gate-open drain finds nothing.
- **Regained-line witness (round 2 finding D):** when the durable `${uid}-bingo` witness is in the local cache, a regained line still enqueues + attempts the plain bingo Moment write but does NOT re-queue the ceremonial First-to-BINGO candidate.
- **Offline win on an unconfirmed board (round 3 finding A, now natural):** the player's own tap completing a line on a cache-only board fires exactly one Moment (with saved-name gating intact) directly from the mark's verdict — the drain revalidates against the action's own folded `res.cells`, so board confirmation is not required — and the later passive server confirmation of the same standing win fires nothing twice. The `localActionsPending` count (round 3 A / round 4 A) is MOOT.
- **Passive snapshots never broadcast (structural — subsumes findings 5, B, C, round 3 passive):** a cache-only → server-confirmed board revealing a standing bingo, and an account switch onto a new uid's standing-bingo board, both broadcast NOTHING — only a mark (a click) broadcasts.
- **Empty-board guard (round 3 finding B, PRESERVED on the drain path — PR #110 finding 3):** a held blackout whose board is GONE at gate-open publishes NOTHING — the drain neither fires (`isBlackout([])` is vacuously true) nor drops (the win may still stand server-side) against an empty board; when the board returns with the blackout standing, it fires exactly once.
- ProofFeed renders Moments and Proofs merged newest-first (fed an already-merged list; the sort itself is `mergeFeed`'s unit test), each distinctly; a Moment carries no media and no report/delete affordances, a Proof keeps them; an empty Feed shows the empty state.
