# Google sign-in & AuthContext deal-error hardening (w1-auth-google)

Feature: Google is the only login (PRD non-goal: no non-Google login), and the client-driven honor-system deal (ADR 0001) that runs on sign-in no longer fails silently. When `joinAndDeal` throws â€” chiefly the ADR-0003/0004 `dealBoard` guard that rejects a Board when the active non-free Prompt pool is below the 24 a card needs â€” the Player sees a Player-worded retry surface instead of the blank Board the old `App.tsx` `.catch(() => {})` left behind.

## Contract

- `src/auth/AuthContext.tsx` owns the post-sign-in deal. It runs `joinAndDeal` when the User becomes known, fires `track('join_event')` on success, and exposes `dealError: string | null`, `dealing: boolean`, and `retryDeal(): void`. Every production endpoint uses an immutable first-party auth handler: `gaycruisebingo.vercel.app`, `gaycruisebingo.com`, and `gaycruisebingo.firebaseapp.com` pin `authDomain` to their own hostname at Firebase initialization. A signed-out `gaycruisebingo.web.app` visitor is history-replaced once to the same-project `firebaseapp.com` app before authentication; that stable callback is already authorized and does not depend on the blocked custom domain. The handoff fires on every signed-out settle â€” first load and a mid-session sign-out alike â€” and every navigation path to it (the auth-settled branch, the bounded settle timer, and the sign-in tap fallback) routes through one deduplicated chokepoint, so at most one cross-origin `location.replace` fires per document and no auth transaction ever starts on `web.app`. The chokepoint computes its target from the live location at navigation time, so a mid-session sign-out lands on the route, query, and hash the Player was actually on â€” never a stale mount-time snapshot. While an app-owned redirect sign-in return is completing on the current origin (the same-origin pending marker was present at mount and `getRedirectResult` has not settled), the handoff is suppressed and the settle timer re-arms once the return settles, so the navigation can never interrupt a returning sign-in mid-completion. Mobile browser tabs use one top-level `signInWithRedirect` transaction so iOS Safari never creates a popup tab that loses Firebase's initial session state or closes without returning to the app; "mobile" includes iPadOS Safari's desktop-UA masquerade, detected as `platform === 'MacIntel'` with `maxTouchPoints > 1` â€” an accepted tradeoff under which a future touch-enabled Mac would also get redirect sign-in (benign: redirect is fully supported on desktop), revisited when a real capability signal distinguishes the two. Installed PWAs, desktop browsers, and the emulator keep `signInWithPopup`; the standalone PWA window provides the stable opener that the browser-tab failure lacks. The provider coalesces concurrent sign-in calls into one Firebase transaction, finishes a redirect result once after reload, and preserves the 18+ acknowledgement that gated the original tap. Redirect completion does not depend on the same-origin pending marker: `getRedirectResult` is consulted on every mount, and a non-null result â€” which only an actual redirect return can produce â€” restores the `login` event and the checked 18+ attestation even when the marker was lost mid-redirect; the marker scopes failure reporting, so a redirect rejection emits `login_failed` only when the marker proves an app-owned redirect was in flight, and an ordinary mount can emit neither `login` nor `login_failed`. Success fires `track('login', { method: 'google' })`; failures fire `track('login_failed', { method: 'google', code })` with an allowlisted Firebase code and no raw provider message. Google remains the only provider.
- Each deal attempt is tied to the latest launch. A late result from a superseded attempt â€” the initial deal is still in flight when the Player signs out and a different Google account signs in â€” is dropped, not applied, so it can never clobber the current User's board or error surface; auth changes also clear any stale `dealError`/`dealing` so the incoming User never flashes the previous account's surface.
- A raw deal failure is translated to Player-facing copy: the pool-below-24 guard yields a "prompt pool is below the 24 a card needs" message; any other failure yields a connection-worded fallback. The same classifier also sets a TYPED `dealErrorReason` (`'pool-shortfall' | 'connection' | null`) in lockstep with the message, so the pool-recovery auto-retry can arm on the cause rather than the copy (see `specs/w1-deal-auto-retry.md`). Both are recoverable via `retryDeal`, which re-invokes `joinAndDeal` for the current User with no full reload. The prior error stays visible while the retry is in flight (the surface is never unmounted mid-retry, so the Player is never dropped onto a blank Board); `dealError` is replaced only when the new attempt settles â€” cleared on success, updated on a fresh failure.
- `src/App.tsx` no longer swallows the deal error. Once a User is signed in, a non-null `dealError` renders `<DealError>` as the Card tab's route content â€” the failure is never hidden behind a blank Board â€” while the app shell, Nav, and every other route stay mounted. In particular the `/items` Prompts tab remains reachable, because adding Prompts there is exactly how the pool-below-24 shortage is recovered; a full-screen takeover would lock the admin/player out of the fix.
- `src/components/SignIn.tsx` exports `DealError`, the retry surface: it shows the message with `role="alert"` and a Retry button that calls `onRetry`, disabled and relabeled while `retrying`. It owns the MANUAL retry; the AUTOMATIC pool-recovery retry lives at the app shell (`src/components/PoolRecoveryWatcher.tsx`, mounted in `AuthProvider`), not on this panel, so it survives the navigation to `/items` that unmounts the panel â€” see `specs/w1-deal-auto-retry.md`.

## Acceptance criteria

- Given the active Prompt pool has fewer than 24 non-free Prompts, when a User signs in and `joinAndDeal` throws, then the Player sees a retry surface on the Card tab explaining the deal failed (not a blank Board).
- Given a deal failure is showing, when the Player uses the still-mounted Nav, then the other tabs â€” the `/items` Prompts tab in particular â€” remain reachable so Prompts can be added to recover, and returning to Card still shows the retry surface.
- Given a signed-in User whose deal failed, when the Player taps Retry, then `joinAndDeal` is re-invoked without a full reload, the retry surface stays mounted (disabled `Dealingâ€¦` button) while the retry is in flight, and the surface clears once a deal succeeds.
- Given a deal is in flight when the Player signs out and a different Google account signs in and deals, when the original deal settles late, then its result is ignored and the new account's board / error surface is untouched.
- Given a deal failure that is not the pool guard, when it is surfaced, then the Player sees the connection-worded fallback copy.
- Given a signed-out Player opens `gaycruisebingo.web.app`, when Firebase settles the empty session, then the app history-replaces to the same path on `gaycruisebingo.firebaseapp.com` before rendering SignIn, so the Player sees one acknowledgement and starts no auth transaction on `web.app`. If Firebase stalls while the browser remains online, the same handoff fires after a bounded 3-second persistence-settle window; an offline transition cancels the timer, and a restored current User suppresses it, preserving cached sessions.
- Given a signed-in `web.app` session, when the Player signs out mid-session, then the same handoff fires â€” the signed-out settle lands on the canonical origin rather than a `web.app` SignIn whose tap would hand off anyway â€” and a sign-in tap racing the started handoff neither navigates a second time nor starts an auth transaction on `web.app`.
- Given an app-owned redirect sign-in return is completing on the current origin, when the settle window elapses first, then the handoff is deferred rather than interrupting the return, and it re-arms once the redirect result settles signed-out.
- Given a Player completes a mobile redirect sign-in but the same-origin pending marker was lost mid-redirect, when the app mounts with the returned redirect result, then `track('login', { method: 'google' })` fires and the checked 18+ acknowledgement is persisted anyway; an ordinary mount (null redirect result, no marker) emits neither `login` nor `login_failed`.
- Given a Player is on a mobile browser at a first-party auth origin, when they tap sign in, then exactly one top-level redirect starts and no popup window opens or closes.
- Given sign-in is already pending, when another caller invokes sign-in, then it receives the existing operation and Firebase is called only once.
- Given sign-in succeeds, when the User authenticates, then `track('login', { method: 'google' })` has fired.
- Given the Google popup rejects, when the failure surfaces, then `track('login_failed', { method: 'google', code, â€¦ })` fires with the Firebase error code and the error is rethrown to the caller (the success `login` event and attestation do not run).
- Given a healthy pool, when the User signs in, then the Board is dealt and `track('join_event')` fires with no error surfaced.

## Test coverage

`src/auth/AuthContext.test.tsx` (Vitest RTL-jsdom) mounts the real `AuthProvider` with the Firebase boundary mocked and drives the auth callback directly:

- A pool-below-24 `joinAndDeal` rejection surfaces the "24 a card needs" message; tapping Retry re-invokes `joinAndDeal` (a second call), clears the error, and fires `track('join_event')` â€” covering the healthy-deal path on the retry.
- A non-guard rejection surfaces the connection-worded fallback.
- `signIn` fires `track('login', { method: 'google' })` on success; on a popup rejection it fires `track('login_failed', â€¦)` with the Firebase code and rethrows (no `login`, no attestation); iOS uses one redirect and no popup; repeated calls start one transaction; redirect completion persists the acknowledgement once â€” with or without the pending marker, while an ordinary mount's null result and a marker-less rejection emit nothing; and `web.app` history-replaces to `firebaseapp.com` before authentication â€” on the empty-session settle, on a mid-session sign-out, and from a sign-in tap, all through one deduplicated handoff that a pending app-owned redirect return defers (with the settle timer re-arming on settlement).
- A deal left in flight across a sign-out and a sign-in as a different account is dropped when it rejects late (the stale-attempt guard), leaving the new account's surface untouched; and a retry keeps the error surface mounted (dealing) until it settles rather than flashing a blank Board.

`src/w1-auth-google.test.tsx` (Vitest RTL-jsdom) covers the App wiring and the retry affordance:

- With a `dealError` present, `App` renders the `role="alert"` retry surface as the Card tab's content inside the still-mounted shell (`.app` and `.nav` both present), and its Retry button invokes `retryDeal`.
- With the error active, tapping the Prompts tab navigates to `/items` and mounts the ItemPool page (the recovery path), and tapping Card again re-shows the retry surface.
- `DealError` renders the message as an alert, calls `onRetry` when Retry is tapped, and disables/relabels the button while a retry is in flight.
