# W4 — Private bug-report inbox

## Behavior

A signed-in Player can open a persistent, unobtrusive lower-right bug-report control above the fixed tab bar from every authenticated route. The control uses the [Lucide Bug](https://lucide.dev/icons/bug) outline (ISC license) as an inline, theme-token-colored SVG, remains a 44×44 px target on phones, expands to a labeled pill on wider screens, shifts above the install prompt, and disappears while a sheet or celebration is open. It is app-shell utility chrome, not a primary tab.

`BugReport` supports this as its default `variant="floating"` presentation, still exercised in full by this spec's own tests. As of Phase 1.5 (#208, daily-cards-spec § "More menu") the LIVE app mounts only `variant="row"` instead — a plain menu row inside the More tab, replacing the fixed floating control (`App.tsx` no longer mounts the default variant). The trigger+sheet flow itself (capture, submission, retry) is identical in both variants; see specs/d15-more-menu.md § Contract for the relocation.

Because that relocation left the only launcher on `/more`, capture-at-open could only ever photograph the More menu (#324). The flow is therefore split: `BugReport` (either variant) is just the launcher, and `BugReportProvider` — mounted once at the app shell inside `.app` (`App.tsx`) — owns the sheet, the capture state, and a pick-a-screen step. From the sheet's capture-ready or capture-failed state, "Capture a different screen" parks the sheet behind a fixed bar above the tab bar; the reporter navigates to any tab (or opens another sheet — the bar sits above the generic sheet layer so that surface is itself capturable), then captures it or steps back (button or Escape). The description draft and any prior capture survive throughout — including across the launcher's own unmount when leaving `/more` — and re-tapping a launcher mid-pick recalls the parked sheet rather than discarding the draft. Submission reports the route the attached screenshot was captured on, not the submit-time pathname.

Opening the control captures only the rendered `.app` surface while excluding the reporting UI. If full-surface capture fails, the client retries with a compatibility pass that lowers pixel ratio and excludes embedded media surfaces that desktop Safari and stale canvases commonly cannot serialize, then falls back to a viewport-sized canvas renderer that avoids the SVG/foreignObject path used by the first two attempts. The sheet explains which diagnostics are sent and previews the screenshot. A capture failure never discards the description: the Player can retry capture or submit text only. Submission includes bounded description, current app-relative route, event id, deployed commit, user agent, viewport, online state, optional capture error, and optional PNG. It includes no email, token, console log, local-storage value, proof URL, browser chrome, or other application.

The authenticated `submitBugReport` callable is the only write path. It validates the schema and real PNG signature, rate-limits a uid to three reports per rolling 15-minute window, hashes the uid before storing report metadata, and writes screenshot evidence privately with no browser read path. When `BUG_REPORT_APP_CHECK` is enabled after #44, a missing App Check token fails closed.

An operator manually runs `npm run bugs:pull` to atomically materialize new immutable reports under the gitignored `.github/bug-reports/inbox/`. The exporter is idempotent across inbox and imported reports and removes partial output after a malformed report or failed download. Import dedupe is durable across fresh clones and deleted worktrees: `bugs:archive` appends each successful immutable issue receipt to the committed `.github/bug-reports/imported-ledger.jsonl` ledger before moving the report to `imported/`, and `bugs:pull` skips any report ID already recorded there even when the local gitignored `imported/` tree is absent. Ledgered report IDs are skipped before validating the current Firestore source fields, so a historical imported report cannot start failing pulls just because its immutable source document no longer matches the current exporter schema. The ledger stores only opaque report IDs, issue numbers/URLs, and import timestamps, never report descriptions, screenshots, reporter hashes, or other private evidence. Malformed, incomplete, or conflicting ledger rows fail closed instead of being ignored, so a corrupt durable record cannot silently re-open duplicate imports or suppress a report without a valid receipt. If a stale local `inbox/<id>/` directory exists for a ledgered report, `bugs:archive` may clean it up only when the requested issue matches the committed ledger receipt; a different issue target fails before moving anything. The LLM workflow requires confirmation before GitHub writes, distinguishes evidence from inference, deduplicates related reports, and archives each imported source with an issue receipt through `npm run bugs:archive`.

## Testing

- `src/components/w4-bug-report-inbox.test.tsx` covers accessible placement, capture preview/fallback, submission metadata, retry, and success/error states, plus the pick-a-screen flow (#324): parking/reopening the sheet, Escape stepping back before closing, replacement-capture failure preserving the prior screenshot, launcher-unmount survival with the draft intact, and capture-route submission.
- `tests/e2e/d15-tab-more-menu.spec.ts` (the local-only Playwright layer) drives the real pick flow end-to-end: open from More, park, navigate to Card past the unmounted launcher, capture with real html-to-image, and confirm the reopened sheet keeps the draft.
- `tests/functions/bug-reports.test.ts` covers server validation, PNG verification, and rate-limit boundaries.
- `src/data/w4-bug-report-export.test.ts` covers atomic export, idempotency, malformed screenshots, archive receipts, durable ledger skip without a local imported tree, skipping ledgered reports before source validation, fail-closed malformed/incomplete ledger rows, dedupe after the imported tree is wiped, archive partial-failure retryability when the ledger append fails, stale inbox conflict rejection against the committed ledger, stale inbox cleanup when the issue target matches the ledger, and idempotent self-healing ledger append for pre-ledger imports.
- `tests/rules/w0-firestore-rules.test.ts` and `tests/rules/w0-storage-rules.test.ts` prove direct Firestore and Storage access is denied.
