# More menu contents (d15-more-menu)

Feature: the Phase 1.5 build-out of the More tab's actual content, implementing `plans/daily-cards-spec.md` § "More menu (⋯ tab)". `#203` (`specs/d15-tab-contract.md`) is the thin mount-point revision that gives the More tab its avatar icon and carries an interim placeholder (`ProfileEditor` avatar + sign-out button only) so neither affordance regressed while this ticket was outstanding. This ticket replaces that placeholder wholesale with the full menu: a Profile card, Theme (gaining the new "Auto — match the day" default), a Play section (schedule, suggest a square, how to play, install), a Support section (bug report, 18+ advisory), an admin-only Admin row badged with the pending-approvals count, sign out, and a version footer. `ItemPool` — no longer a routed tab since `#203` — mounts here as a sub-panel; `Admin` did too until `admin-console-ia` made it the ROUTE-driven console at `/more/admin` (the Admin row navigates there); `BugReport` and `AcceptableUse` relocate their triggers here too, off their former fixed-position/standalone mounts.

## Contract

- `src/components/More.tsx` — renders the menu in fixed spec order: Profile card (`ProfileEditor`) → Theme (`ThemeSwitcher`) → Play (Cruise schedule, Suggest a square, How to play, Install the app when installable) → Support (`BugReport` `variant="row"`, `AcceptableUse` `variant="row"`) → Admin (admins only, badged) → Sign out (dashed/dim styling, `signOutUser()` from `useAuth()`) → version footer (`__APP_VERSION__` + `eventTitle`). Cruise schedule / Suggest a square / How to play open as sub-panels reusing the app's existing `.sheet-backdrop`/`.sheet` chrome (focus moves to the panel title on open, Escape or a backdrop click closes it). The Admin row navigates to the ROUTE-driven admin console (`/more/admin`, `specs/admin-console-ia.md` — originally a `MorePanel` like the others). The Admin gate reuses the same `event?.admins?.includes(user.uid)` check `Nav.tsx` computed pre-`#203`.
- `src/hooks/useData.ts` — `usePendingItemCount(enabled)`: its own small `status === 'pending'` subscription for the Admin row's badge, gated by `enabled` (an admin-only Firestore read per `firestore.rules`) so a non-admin viewer — who has More mounted unconditionally alongside everyone else — opens no subscription at all. `0`/hidden until `#210` starts writing pending items; the field itself shipped with `#200`, before anything writes it.
- `src/components/Admin.tsx` — derives the same pending count from its own already-subscribed `items` (no extra listener) to badge its "Prompts" heading, so the console and the More badge can never disagree even though each computes independently.
- `src/theme/ThemeContext.tsx` — a Player's theme pick is now a `ThemePreference` (`ThemeId | 'auto'`), not just a `ThemeId`. `'auto'` resolves to a caller-supplied `autoThemeId` (today's Day's theme) falling back to the existing event/player `defaultTheme`, as a pure per-render derivation — no separate "adopt the async default" effect needed. Persistence is unchanged in spirit: only an explicit concrete pick is ever saved (`localStorage['gcb.theme']`); picking `'auto'` itself is never saved, and additionally clears any earlier explicit pick so resolution re-derives on every future load.
- `src/theme/autoTheme.ts` — `todaysDayTheme(event, now)`: pure, Firestore-free resolution of "today's Day" from `EventDoc.days` for Auto — the last Day whose `unlockAt` has passed relative to `now`, so a Day counts as "current" from its own (possibly mid-morning) unlock moment until the next Day's unlock, not by calendar date alone; `unlockAt` ties break to the lowest `index`, the same tie-break as the pre-unlock fallback, so the resolution cannot flip at the boundary when a tied pair unlocks. Before ANY Day has unlocked (pre-cruise), it resolves the FIRST Day's theme — earliest `unlockAt`, ties broken by lowest `index` — mirroring the Board's `defaultViewedIndex` Day-0 fallback, so Auto matches the embark Day the rest of the app already presents instead of dropping to the event default (`#299`). Returns `null` only with no Days configured (or no Event yet), so the caller falls back to the event/player default instead of guessing. `src/main.tsx` is the one caller, resolving it from the subscribed `EventDoc` and handing it down to `ThemeProvider` precomputed — the same pattern `defaultTheme` already used, keeping `theme/` itself Firestore-free.
- `src/components/ThemeSwitcher.tsx` — leads with an **Auto — match the day** chip (the default) ahead of every `THEMES` entry in its existing fixed order, so new party themes keep auto-picking up here for free. A concrete pick still saves both locally and cross-device (`savePlayerTheme`); Auto saves neither.
- `src/hooks/useInstallPrompt.ts` (new) — the `beforeinstallprompt`/`appinstalled` capture-and-track logic extracted out of `InstallPrompt.tsx` so More's persistent "Install the app" row can reflect the same installability without a second implementation. Each mount point (the banner and the row) gets its own hook instance; `install_pwa` still fires exactly once per real install because each instance dedupes against its own tracked-install ref, and the two are ordinary independent `window` listeners with no shared state needed between them.
- `src/components/BugReport.tsx`, `src/components/AcceptableUse.tsx` — both gain a `variant?: 'floating' | 'row'` prop. `'floating'` (the default) is each component's original fixed-position trigger, kept intact and still exercised by their own specs (`specs/w4-bug-report-inbox.md`, `specs/w3-security-hardening.md`). `'row'` renders the identical trigger+sheet as a plain `.more-row` menu item; the live app now mounts only `variant="row"`, exclusively inside More, so exactly one of each affordance is ever reachable at a time — `App.tsx` no longer mounts `BugReport` directly, `Board.tsx` no longer mounts `AcceptableUse` directly, and `main.tsx`'s pathname-gated `AcceptableUse` mount is gone (More is a frozen tab-bar stop reachable from every signed-in route, so no per-route wiring is needed any more). #324 later split `BugReport` into launcher + `BugReportProvider`: `App.tsx` hosts the provider (the report sheet and pick-a-screen bar at the shell, so a report opened from More can capture any screen), but the only launcher is still More's Support row — the one-affordance-at-a-time contract here is unchanged; see specs/w4-bug-report-inbox.md.
- `src/components/ProfileEditor.tsx` — the trigger markup gains a name/`@handle` text block alongside the avatar, wrapped in the same `.more-row` shape as every other menu row, so tapping anywhere in the row (not just the avatar) opens the existing sheet. The sheet itself is unchanged.

## Acceptance criteria

- Given a signed-in Player, when they open More, then they see Profile, Theme, Play (schedule / suggest / how-to-play / install-when-installable), Support (bug / 18+), Sign out, and a version footer, in that order.
- Given a non-admin Player, when they open More, then no Admin row renders.
- Given an admin with 3 pending items, when they open More, then the Admin row shows a "3" badge; given an admin with 0 pending items, then the row renders with no badge.
- Given a Player picks "Auto — match the day", when they reload, then no explicit theme is persisted (`localStorage['gcb.theme']` stays unset) and the resolved app theme follows today's Day — the first Day's theme before any Day unlocks (`#299`), falling back to the event/player default only with no Days configured.
- Given a Player makes an explicit theme pick, when Auto was previously selected (or vice versa), then the explicit pick persists and overrides Auto, unchanged from the pre-Auto persistence contract.

## Test coverage

- `src/components/d15-more-menu.test.tsx` (RTL-jsdom) — drives the real `More.tsx` with its child components and data hooks stubbed. Every section/row renders in spec order; no Admin row for a non-admin Player or one absent from `event.admins`; an admin's Admin row renders with no badge at zero pending and a numeric badge at a positive count; the Admin row sits between Support and Sign out; Sign out invokes `signOutUser()` from `useAuth()`.
- `src/theme/w1-themes.test.tsx` (extended) — picking "Auto — match the day" resolves to a supplied `autoThemeId` and never writes `localStorage['gcb.theme']`; with no `autoThemeId` (no Days configured / Event not loaded) Auto falls back to `defaultTheme`; an explicit pick after Auto still saves and overrides; picking Auto after an explicit pick clears the saved value so a later reload re-resolves to Auto. This is where `ThemeContext`'s existing persistence/default suite already lives (see that file's own banner), so its Auto-mode coverage extends it in place rather than forking a second `ThemeContext.test.tsx`.
- `src/theme/d15-auto-theme.test.ts` (new) — `todaysDayTheme()`: returns the last Day whose `unlockAt` has passed relative to `now`, treats a Day as current from its own unlock moment (not by calendar date), breaks unlocked-Day `unlockAt` ties by lowest `index` consistently with the pre-unlock fallback, resolves the first Day's theme pre-cruise (order-independent, index tie-break — the `#299` regression), and returns `null` only with no Days configured or no Event loaded.
