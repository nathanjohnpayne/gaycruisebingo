# w1-prompt-pool — Prompt pool add/report: pre-sail framing + a Phase 0 client-side rate limit

The Prompt pool is the community-editable set of things-that-might-happen a Board is dealt from. Per ADR 0003 a Board freezes the moment a Player joins, so a Prompt added afterward can never land on that Player's own card — it only ever joins the pool for a future deal (a late joiner, or a future Event). This ticket (#28) strengthens the add-a-Prompt copy to say that plainly, and adds a client-side rate-limit guard on add/report so the pool can't be spammed by a double-tap or an over-enthusiastic Player, in the same Phase-0-is-presentational-only posture ADR 0004 already takes for the reactive-moderation auto-hide. Neither change touches the `items` create-if-valid / report-only-increment contract #18 established, and the report stays increment-only — the threshold-based hide is #37.

## The change

- `src/components/ItemPool.tsx` — the muted note under the add bar is rewritten to lead with the PRD's own words, "get your prompts in before we sail," and to say explicitly that a Prompt added after a Player's card is dealt joins the pool for a *future* card, not theirs. Two new pieces of local UI state, `addThrottled`/`reportThrottled`, disable the Add button / a Report button and show a `role="alert"` message while a client-side rate-limit window is open, and clear automatically once it passes (a real `setTimeout`, cleaned up on unmount).
- `src/data/api.ts` — a small, self-contained rate-limit helper (`checkItemRateLimit` + the `ITEM_RATE_LIMIT_MS` constant it's judged against, plus the read-only `itemRateLimitRemainingMs` companion the UI timer is armed from — see "Throttle UX" below) is added next to `addItem`/`reportItem`. Neither `addItem` nor `reportItem` changed behavior; the guard lives one layer up, at their one real call site — see "Why the guard lives in the caller, not the write" below.

## Pre-sail framing (ADR 0003)

The note under the add bar now reads: "Get your prompts in before we sail—once your card is dealt it's frozen, so a prompt added after that joins the pool for a future card, not yours." followed by the live pool count ("N in the pool."). This replaces the old, softer "New prompts join the pool for future cards." — the old copy never told a Player *why* a mid-cruise add wouldn't show up on their own card, which is exactly the confusion ADR 0003 flags as the thing worth messaging up front. The live count is kept (renamed from "in play" to "in the pool") so the pool-density framing (~30–50 Prompts, seeded and maintained by `w1-event-seed` and admin pruning — out of scope here) stays visible without hard-coding the target range into player-facing copy that would go stale as the pool grows or shrinks.

## Client-side rate limit (ADR 0004 posture, Phase 0)

No Cloud Functions exist yet, so — exactly like ADR 0004's Phase 0 auto-hide — this is a **client-side, presentational** guard, not a security boundary: it throttles the honest common case (a double-tap, a fast re-submit) and is trivially bypassable by a motivated caller (a second tab, a raw network call). Server-authoritative rate limiting (a Function or rules-enforced quota) is a Phase 1 concern. `checkItemRateLimit(key, now = Date.now())` keeps a module-scope `Map<string, number>` of last-fired timestamps (the same module-scope-state pattern `setMark`'s `markChains` already uses) and returns `true` — stamping `now` as the new last-fired time — only when `key` last succeeded more than `ITEM_RATE_LIMIT_MS` (3s) ago or never. `ItemPool` calls it with `` `add:${uid}` `` before `addItem` and `` `report:${uid}` `` before `reportItem`, so the limit is per-signed-in-identity, not one global bucket shared by every tab on a shared browser — and, deliberately, the SAME bucket regardless of *which* Prompt is being reported, since the thing being protected is the Player's action rate, not any one Prompt's report count.

### Why the guard lives in the caller, not the write

`addItem`/`reportItem` themselves are unchanged. The guard is evaluated exactly once per user action, in `ItemPool.tsx`, before the write is invoked — not inside `addItem`/`reportItem`. Checking in both places would consume the SAME window twice for one logical action: the caller's check would stamp `now` and return `true`, and an identical check re-run inside the write a moment later would see that same fresh stamp and incorrectly report `false`, silently dropping the very write the caller's own check just approved. A single call site avoids that self-defeating double-consumption while keeping the decision in one place (`checkItemRateLimit`) that a future caller — or a Phase 1 replacement — can reuse without re-deriving the same bug.

### Throttle UX

While throttled, the relevant control (`Add`, or a `Report` button) is `disabled`, and a `role="alert"` message appears ("Slow down—you can add/report again in a few seconds."). Both states clear at the moment `checkItemRateLimit` would itself next allow the action — not a fixed `ITEM_RATE_LIMIT_MS` from whichever attempt happened to get blocked. A blocked attempt arms a real `setTimeout` for the ACTUAL time left on the window (`itemRateLimitRemainingMs(key, now)`, judged against the SAME `now` as the `checkItemRateLimit` call it followed), so a retry late in the window — say, with only a few hundred ms left before `key`'s last SUCCESSFUL action ages out — re-enables the control that much later, not a fresh full `ITEM_RATE_LIMIT_MS` after the retry; re-arming a full window from the blocked attempt itself would keep the UI disabled well after the guard would already allow the next try (Codex P2, PR #92). It is not a live countdown, just a single fire-and-forget timer per throttle episode, cancelled and rescheduled (always to that same real expiry, since `itemRateLimitRemainingMs` is anchored to the last SUCCESSFUL call, not to the retry) if a new blocked attempt lands first, and cleared on unmount so a Player who navigates away from Prompts mid-window never triggers a `setState` after unmount. The Add input's Enter handler is gated by the SAME `addThrottled` state the button's `disabled` reads, rather than re-deriving its own answer from a fresh guard check — so the keyboard path can never submit a Prompt while the UI is showing "throttled," and it recovers in lockstep with the button once the window passes.

## Claim → test

Every claim below maps to an assertion in `src/components/w1-prompt-pool.test.tsx` (layer: RTL-jsdom, `npm test`). `addItem`/`reportItem` are stubbed (no real Firestore write in this layer); `checkItemRateLimit`/`ITEM_RATE_LIMIT_MS` are the REAL implementation from `../data/api`, so the throttle assertions exercise the actual guard rather than a re-implementation of it in the test.

- **Given** the Prompts tab **when** it renders **then** the note leads with "get your prompts in before we sail" and says a mid-cruise add "joins the pool for a future card, not yours," alongside the live pool count. (Test: "messages adding a prompt as pre-sail, and mid-cruise adds as joining a FUTURE card, not the frozen one".)
- **Given** a signed-in Player types a Prompt **when** they submit it **then** `addItem` is called with their uid and the raw (untrimmed-by-the-caller — `addItem` itself trims) text, and the input clears once the write resolves. (Test: "calls addItem with the trimmed text and clears the input on success".)
- **Given** blank or whitespace-only input **when** the Player tries to add **then** the Add button stays disabled and `addItem` is never called. (Test: "does nothing for blank/whitespace-only text".)
- **Given** a Player submits Add twice in rapid succession **when** the second submit lands inside the rate-limit window **then** `addItem` fires only once, a `role="alert"` throttle message appears, the Add button is disabled, a THIRD attempt still inside the window is also suppressed, and — once the window passes — a further Add succeeds and calls `addItem` again. (Test: "throttles a rapid second Add — addItem fires once, shows a message and disables Add, then recovers once the window passes".)
- **Given** a blocked retry lands PARTWAY through the rate-limit window (not at its very start, e.g. ~100ms of a 3s window left) **when** the guard's REAL remaining time elapses **then** the control recovers at that real expiry — not a freshly re-armed full `ITEM_RATE_LIMIT_MS` counted from the blocked retry — and Enter-in-input neither calls `addItem` while the UI shows throttled nor waits any longer than the button does; once the window passes, Enter succeeds. (Test: "recovers at the REAL remaining time — not a re-armed full window — when a blocked retry lands mid-window, and keeps Enter in lockstep with the disabled button (Codex P2, PR #92)".)
- **Given** a Player reports one Prompt and immediately reports a DIFFERENT Prompt **when** the second report lands inside the rate-limit window **then** `reportItem` fires only once (for the FIRST Prompt), both Report buttons are disabled, the alert message appears, and — once the window passes — reporting the second Prompt succeeds. This proves the limit is per-Player, not per-Prompt. (Test: "throttles a rapid second Report — even across two DIFFERENT prompts — then recovers once the window passes".)

## Acceptance criteria

- Given the pre-sail window, when a Player adds a Prompt, then it persists to the pool (via `addItem`, called directly from the click handler with no intervening step) and the pre-sail framing is visible on the same screen — `src/components/w1-prompt-pool.test.tsx` (framing + add-clears-input). The PRD's "< 5s" add-a-Prompt metric is a UX budget, not something this unit-test layer measures in wall-clock time; the structural claim it backs is that no new step was inserted between the tap and the write.
- Given a Player adds/reports rapidly, when they exceed the client rate-limit, then further add/report is throttled (disabled control + human message) until the window passes — `src/components/w1-prompt-pool.test.tsx` (the Add and Report throttle tests).
- Pre-sail framing strengthened per ADR 0003 — framing test above.
- Pool stays dense (~30–50): unaffected by this ticket: seeded and maintained by `specs/w1-event-seed.md` and admin pruning; this ticket only keeps the live count visible in the strengthened copy.
- Report stays increment-only — `reportItem`'s body is unchanged; the threshold-based hide remains #37.

## Scope and honest bounds

- The rate limit is client-side/presentational ONLY (Phase 0), matching ADR 0004's posture for the reactive-moderation hide — it protects the pool from accidental spam, not from a motivated bypass (a second tab, or a raw call to `addItem`/`reportItem` never routed through `ItemPool`'s guard). Server-authoritative limits are a Phase 1 concern, not built here.
- The throttle window (`ITEM_RATE_LIMIT_MS`, 3s) is a single fixed constant shared by add and report; it is not configurable per-Event and carries no admin UI.
- No new GA4 events: `track('add_item')`/`track('report_item')` (already in the catalog) still fire only on an actual write, never on a throttled attempt.
- This spec covers `ItemPool.tsx` and the `addItem`/`reportItem` region of `api.ts` only; the rest of `api.ts` (the per-board `setMark` serialization chain, `joinAndDeal`, `ensureUserProfile`, etc.) is untouched.
