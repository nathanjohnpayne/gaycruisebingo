# Pool-recovery auto-retry (w1-deal-auto-retry)

Feature: when a Player's deal fails because the active non-free Prompt pool is below the floor a card needs (`MIN_POOL` = 24, ADR 0003/0004), and the pool then RECOVERS from below-floor to above-floor, the deal is retried AUTOMATICALLY — the Player who added Prompts (or an Admin who did) does not have to notice the recovery and press the manual Retry themselves. This is the deliberate follow-up split out of PR #66 (issue #70): the manual Retry on the Card-route `DealError` panel already ships (#26) and works everywhere; this adds only the automatic path, designed to honor the three constraints each prior PR #66 attempt violated.

## Contract

- `src/auth/AuthContext.tsx` — exposes a TYPED cause for a deal failure alongside the Player-worded `dealError` string: `dealErrorReason: 'pool-shortfall' | 'connection' | null`, non-null exactly when `dealError` is non-null. The single classifier `isPoolShortfall(err)` (the `dealBoard` "…24 prompts…" throw the below-floor guard raises) drives BOTH the Player-worded copy and the typed reason, so the message and the reason can never disagree about the cause. `dealError` and `dealErrorReason` are set and cleared only through `failDeal`/`clearDealError`, so the reason is always in lockstep with the message — a stale reason can never arm the watcher after the error clears. Only a `joinAndDeal` → `dealBoard` pool-shortfall yields `'pool-shortfall'`; every other deal/bootstrap rejection (connectivity, permission) yields `'connection'`.
- `src/components/PoolRecoveryWatcher.tsx` — the watcher, mounted at the app SHELL inside `AuthProvider` (above the tab Router, beside `ConfirmWinMoments`), so it is mounted for the WHOLE recovery journey and survives the exact recovery path — a navigation to `/items` to add Prompts. It ARMS a live pool subscription (`useItems`) only while `dealErrorReason === 'pool-shortfall'` (rendering `null`, opening no subscription, otherwise), so the pool listener is open exactly when someone is watching for recovery. The edge is driven strictly by SERVER-BACKED snapshots and is consumed only when a retry actually fires — the same cache-vs-server / do-not-consume-the-edge-early discipline #116 (doubts/moments) and #117 (offline) established. Two properties are load-bearing:
  - Server-committed gate (BOTH per-snapshot metadata flags, NOT the lifetime `hasServerData` latch). The complete `{ includeMetadataChanges: true }` discipline needs `fromCache` (cache vs server) AND `hasPendingWrites` (local-optimistic vs server-committed); a snapshot is fully server-committed only when `!fromCache && !hasPendingWrites`. `useColSub` latches `hasServerData` true for the whole subscription once the first server snapshot arrives, so gating on it would let a later cache/local snapshot read as a server crossing and consume the edge before the SERVER pool crossed the floor — and the later real server confirmation would then find the edge already spent. And gating on `fromCache` alone is insufficient: when the player/admin adds the FINAL prompt from THIS client, Firestore emits an optimistic snapshot with `fromCache === false` (the listener is current, so it is not a cache read) but `hasPendingWrites === true` (the write is not yet server-acked) — a `fromCache`-only gate would fire the retry before the commit, and a rejected write or a still-thin server read (`joinAndDeal` sees < `MIN_POOL`) would then spend the edge with no later crossing to re-fire. So `useItems` surfaces BOTH per-snapshot flags (already computed by `useColSub`), and the watcher evaluates BOTH the baseline and the crossing only on fully server-committed snapshots. A cache snapshot (including the initial empty `useItems()`, which is cache-origin) OR a local optimistic prompt-add echo is neither a baseline nor a trigger. The first server-committed snapshot under an arming is the baseline (never a trigger); only a real below-floor → above-floor crossing from there fires.
  - Dealing does not consume the edge. When a server-committed crossing arrives while a deal is already in flight (a manual Retry, or a prior auto-retry that read the still-below pool and will fail), the crossing is PRESERVED — the below baseline is left intact — rather than consumed-without-firing. `dealing` is an effect dependency, so when the in-flight deal settles the effect re-runs and fires against the still-standing server-committed above-floor snapshot, exactly once. Without this, a manual Retry pressed just before the final prompt-add is observed would swallow the only recovery edge and strand the Player on the error.

  It fires the SAME `retryDeal` the manual button does, so it inherits #117's write-safety gate (deal only when `online && attestedAuthoritative && attested`; offline or on a merely provisional attestation, `retryDeal` re-runs the bootstrap instead of creating board/player rows). After firing, the baseline latches to above-floor, so a failed auto-retry (the error re-appears with the pool still healthy) produces no new crossing and cannot spin; only a genuine dip back below the floor followed by another crossing re-arms and fires again — once per recovery, never per snapshot. When the arming clears (the deal succeeds, or the failure changes to a non-pool cause) the watcher disarms and closes the subscription; a later re-arming re-seeds a fresh baseline.
- `src/hooks/useData.ts` — `useItems` surfaces the per-snapshot `fromCache` AND `hasPendingWrites` (both already computed by `useColSub` from the snapshot metadata — `useColSub` gained the `hasPendingWrites` state for this) alongside the lifetime `hasServerData` latch, so the watcher can gate its edge on whether the current snapshot is fully server-committed. Existing `useItems` consumers (Board) ignore the added fields.
- `src/components/Board.tsx` / `src/components/SignIn.tsx` (`DealError`) — unchanged by this feature except in prose: the Card-route `DealError` panel keeps its MANUAL Retry, and Board keeps its pre-deal `useItems(!board)` guard. The automatic path lives at the shell, NOT on the panel, precisely because the panel unmounts on the `/items` recovery path (the PR #66 finding). The two subscriptions coordinate: Board holds the pool listener before a deal (to size the thin-pool guard), the shell watcher holds it while a pool-shortfall error is up (to watch for recovery), and a dealt Board holds none.

## Acceptance criteria

- Given a signed-in Player whose deal failed on the below-floor guard (`dealErrorReason === 'pool-shortfall'`), when the active non-free pool crosses `MIN_POOL` upward (server-confirmed), then the deal is retried automatically exactly once — the same deal path the manual Retry invokes.
- Given a `DealError` from a NON-pool failure (connection/permission — `dealErrorReason === 'connection'`), when the first pool snapshot arrives healthy, then NO auto-retry fires: the watcher never arms a pool subscription for a non-pool failure.
- Given an armed watcher, the first fully server-committed pool snapshot is a baseline, not a trigger: a first snapshot that is already healthy does not auto-retry (only a later below→above crossing does), and cache-only or local-optimistic snapshots (including the initial empty pool) neither baseline nor trigger.
- Given a server-committed below-floor baseline, when a CACHE snapshot (an IndexedDB replay) OR a LOCAL OPTIMISTIC snapshot (this client's own not-yet-server-acked prompt-add — `fromCache === false`, `hasPendingWrites === true`) then shows above-floor, then NO auto-retry fires and the recovery edge is NOT consumed: only the next fully server-committed above-floor snapshot (`!fromCache && !hasPendingWrites`) fires it, never the lifetime `hasServerData` latch and never a `fromCache`-only read.
- Given a server-committed above-floor crossing observed while a deal is already in flight, then the crossing is NOT consumed: when that deal settles the retry fires against the still-standing server-committed above-floor snapshot, exactly once — a manual Retry racing the final prompt-add never swallows the recovery.
- Given an auto-retry that itself fails (the error re-appears with the pool still at/above the floor), the watcher does not loop: no further retry fires until the pool dips below the floor and crosses it again, which fires exactly one more retry.
- Given the Player is offline (or holds only a provisional, non-authoritative attestation), when the pool crosses the floor and the watcher fires, then no board/player row is created: the retry goes through `retryDeal`, which gates the deal on `online && attestedAuthoritative` and otherwise re-runs the cache-first bootstrap (no regression of #117's invariants).
- The floor counts only active non-free Prompts: Free-Space docs never count toward `MIN_POOL`, so a below→above crossing needs `MIN_POOL` real Prompts.

## Test coverage

`src/components/w1-deal-auto-retry.test.tsx` (Vitest, RTL-jsdom integration against the REAL `AuthProvider` + real `PoolRecoveryWatcher`, with the Firebase/auth + data-layer boundary and the `useItems` pool subscription mocked so pool snapshots are driven by hand):

- Edge-trigger — a real pool-shortfall deal failure arms the watcher (opens the pool subscription, `enabled: true`); a cache-only healthy snapshot is ignored; the first server-backed (below-floor) snapshot is the baseline; the below→above crossing fires exactly one auto-retry that deals the board and clears the error; a later snapshot after success does not deal again.
- Seeded-baseline — even when armed, a first server-backed snapshot that is already healthy is a baseline, not a trigger (no auto-retry; the error stays up for the manual Retry); only a later dip-then-cross fires.
- Free-Space exclusion — a crossing needs `MIN_POOL` real Prompts; a snapshot of 23 real Prompts plus Free-Space docs stays below the floor and does not fire.
- No-fire-on-initial-load — a connection failure never arms the watcher (no pool subscription is ever opened), so a healthy first pool snapshot does not auto-retry.
- Server-committed gate (F1) — after a server-committed below-floor baseline, a CACHE above-floor snapshot fires nothing and does not consume the edge; the subsequent server-committed above-floor snapshot fires exactly once. A separate case proves a LOCAL OPTIMISTIC above-floor snapshot (`fromCache === false`, `hasPendingWrites === true`) is likewise neither a trigger nor a consume — only the server-acked commit fires. Mutation-checked: dropping either flag from the `serverCommitted = !fromCache && !hasPendingWrites` gate fails a distinct one of these two cases (the latch-only revert, the `fromCache`-only revert).
- Dealing-does-not-consume (F2) — a server-backed crossing observed while a manual-Retry deal is in flight fires nothing during the deal; when that deal settles (failing on the still-below pool it read) the preserved crossing fires exactly once; a failed auto-retry then still needs a fresh below→above crossing to fire again. Mutation-checked: consuming the edge before the dealing guard fails this test.
- Fire-once + no-loop — a failed auto-retry does not spin across further at/above-floor snapshots; only a genuine second below→above recovery fires again, once each.
- #117 coordination — while offline, the crossing fires the watcher, but `retryDeal` gates the deal off (`joinAndDeal` is not called) and routes through the cache-first bootstrap instead, so no rows are created for an offline Player.

The typed reason plumbing in `AuthContext` (that a pool-shortfall failure yields the pool-worded copy, and every other failure the connection copy) rides on the existing `src/auth/AuthContext.test.tsx` deal-error coverage; this spec owns the watcher and the reason's role as the arming marker.
