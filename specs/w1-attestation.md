# Persist the 18+ attestation as a timestamped profile attestation (w1-attestation)

Feature (#23): the 18+ acknowledgement was a local-only checkbox (`SignIn.tsx` `ack`) that gated only the sign-in button, was never written to Firestore, and so did not survive a reload or gate re-entry. It becomes a persisted `users/{uid}.attestedAdultAt` (ms-epoch) timestamp that gates first Event entry and re-prompts any signed-in User whose settled profile lacks it. Per ADR 0001 this is an honor-system self-attestation — we record the User's OWN statement, we never verify IDs.

## What already landed (staleness)

Two dependencies shipped before this ticket, so this ticket adds only the client persistence + the gate:

- **The type** — `UserDoc.attestedAdultAt?: number` already exists (`src/types.ts`, added by #16 and pinned by `src/data/w0-type-contract.test.ts`). This ticket consumes it and does not re-declare it.
- **The rule** — `users/{uid}` is already owner-writable and the owner self-write already shape-checks `attestedAdultAt` as a number (`firestore.rules`, the `allow create, update` block, from #18 and pinned by `tests/rules/w0-firestore-rules.test.ts`). The self-write is allowed BY DESIGN (ADR 0001): recording your own attestation is a self-statement, not identity verification. So **no firestore.rules change is required or made** — this ticket delivers the client half gated on the existing allowance, and `tests/rules/w1-attestation.test.ts` pins that allowance under this ticket's slug so a future "lock-down" of the self-write trips a named test.

## Contract

- `src/data/api.ts` (profile region only) adds two small functions and leaves `ensureUserProfile` (the #77-pinned create-only bootstrap) and the whole Mark path untouched:
  - `attestAdult(u, now = Date.now())` persists the attestation inside a `runTransaction`, **create-only for the field**: it reads `users/{uid}`, and only when no numeric `attestedAdultAt` is already present does it stamp `now`. An existing EARLIER stamp is never overwritten, so re-attesting keeps the first timestamp. It takes the full `User` (not just the uid) because the ABSENT-row branch is load-bearing (Codex P2, PR #112, Finding 2): during first-time sign-in this transaction can win the create race with `ensureUserProfile` on the not-yet-written row, and a stamp-only write there would strand a PARTIAL profile — the create-only `ensureUserProfile` retry then sees `exists()` and no-ops, so `displayName`/`photoURL`/`createdAt` would be missing forever. So on an ABSENT row it writes the FULL `bootstrapProfile` payload `ensureUserProfile` would have written PLUS the stamp (one create), and on a PRESENT row it merges ONLY the stamp so no other field is clobbered. `bootstrapProfile` is the single extracted source of truth for the bootstrap shape, shared by `ensureUserProfile`, so the two writers cannot drift. The transactional read keeps the race safe: a concurrent `ensureUserProfile` create invalidates this attempt's read set and Firestore re-runs onto the now-present row, which takes the merge-only branch.
  - `readAdultAttestation(uid)` point-reads `users/{uid}` and returns the numeric `attestedAdultAt` when present, else `null` for a profile DEFINITIVELY without one (missing doc or missing field). It never writes.
- `src/auth/AuthContext.tsx` owns the gate/re-prompt (protected `src/auth/**` path — the change is minimal):
  - It tracks a **tri-state** `attested` for the current User: `undefined` = UNKNOWN (bootstrap unsettled or an indeterminate read), `true` = attested, `false` = a settled profile with no stamp. On every auth change the attestation re-arms to UNKNOWN, mirroring `profileReady` (#77); after `ensureUserProfile` settles it reads `readAdultAttestation`, and a thrown read stays UNKNOWN (never a re-prompt — but never a silent stall either: the failure surfaces as a retryable error, see the failure-state bullet below). Only the latest auth change settles it, and a value the User just attested optimistically is never downgraded. That no-downgrade guard is made robust per uid (Codex P2, PR #112, Finding 3): `attest()` records the uid in a session-scoped `attestedUidsRef` BEFORE the optimistic flip, and the settle consults it. Because the callback re-arms `attested` to UNKNOWN at the top of every auth change, a read that lands before the attest transaction is visible would otherwise erase the optimistic `true` and settle `false`, re-prompting a User who just attested. The transaction's success is authoritative — it wrote the stamp — so a uid marked in the ref is never settled back to `false`.
  - `needsAttestation = user != null && profileReady && attested === false`. Because it is gated on `profileReady`, a missing stamp DURING load is UNKNOWN, not absent, so the prompt never flashes mid-bootstrap (the `knownFirstBingoAt` tri-state discipline). When `needsAttestation`, the provider renders the `SignIn` re-prompt in place of its children — full-screen, mirroring the signed-out gate `App` renders on `!user` — so the User is re-prompted BEFORE they reach the Board.
  - `attest()` persists the current User's attestation (`attestAdult(auth.currentUser)` — the full `User`, so a create-race win writes the complete profile per Finding 2) and optimistically flips `attested` true so the gate lifts at once; it first marks the uid in `attestedUidsRef` (Finding 3). A failed write stays optimistically attested for the session and re-attempts on the next sign-in (honor-system, never a hard gate). `signIn()` calls `attest()` after the Google popup, because the 18+ checkbox gated the sign-in — so a first-time User is not re-prompted for the box they just ticked.
  - The Board deal is sequenced BEHIND the attestation, not just the re-prompt UI (Codex P1, PR #112, Finding 1): `joinAndDeal` — which creates the Event board + player row — runs from an effect gated on `attested === true`. So a signed-in returning User whose settled profile lacks the stamp does NOT get a board/player row created behind the re-prompt: the gate gates the SIDE EFFECT, not only what is rendered. When such a User then attests, `attested` flips true and the effect fires the DEFERRED deal exactly once; an already-attested User deals as before (the read settles `true` straight away); a first-time User deals once the signed-in attest flow settles true (including the optimistic-`true` path above). The existing `dealAttempt` guard plus `joinAndDeal`'s board-exists early-return keep the flip from double-dealing.
  - **Failure state** (Codex P2, PR #112 round 2, finding 3548428646): a THROWN `ensureUserProfile`/`readAdultAttestation` (transient Firestore failure) is an explicit ERROR terminal state, not a silent UNKNOWN. With the deal gated on `attested === true`, leaving the attestation UNKNOWN forever would strand a signed-in User on the Board's endless "Dealing your card…" — no deal, no re-prompt, no error, no retry (the pre-gating behavior at least surfaced `runDeal`'s retryable `dealError`). So the auth callback (unless the uid is sticky-attested, whose gate settles `true` and whose deal reports its own failures) surfaces the failure through the SAME Player-worded, retryable `dealError` the deal uses (the #61 `DealError` panel on the Card tab), and `retryDeal` branches: with the attestation settled `true` it re-runs `joinAndDeal` as before, otherwise it re-attempts the bootstrap (`retryBootstrap`, guarded by `profileAttemptRef` like the callback it re-runs, so an auth change supersedes it) — never `joinAndDeal` while the attestation is unsettled (Finding 1 holds). A retry that settles `true` fires the deferred deal via the gate (its settle replaces `dealError` — the P3 no-up-front-clear discipline); a retry that settles a definite `false` hands over to the full-screen re-prompt and drops the stale error; a repeat failure re-arms the same error+retry surface. Genuine in-flight loading is unchanged — only the failure terminal state changed.
- `src/components/SignIn.tsx` binds the one 18+ acknowledgement to a PERSISTED write in both entry points, so the ephemeral `ack` no longer gates entry on its own: signed OUT the checkbox gates Google sign-in (which persists after the popup); signed IN but un-attested (the re-prompt) the checkbox records the persisted self-attestation via `attest()` before the Board. It reads `user` from context to pick the mode and relabels its button accordingly ("Continue with Google" vs "Enter the event"). `DealError` is unchanged.

## Acceptance criteria

- Given a first-time User, when they attest 18+ and sign in, then `users/{uid}.attestedAdultAt` is persisted (via `signIn` → `attest`) and survives a reload — the gate reads it back and lets them through. (Tests: "attesting persists the timestamp" data-layer; "lets an already-attested User pass straight through to the Board" gate.)
- Given a User whose settled profile has no `attestedAdultAt`, when they return, then they are re-prompted before entering the Event. (Test: "re-prompts a signed-in User whose settled profile lacks attestedAdultAt".)
- Given the profile bootstrap is still in flight, when the attestation is not yet known, then the prompt is NOT shown (UNKNOWN ≠ absent), and it appears only once the bootstrap settles without a stamp. (Test: "does not flash the prompt while the profile bootstrap is still in flight".)
- Given a User who already attested earlier, when they attest again, then the earlier timestamp is preserved (create-only for the field). (Test: "never overwrites an existing EARLIER attestation".)
- Given a signed-in returning User whose settled profile lacks `attestedAdultAt`, when the re-prompt stands, then `joinAndDeal` does NOT run (no board/player row is created behind the gate) until they attest, and then it runs EXACTLY once — the gate gates the side effect, not just the UI (Codex P1, PR #112, Finding 1). (Tests: "lifts the gate when the re-prompted User attests, deferring the deal until then"; "deals exactly once for an already-attested returning User".)
- Given a first-time User whose `attest()` optimistically set the attestation, when a later auth-state callback's `readAdultAttestation` lands before the attest write is visible, then the User stays attested (the per-uid sticky marker blocks the downgrade), is not re-prompted, and the deferred deal fires once (Codex P2, PR #112, Finding 3). (Test: "keeps a just-attested first-time User attested through a stale callback read, dealing once".)
- Given the attestation transaction wins the create race with `ensureUserProfile` on an absent `users/{uid}` row, then it writes the COMPLETE profile (the shared `bootstrapProfile` shape) plus the stamp, so the profile is never left partial (Codex P2, PR #112, Finding 2). (Test: "writes a COMPLETE profile (bootstrap + stamp) when it wins the create race on an absent row".)
- Given `ensureUserProfile` or `readAdultAttestation` throws for a signed-in User, then the retryable deal-error surface renders — never an indefinite dealing state and never a re-prompt for an UNKNOWN attestation — and Retry re-attempts the bootstrap: a success settles the attestation (an attested settle fires the deferred deal exactly once; an unattested settle hands over to the re-prompt), while a repeat failure keeps the error+retry surface (Codex P2, PR #112 round 2). (Tests: "surfaces a thrown bootstrap read as the retryable deal error — no deal, no re-prompt, no spinner"; "Retry re-runs the bootstrap: repeat failure keeps the error; success settles attested and deals once"; "a Retry that settles UNATTESTED hands over to the re-prompt gate, then attesting deals once".)
- Given the owner records their own numeric `attestedAdultAt`, then the existing self-write allows it, while a cross-user write or a non-numeric value is denied. (Tests: the `tests/rules/w1-attestation.test.ts` owner/non-owner/shape cases.)
- Attestation is a User-level self-write (ADR 0001), not a verification check; no firestore.rules change ships with this ticket.

## Test coverage

- `src/data/w1-attestation.test.ts` (Vitest, Firestore boundary mocked) drives the REAL `attestAdult` and `readAdultAttestation`: attesting merges ONLY the stamp onto a present row; an existing earlier stamp is never overwritten; a create-race win on an absent row writes the COMPLETE bootstrap profile plus the stamp (Finding 2); the read reports the stamp or `null`.
- `tests/rules/w1-attestation.test.ts` (Vitest, Firestore emulator) pins the EXISTING owner self-write: the owner may create/update their own `attestedAdultAt` as a number, a non-owner may not write another User's, and a non-numeric value is denied by the shape guard.
- `src/components/w1-attestation.test.tsx` (Vitest RTL-jsdom) mounts the real `AuthProvider` + `SignIn` with the Firebase/data boundary mocked: a no-attestation User is re-prompted after settle; an attested User passes straight through; the loading state does not flash the prompt; attesting from the re-prompt lifts the gate AND fires the deferred deal, while `joinAndDeal` never runs behind the re-prompt (Finding 1); an already-attested User deals exactly once; a just-attested first-time User stays attested through a stale callback read, dealing once (Finding 3 sticky marker); and a THROWN bootstrap renders the retryable deal-error surface whose Retry re-runs the bootstrap — repeat failure keeps the error, an attested settle deals exactly once, an unattested settle hands over to the re-prompt (round 2 failure state).
