# Multi-event schema readiness (design-only)

The data model is already Event-scoped: everything a Player touches lives under `events/{eventId}/‚Ä¶`, so a second cruise is simply a new Event doc plus a build pointed at it. v1 deliberately ships a SINGLE active Event with NO room-browsing / join-code UI ‚Äî a PRD non-goal ("Full multi-tenant 'rooms' product", `docs/projects/gaycruisebingo/prds/gaycruisebingo.md:39`) consistent with [ADR 0003](../docs/adr/0003-pool-is-pre-cruise.md) (the pool is a pre-cruise activity "plus latecomers and future Events"; no re-deal or square-swap at launch) and the architecture goal that the app "stay reusable for future cruises without a rewrite" (`docs/architecture/0002-application-architecture.md:13`). This spec records the deferred multi-Event path without building any of it. Every file:line below cites the current tree, not the issue-time snapshot (e.g. the `EVENT_ID` constant the issue placed at `firebase.ts:37` now lives at `src/firebase.ts:48`).

**Decision summary:** a second cruise = update the seed's baked Event metadata (name/dates/prompt pool), seed the new Event doc, sweep the baked sailing branding, repoint a build at it via `VITE_EVENT_ID`. No rules, index, or Functions changes; no re-deal of existing Boards; no cross-Event browsing; no join codes. In the app itself the only single-event hardcode is the build-time `EVENT_ID` constant, and lifting it later is a one-file change when the id resolves from the hostname or per-site build env (an `/e/<eventId>` path prefix costs more ‚Äî see the seam section).

## What already parameterizes cleanly (evidence)

- **Client data layer ‚Äî every ref is built under `events/{EVENT_ID}/‚Ä¶`.** The converter-attached read helpers in `src/data/paths.ts:16-46` (`eventRef`, `itemsCol`, `boardRef`, `playersCol`, `playerRef`, `proofsCol`/`proofRef`, `claimsCol`/`claimRef`, `tallyMarkersCol`, `momentsCol`/`momentRef`) and the raw write-path refs in `src/data/admin.ts:6-14`, `src/data/proofs.ts:8-17`, `src/data/profile.ts:8`, `src/data/moments.ts:21`, and `src/data/api.ts:33-36` (plus the in-function refs at `api.ts:352-353` and `api.ts:411`) all interpolate the same `EVENT_ID`. Proof media uploads are Event-scoped too: `src/data/storage.ts:35` writes `proofs/${EVENT_ID}/${uid}/${proofId}.${ext}`. Nothing outside `src/data/` and `src/firebase.ts` constructs an `events/‚Ä¶` path (verified by grep: non-test references to `EVENT_ID` or an `'events'` path segment exist only in `src/firebase.ts`, the seven `src/data/` modules above, and comments/test mocks).
- **Firestore rules are fully `{eventId}`-parameterized.** The whole game surface nests inside `match /events/{eventId}` (`firestore.rules:28`) ‚Äî `items` (`:42`), `players` (`:68`), `boards` (`:80`), `proofs` (`:85`), `claims` (`:157`), `tally` + `markers` (`:173`, `:184`), `doubts` (`:201`), `moments` (`:225`) ‚Äî and admin power is per-Event by construction: `isAdmin(eventId)` reads that Event doc's own `admins` array (`firestore.rules:8-11`). Even the proof `storagePath`/`mediaURL` pinning interpolates `eventId` (`firestore.rules:139`, `:144`). N events need zero rules edits.
- **Storage rules mirror it.** `match /proofs/{eventId}/{uid}/{file}` (`storage.rules:29`) with per-Event moderation via `isEventAdmin(eventId)` (`storage.rules:7`, `:36`).
- **Cloud Functions are Event-agnostic.** `moderateProof` derives the Event from the uploaded object's path (`functions/src/index.ts:31-32`, `// proofs/{eventId}/{uid}/{proofId}.jpg`) and writes the flag back to `events/${eventId}/proofs/${proofId}` (`:53`) ‚Äî it serves any number of Events with zero config. It is the only Cloud Function; the crawler `share` OG endpoint was removed with the server-side OG pipeline (ADR 0005, #39).
- **Indexes are declared per collection id, not per Event path.** The single composite index (`players` by `bingoCount`/`squaresMarked`/`firstBingoAt`) uses `"queryScope": "COLLECTION"` (`firestore.indexes.json:4-5`), which applies to every collection named `players` regardless of its parent Event ‚Äî a new Event inherits it automatically.
- **The seed is multi-Event in its TARGET, not its payload.** `scripts/seed.mjs:123` reads `process.env.VITE_EVENT_ID` (defaulting to `med-2026`) and writes `events/${EVENT_ID}` (`:132`) with deterministic prompt ids, so re-running it against a new id is idempotent ‚Äî but the payload it writes there is baked: `EVENT_SEED` hard-codes the Med 2026 `name`/`sailStart`/`sailEnd` (`scripts/seed.mjs:27-30`) and `ITEMS` (`:70-103`) is the Med 2026 prompt pool. The only env-driven inputs are the Event id (`:123`) and the `ADMIN_UID` roster (`:124`), so a second cruise edits the payload before seeding (runbook step 1).
- **The schema anticipates retiring an Event.** `EventDoc.status` is `'active' | 'archived'` (`src/types.ts:28`). Nothing reads the field yet (its only non-test occurrence is the type definition) ‚Äî that consumer is future work, but no migration is needed to start setting it.
- **Cross-Event surfaces are global by design.** `users/{uid}` profiles (`firestore.rules:14-26`; `UserDoc` in `src/types.ts:77-86`, including the 18+ `attestedAdultAt` self-attestation) and `avatars/` (`storage.rules:23`) are deliberately outside the Event subtree: a Player's identity, avatar, and attestation carry to the next cruise, while their Board/Player rows and stats are per-Event docs that start fresh.

## What hardcodes the single Event (honest inventory)

- **`src/firebase.ts:48` ‚Äî the one real hardcode, and the intended one.** `export const EVENT_ID = import.meta.env.VITE_EVENT_ID || 'med-2026';` is a module-scope constant baked at build time (`import.meta.env` is compile-time; `src/vite-env.d.ts:12` declares the env var). Every Event path in the app derives from this single symbol via the seven `src/data/` importers (`paths.ts:2`, `api.ts:18`, `proofs.ts:2`, `moments.ts:2`, `profile.ts:2`, `admin.ts:2`, `storage.ts:2`). One deployed bundle therefore serves exactly one Event.
- **Subscription keys assume the Event can never change mid-session.** The `useData` hooks key their snapshot subscriptions on Event-less strings ‚Äî `'event'` (`src/hooks/useData.ts:82`), `'items'` (`:94`), `` `board:${uid}` `` (`:103`), `` `player:${uid}` `` (`:107`), `` `tally:${itemId}` `` (`:122`), `'players'` (`:139`), and the Feed pair: `'proofs'` (`useProofFeed`, `:143-148`) and `'moments'` (`useMoments`, `:162-163`). Correct today (the id is a compile-time constant), but the subscription effects re-run only when the KEY changes, so a future in-session Event switch must fold the Event id into every key in this inventory ‚Äî the Feed pair included, or `useProofFeed`/`useMoments` would keep streaming the previous cruise's proofs and Moments ‚Äî or stale subscriptions keep rendering the old cruise.
- **Telemetry has no per-Event dimension.** Neither `src/analytics.ts` nor `src/posthog.ts` references `EVENT_ID`, so two cruises' analytics would be separable only by time window. Acceptable for sequential Events; add an event-id property/super-property if cruises ever overlap.
- **Sailing-specific branding is baked outside the schema.** `index.html:13` ("Trieste to Barcelona, July 2026"), the absolute static OG url/image (`index.html:20-21`), and the PWA manifest name (`vite.config.ts:13-14`). These are deployment/branding concerns, not data-model blockers ‚Äî and the schema already carries the per-Event equivalents (`EventDoc.name`/`sailStart`/`sailEnd`, `src/types.ts:25-27`) ‚Äî but a second sailing on the same domain should sweep this copy when it repoints the build. All of it lives in the CLIENT bundle now; the removed crawler `share` Function (ADR 0005, #39) previously baked the same "Trieste to Barcelona." OG copy, so a branding sweep no longer touches any Cloud Function.
- **No cross-Event queries exist, by design.** There are no `collectionGroup()` queries anywhere in `src/` or `functions/src/` (verified by grep), so nothing today reads across Events ‚Äî which is exactly what keeps the rules and indexes this simple.

## Rules / indexes / hosting implications of running N Events

- **Rules: no changes for the intended sequential shape ‚Äî but what the rules provide is PATH-SCOPING plus per-Event ADMIN scoping, NOT tenant isolation.** Writes are owner-scoped, not membership-scoped: every clause is `{eventId}`-scoped, a cross-Player or forged-uid write is denied, and `isAdmin()` costs one `get()` of the Event doc per evaluation regardless of how many Events exist ‚Äî but nothing conditions any write on BELONGING to the Event. Reads are wider still: any signed-in account ‚Äî whichever Event it plays, or none ‚Äî can read every Event's doc (`firestore.rules:28-29`), items (`:42-43`), players (`:68-69`), active proofs (`:85-88`), tally + markers (`:173-185`), doubts (`:201-202`), and moments (`:225-226`); only Boards (`:80-81`) and claims (`:157-158`) are owner/admin-gated. Storage mirrors the gap: proof MEDIA under `proofs/{eventId}/{uid}/{file}` is likewise readable by any signed-in account (`storage.rules:29-30`), so gating Firestore alone would still leave cruise A's photos/audio fetchable by path from a cruise B account. Sequential cruises with the same community can accept all of that ‚Äî an archived cruise's data stays exactly as readable as it was live, to the same signed-in audience. A multi-Event deployment with DISTINCT cohorts (cruise B's accounts must not browse cruise A's players/proofs/media) is a REQUIRED rules workstream before that shape ships ‚Äî and the obvious read-only fix is SELF-DEFEATING: a membership read gate like `exists()` on the reader's own `players/{uid}` row is satisfiable by the reader itself, because that row is self-creatable under ANY `eventId` (`allow create, update: if isOwner(uid) || isAdmin(eventId)`, `firestore.rules:68-70`, no membership precondition), and items, proofs, claims, and doubts likewise accept signed-in creates under any Event (`:45`, `:92`, `:159`, `:205`). The workstream therefore gates membership WRITES as well as reads: membership must derive from a non-self-writable source ‚Äî an admin-written roster or invite doc (the Event doc's own `admins` array, `firestore.rules:8-11`, is the in-tree precedent) or a join conditioned on Event state ‚Äî with the read gate keyed off THAT, plus an archival read gate keyed on `EventDoc.status`, PLUS the matching Storage-side read gate ‚Äî `storage.rules` can make the same per-Event Firestore lookup its `isEventAdmin()` already performs (`storage.rules:7-10`), and the token story needs deciding too, since already-minted `getDownloadURL` tokens in proof `mediaURL`s are capability URLs that bypass rules ‚Äî none of which exists today.
- **Indexes: no changes for per-Event queries** (COLLECTION scope follows the collection id, above). A future cross-Event feature (e.g. an all-cruises leaderboard) would need a `COLLECTION_GROUP` index plus a collection-group read rule ‚Äî a new feature decision, explicitly out of scope here.
- **Hosting: one bundle = one Event.** `firebase.json` serves a single `dist` with an SPA rewrite, and the Event id is baked at build. Sequential cruises (the intended shape) just repoint the same site: set `VITE_EVENT_ID`, build, deploy hosting only (`npm run deploy:hosting` ‚Äî runbook step 4). Truly concurrent Events would need either one hosting site/target per Event (per-site build env) or the startup-time resolution seam below.
- **Firestore: Events coexist in one project.** A finished cruise's subtree stays where it is (flip `status: 'archived'` once something consumes it); nothing about starting Event N+1 touches Event N's data.

## How to run a second cruise (the runbook)

**Given** a reader asks "how do we run a second cruise?" **when** they read this spec **then** the answer is: update the seed payload, create the new Event doc, sweep the baked branding, and point a build at it ‚Äî no new UI (ADR 0003).

1. Update the seed payload for the new sailing. This step is a manual edit of `scripts/seed.mjs` today, not an env var: `EVENT_SEED` hard-codes the Med 2026 `name`/`sailStart`/`sailEnd` (`scripts/seed.mjs:27-30`) and `ITEMS` (`:70-103`) is the Med 2026 prompt pool, while the only env-driven inputs are the Event id (`:123`) and the `ADMIN_UID` roster (`:124`) ‚Äî so running step 2 with only a new `VITE_EVENT_ID` would stamp the OLD cruise's name/dates (and pool) onto `events/<new-id>`. Edit those constants first; the shape tests (`src/test/w1-event-seed.test.ts`) pin `settings`/`claimMode` and the 30‚Äì50 unique-prompt density band, not the name or dates, so a metadata edit passes them and a replacement pool must stay in that band. And the free-space copy is not in this script at all: the centre square's text is baked into the CLIENT bundle ‚Äî `FREE_TEXT` in `src/data/seed.ts:2`, imported by the deal path (`src/data/api.ts:21`) and dealt onto every new Board by `joinAndDeal` (`api.ts:153`) ‚Äî so a sailing that changes it edits `src/data/seed.ts` too, and that edit ships only with step 4's rebuild (`SEED_ITEMS` in the same module is a test-only mirror of the pool; nothing else in the app reads it). (A future seed change could take name/dates from env or a config file; that does not exist today and is not required for a sequential second cruise.)
2. Seed the next Event: `VITE_EVENT_ID=<new-id> ADMIN_UID=<uid,uid> GOOGLE_CLOUD_PROJECT=gaycruisebingo node scripts/seed.mjs` ‚Äî creates `events/<new-id>` with its admin roster and prompt pool (`scripts/seed.mjs:123`, `:132`). `GOOGLE_CLOUD_PROJECT` is part of the script's own documented usage (`scripts/seed.mjs:7`): on the ADC path (`:126-128`) `initializeApp` gets no explicit project id, so without the env var the Admin SDK falls back to whatever ambient project the environment supplies. It is only skippable when the gitignored `serviceAccountKey.json` fallback sits in the project root (`:126-128`), since the key file names its project itself. One-off dependency prerequisite on a fresh checkout: `firebase-admin` is deliberately ABSENT from `package.json` ‚Äî the seed is its only consumer and dynamic-imports it at run time (`scripts/seed.mjs:113`, `:118-119`), so install it first with `npm i -D firebase-admin` (or ephemerally, `npm i --no-save firebase-admin`) per the script's own header (`scripts/seed.mjs:4`) and the seed docs (`docs/app/README.md:65-68`).
3. Sweep the baked sailing branding BEFORE rebuilding ‚Äî the hardcode inventory above names every spot: the app-shell description "Trieste to Barcelona, July 2026" (`index.html:13`), the absolute static OG url/image (`index.html:20-21`), and the PWA manifest name/short_name (`vite.config.ts:13-14`). All three live in the client bundle, so every branding edit now ships with step 4's hosting rebuild ‚Äî no Cloud Function redeploys for a branding sweep. (Before ADR 0005/#39 removed the crawler `share` Function, changing its baked "Trieste to Barcelona." OG copy was the one case that also redeployed a Function; that surface is gone.)
4. Repoint the build: set `VITE_EVENT_ID=<new-id>` in the build environment and `npm run deploy:hosting` (`package.json:18` ‚Äî build plus `op-firebase-deploy --only hosting`). Do NOT use plain `npm run deploy` here: it runs an unscoped `firebase deploy` (`package.json:17`), and `firebase.json` also targets Functions, Firestore rules/indexes, and Storage rules (`firebase.json:32-41`) ‚Äî that would (re)deploy the Phase-1 Functions this runbook otherwise leaves untouched. The `--only` scoping is the established deploy pattern (`docs/app/phase-1-deploy.md:18`, `:57`; `scripts/firebase/op-firebase-deploy:17`).
5. Optionally flip the previous Event to `status: 'archived'` (`src/types.ts:28`) ‚Äî supported by the schema today, enforced by nothing yet.

Rules, indexes, and Functions all ship untouched ‚Äî a sequential second cruise is copy/config only, and no branding edit reaches a Cloud Function anymore (ADR 0005/#39 removed the crawler `share` Function that once carried baked OG copy). Players keep their global identity (`users/{uid}`, avatar, 18+ attestation) and join the new Event, which deals a FRESH Board under `events/<new-id>/boards/{uid}`. There is NO re-deal or square-swap of an existing Board (ADR 0003), NO cross-Event browsing, and NO join codes.

## Recommended migration seam (deferred)

- **Now (v1):** nothing. **Given** v1 **when** it ships **then** there is a single active Event and no room-browsing / join-code UI (PRD non-goal; ADR 0003).
- **Second cruise (sequential):** the runbook above ‚Äî no rules or logic changes; today's edits are copy/config only: the seed's baked Event metadata (`scripts/seed.mjs:27-30`, `:70-103`), the client free-space text if it changes (`src/data/seed.ts:2`), and the branding sweep (runbook step 3 ‚Äî all client-bundle copy now, so it ships with the hosting rebuild and touches no Cloud Function).
- **If concurrent Events ever matter:** keep `EVENT_ID` a startup-resolved constant and resolve the id once at startup in `src/firebase.ts:48` ‚Äî from the HOSTNAME (one subdomain per cruise) or per-site build env, falling back to `VITE_EVENT_ID`. That variant really is a one-file change: hostname/env resolution touches no routing, every consumer is a lazy helper function that reads the exported symbol per call, and the test suites already substitute it wholesale (`vi.mock('../firebase', ‚Ä¶)` with a stand-in `EVENT_ID` ‚Äî `'med-2026'` at `src/data/w2-tally.test.ts:32`, `'test-event'` at `src/components/w1-prompt-pool.test.tsx:31`), so there is no signature churn in the seven importing modules. An `/e/<eventId>` PATH PREFIX is the more invasive alternative, not a `firebase.ts`-only change: the SPA registers only absolute routes (`/`, `/feed`, `/leaderboard`, `/items`, `/admin` ‚Äî the frozen tab table, `src/components/tabs.ts:29-38`) and wildcard-redirects anything unmatched back to `/` (`src/App.tsx:47-51`), so `/e/foo` would be swallowed by the fallback and every tab `NavLink` (`src/components/TabBar.tsx:16`) would strip the prefix on navigation. Supporting the prefix means parsing it out of `location.pathname` before the router exists AND threading it into a `<BrowserRouter basename>` in `src/main.tsx:71` (mounted with no basename today) so the Route table and NavLinks re-base together ‚Äî router + nav work across multiple files, plus a sweep of prefix-unaware absolute URLs (OG/share links) ‚Äî for no benefit over subdomains here.
- **Explicitly NOT the seam:** in-session Event switching, a room browser, or join codes. That is the PRD non-goal, and it is also the costliest variant by far ‚Äî Event-keyed subscription keys throughout `src/hooks/useData.ts`, an Event picker surface, and per-Event routing. If it is ever wanted, it is a product decision first, not a refactor.
