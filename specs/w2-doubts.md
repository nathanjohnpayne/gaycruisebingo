# w2-doubts — Doubts (ask-for-proof): count on Square + Tally, satisfied by a Proof

A Doubt is one Player publicly asking another to back up a specific marked Prompt — "pics or it didn't happen". It is how the honor-system "the group is the verification" principle (ADR 0001) is applied in-app: SOCIAL PRESSURE, NEVER A GATE. A Doubt never blocks, unmarks, or discounts the Mark, never adds a Claim-like pending state, and never touches the Leaderboard — it only turns up the social heat. The open-Doubt count surfaces in two places — on the marked Square and on that Prompt's Tally entry (ADR 0002) — and attaching a Proof SATISFIES the Doubt, cooling the count from open to answered without ever gating play.

## Data model (merged-main, #16/#18 — type consumed; rules create-path tightened in PR #106: createdAt bound + id↔triple binding + standing-Mark binding)

`DoubtDoc` already ships in `src/types.ts` (from #16) and the `doubts` rules block already ships in `firestore.rules` (from #18): #33 CONSUMED both — the `DoubtDoc` type is read, never modified — and the PR #106 review rounds then TIGHTENED the rules block's create path three times (Codex P2): round 1 finding 3 bounded `createdAt` near `request.time` (closing the same boundary defect the moments block closed in PR #99 finding 3), round 2 finding 2 BOUND the doc id to the payload triple (see below and § Once-only), and round 3 finding 1 required the target's STANDING Mark (see the create bullet). A Doubt lives in a flat per-event collection `events/{EVENT_ID}/doubts/{doubtId}`, doc id DETERMINISTIC — `${fromUid}_${targetUid}_${itemId}` (`doubtDocId` in `src/data/doubts.ts`): ONE slot per (doubter, target, Prompt); the converter still pins `id` on read. `DoubtDoc` is `{ id, itemId, cellIndex, fromUid, fromDisplayName, targetUid, targetDisplayName, createdAt, satisfiedAt?, satisfiedProofId? }`.

The rules block:

- **create** — `signedIn()` AND `fromUid == request.auth.uid` (raised BY the caller; a forged `fromUid` is denied) AND `targetUid is string` AND `targetUid != request.auth.uid` (ON someone else; no self-doubt) AND `itemId is string` AND **`doubtId == auth.uid + '_' + targetUid + '_' + itemId`** (Codex P2, PR #106 round 2 finding 2 — the id↔triple binding, mirroring the moments id↔kind binding #103/#105: without it any signed-in Player could pre-create ANOTHER doubter's deterministic slot and permanently deny that Player's raise; `_` appears in no Firebase-minted uid or Firestore auto-id, and the id is constructed, never parsed) AND **`exists(.../tally/{itemId}/markers/{targetUid})`** (round 3 finding 1 — the standing-Mark binding: a Doubt is an accusation against a STANDING Mark, the social model below, and the target's attributed Tally marker is the PUBLIC half of that Mark with doc id == marker uid, so the lookup is exact. Without it a direct-write client could pre-seed Doubts against (target, Prompt) pairs the target never marked, ambushing their Square badge the moment they later mark that Prompt — the round-1-finding-2 dormant filter only scopes the SHEET header; the badge renders on the viewer's own marked Square, exactly the ambush surface) AND `cellIndex is number` AND `createdAt is number` **within +60s / -24h of `request.time`** (finding 3 — the same near-now window proofs + moments use, so a fast doubter clock or a forged far-future stamp cannot make a Doubt permanently unanswerable; the client writes `Date.now()`, in bounds by construction). There is no `hasOnly()`/`keys()` constraint, so the attributed display names ride along unrejected.
- **standing-Mark interplay (round 3 finding 1 × finding 2)** — the marker-exists binding gates CREATE only; rules never cascade deletes. A Doubt legitimately raised against a standing Mark whose target THEN unmarks survives as a doc (its marker deleted), and the display layer's dormant/reappear semantics (round 1 finding 2) are exactly what handles it: the dormant Doubt vanishes from the header count and reappears if the target re-marks — legitimate history, not an ambush, because it could only ever have been CREATED against a Mark that stood. An offline-queued raise whose target unmarks before the queue drains is denied at drain time (the client's benign debug path, no analytics).
- **once-only (round 2 finding 2)** — a second create on an existing slot is a doc-exists `update`, which the doubter (neither the target nor an admin) is denied: the cross-client duplicate backstop. Two tabs or devices racing the same raise collapse onto one doc; the open count cannot inflate, and — with `demand_proof` fired only on the write's SUCCESS settle (round 3 finding 2) — neither can the analytics stream, even for the duplicate that reaches the server and is denied. The SETTLED semantic follows: doubted → proved → settled — the same doubter cannot stack a fresh Doubt on the same (target, Prompt) in place. The structural re-raise escape already in the rules is retract-then-re-raise: the doubter deletes their own Doubt (allowed below) and a create on the now-empty slot lands with a fresh `createdAt` (deliberately not surfaced in the UI this wave).
- **read** — public (`signedIn()`): the group sees the social heat, no anonymity (ADR 0002).
- **update** — an admin, or the doubted Player (`isOwner(resource.data.targetUid)`), changing ONLY `satisfiedAt`/`satisfiedProofId` (`affectedKeys().hasOnly([...])`) — a social "answered" resolution, never a content rewrite.
- **delete** — an admin, or the doubter (`isOwner(resource.data.fromUid)`) retracting.

The `demand_proof` GA4/PostHog event is ALSO already in the catalog (`src/analytics.ts`, added by #38); this ticket only fires it from the one raise path — on the write's SUCCESS settle, so only a PERSISTED Doubt counts (Codex P2, PR #106 round 3 finding 2) — so `src/analytics.ts` is likewise consumed unchanged.

## The change

- `src/data/doubts.ts` (new) — `raiseDoubt({ fromUid, fromDisplayName?, targetUid, targetDisplayName?, itemId, cellIndex, currentlyOpen? })` self-publishes a Doubt at the DETERMINISTIC once-only slot `doubts/{fromUid}_{targetUid}_{itemId}` (`doubtDocId`, Codex P2, PR #106 round 2 finding 2) in the rules-block shape. Attribution reuses the SHARED saved-player helper `markerDisplayName` (`src/data/attribution.ts` — the SAME the Tally marker + Moment use, not forked), bounding both names to the rules' non-empty ≤100 contract, and — when the caller has NO known identity — falling back to the CACHED player row's denormalized name (the same second-argument fallback `setMark` passes, via a cache-only `getDocFromCache` read) before `'Anonymous'`, never a possibly-stale auth name (round 2 finding 3). The write is a plain **offline-queueable, fire-and-forget `setDoc`** — NOT a `runTransaction` (which needs a server round-trip and rejects offline) — mirroring the mark path's style (`setMark` in `api.ts`, `broadcast` in `moments.ts`): the promise is not awaited so raising never blocks the UI, an offline write pends durably in the persistent cache and drains on reconnect (ADR 0006), and an ONLINE rejection is logged (never silently swallowed) but never surfaced as a retry — a Doubt is low-stakes social pressure. Every duplicate path is a no-op — no write, no analytics: a self-doubt (`fromUid === targetUid`, matching the rules' `targetUid != auth.uid`); a raise whose optional `currentlyOpen` — the Prompt's open Doubts the caller passes in — already holds one by THIS doubter against THIS target for THIS Prompt (finding 1's subscription-echo backstop); a slot already in the LOCAL cache (round 2 finding 2 — a `getDocFromCache` pre-check mirroring `writeMomentOnce`, skipping the duplicate BEFORE latency compensation can locally refresh `createdAt` and flip a satisfied Doubt back open, indefinitely while offline); and a cross-client duplicate that reaches the server, which lands on the doc-exists `update` rule and is denied — logged at DEBUG as the designed once-only path (this module's payload satisfies the create rule by construction, so a permission denial here is the backstop working, not an error; every other rejection keeps `console.error`). It RETURNS the raise's settle promise (never rejects) so the caller can release its per-target pending affordance when the write settles — still fire-and-forget (not awaited). Fires `track('demand_proof', { itemId })` ONLY once the write settles SUCCESSFULLY (round 3 finding 2): the old tap-time fire also counted demands the once-only backstop then rejected — the cross-tab duplicate — inflating the metric in exactly the race the code treats as a no-op; now only a PERSISTED Doubt counts. Honest offline trade-off: a queued Doubt's `setDoc` settles at reconnect drain, so `demand_proof` tracks THEN rather than at tap time — and not at all if the app closes before draining. Accurate-but-delayed beats inflated for a counting metric. The module also exports the PURE satisfied-by-Proof derivation (below).

All appended as self-contained blocks at the end of the hooks (#37 owns the file's structure this wave).

Scoped deliberately: the shared item pool means another Player commonly holds the same Prompt (ADR 0002), so an un-doubted Player must never see a badge on their own Square over a Doubt aimed at someone else. Satisfaction for the badge derives from the VIEWER'S OWN active Proofs (`useMyProofs`, subscribed once in Board and threaded to every Square — Codex P2, PR #106 finding 4), since a Doubt against the viewer is answered only by the viewer's own Proof. The `TallySheet` gains an open-Doubt count in its header across the markers listed below (an aggregate over a named list is a heat summary, not a mis-aimed accusation), counting ONLY Doubts whose target is a CURRENT marker (finding 2 — a Doubt against a Player who has since unmarked goes dormant and reappears if they re-mark; Doubt docs are never deleted, no cross-writer cleanup), and per OTHER Player's marker row a "pics or it didn't happen" affordance that raises a Doubt against their Mark — the doubter's own row offers none (no self-doubt). The raise button disables SYNCHRONOUSLY on the first tap (a per-target in-flight ref, so a rapid double-tap cannot mint two Doubts — finding 1), cleared when the write settles; a marker this Player has ALREADY doubted — open OR satisfied — also keeps it disabled ("Doubted"): the once-only slot makes a re-raise in place structurally denied (round 2 finding 2), so the button never offers a write the rules would reject (a fresh demand is the retract-then-re-raise flow, deferred). ## Satisfied-by-Proof — PURE DERIVATION, never a stored gate (ADR 0001)

A Doubt is SATISFIED when the doubted Player attaches a Proof for the same Prompt at or after the Doubt was raised. This is DERIVED from the Proofs the Feed already subscribes to — **no write is added to `attachProof`, and the Doubt docs are never mutated on satisfaction** (`satisfiedAt`/`satisfiedProofId` stay absent; they are reserved for a future stored-resolution path, unused here). Deriving rather than gating is the whole point: an open Doubt applies social heat, an answered one cools it, and NEITHER ever blocks, unmarks, or discounts the Mark.

**The join key is (target Player, Prompt), matched by `itemText` — not `cellIndex`.** This is a forced, honest consequence of the merged-main data model, not a shortcut:

- A `ProofDoc` is keyed by `(uid, cellIndex, itemText)` and carries NO `itemId`, so a Doubt cannot join a Proof on `itemId` directly.
- A Player's Board is PRIVATE (`firestore.rules`: `boards/{uid}` is owner/admin-only), and a Tally marker is `{ uid, displayName, markedAt }` with no `cellIndex`. So when a Doubt is raised from another Player's Tally entry, the doubter genuinely CANNOT know the target's own board `cellIndex` — a cross-board `cellIndex` match is impossible, and a pure `cellIndex`-based derivation is therefore genuinely unavailable (which is why `attachProof` is left untouched: the escape hatch's precondition — "derivation impossible" — is met only for the `cellIndex` route, and the `itemText` route below keeps the derivation pure regardless).
- The Doubt's stored `cellIndex` is the doubter's OWN board index for the Prompt (the Square they raised from) — kept for context and the rules' `cellIndex is number` shape check, never used to match the target's Proof.
- The Prompt is matched by `itemText` (the Square/Tally always has the Prompt's text on hand). `itemText` is stable because items are immutable once created (only `reportCount` changes, per the items rules), so a Proof's snapshot of the text still equals the live Prompt text.

`isDoubtSatisfied(doubt, itemText, proofs, now = Date.now())` is `proofs.some(p => p.uid === doubt.targetUid && p.itemText === itemText && p.createdAt >= min(doubt.createdAt, now) - 60_000)` — "at or after", so a Proof answering a Doubt satisfies, while a Proof genuinely predating it does not (the Doubt asks for FRESH pics). Two guards shape the cutoff. **Clock-skew tolerance (Codex P2, PR #106 round 2 finding 1):** the comparison tolerates `DOUBT_SATISFACTION_SKEW_MS` (60s) — exactly the forward skew the rules GRANT every client-set `createdAt` (`<= request.time + 60s`). A doubter clock fast by up to 60s stores a future stamp the rules accept; the target's immediate, honest Proof (normal clock) then lands numerically BEFORE it, and without the tolerance the Doubt stayed open until a SECOND Proof. Within the skew window the true order of two stamps is unknowable anyway, so a Proof up to 60s "older" than the Doubt satisfies — the honest reading of "at or after" under bounded clocks (cost, stated plainly: a Proof genuinely made in the 60s before the Doubt also satisfies it — indistinguishable from the skew case, and harmless for a social count). **Now-clamp (finding 3, defense-in-depth behind the rules' `createdAt` bound):** the cutoff is CLAMPED to no-later-than `now`, so a Doubt that somehow carries a far-future stamp — e.g. a doc written before that bound shipped — is not rendered permanently UNANSWERABLE: any Proof from `now − skew` on answers it, while a normal past-dated Doubt is unaffected (`min` is its own stamp). `now` is injected (defaulting to the wall clock) only so the truth table stays deterministic. `openDoubts` is a generic unanswered-subset filter — it does not itself decide "open count for whom"; the CALLER's input scopes it. The Square badge pre-filters to Doubts targeting the Square's own marker before calling it (so another marker's open Doubt on the same shared-pool Prompt never bleeds onto an un-doubted Player's own Square), while the Tally-sheet header passes the per-Prompt set narrowed to Doubts against a CURRENT marker (finding 2 — dormant otherwise) — both then read `.length`. `doubtStatusFor(uid, …)` collapses a Player's Doubts to `none`/`open`/`satisfied`, with `open` winning any mix.

**Accepted residuals** (all harmless for a social count that never gates play, ADR 0001): two DISTINCT Prompts sharing identical text on one Player's Board would alias under `itemText` (vanishingly rare); a Doubt queued OFFLINE drains with its ORIGINAL `Date.now()` stamp, so one held offline for more than 24h would trip the rules' lower `createdAt` bound and be rejected on drain (finding 3 — the honest cost of bounding the stamp; a >24h offline gap before a low-stakes Doubt reaches the server is implausible at cruise scale, and the unbounded alternative is exactly the defect finding 3 closes — mirroring the moments offline-drain note in `specs/w2-feed-moments.md`); an offline-queued raise whose target UNMARKS before the queue drains is likewise denied at drain time by the standing-Mark binding (round 3 finding 1 — the client's benign debug path, and per round 3 finding 2 it correctly counts nothing); `demand_proof` for an offline-queued Doubt fires at reconnect drain rather than tap time — and not at all if the app closes first (round 3 finding 2: accurate-but-delayed beats inflated); and an ADMIN-doubter's duplicate raise bypasses the once-only doubter denial as an authorized admin `update` (round 2 finding 2 — the pre-existing admin moderation allowance): the count still cannot inflate (same slot, one doc), only its `createdAt` refreshes (re-opening the demand) — accepted at friend-group scale, and the client's cache pre-check stops the honest admin double-tap anyway. The finding-4 scoped proof reads are COMPLETE for their scope — the viewer's own Proofs and one Prompt's Proofs are each read in full (no newest-60 cap, unlike the old Board-wide feed) — so satisfaction no longer silently misses an older Proof that fell outside a capped window. None of these ever blocks, unmarks, or discounts a Mark.

## Constraints (encoded here)

- **Never a gate (ADR 0001).** Raising or leaving a Doubt unsatisfied does not prevent, revoke, or discount the Mark, and adds no pending state. The rules-isolation test pins that a doubter can mutate NEITHER the target's private Board NOR their self-writable player row — a Doubt structurally cannot touch the Mark or its stats — and the Board wiring never routes a Doubt through `setMark`/`toggle`.
- **Never touches the Leaderboard.** A Doubt writes only the `doubts` collection; it changes no `players/{uid}` stat, so `sortPlayers`/`useLeaderboard` never see it.
- **Offline-queueable, fire-and-forget** (ADR 0006): a plain `setDoc`, not a transaction — a Doubt raised in a ship-wifi dead zone queues durably and drains on reconnect, exactly like a bare Mark. `createdAt` is bounded near `request.time` by the rules (finding 3), so the only offline cost is a >24h-queued Doubt rejected on drain (accepted residual, above).
- **Two surfaces, deliberately different scope** (ADR 0002): the Square badge reads `openDoubts(...).length` over Doubts pre-filtered to its own marker (`targetUid`), never another marker's; the Tally-sheet header reads the same derivation over the per-Prompt set narrowed to CURRENT markers as a heat summary — a Doubt against a Player who has since unmarked is DORMANT (finding 2), never deleted, and reappears if they re-mark. A shared item pool means both scopes are reachable in normal play (multiple Players commonly hold the same Prompt), so this split is load-bearing, not cosmetic. The rules' standing-Mark binding (round 3 finding 1) is the CREATE-side complement: every Doubt was necessarily raised against a Mark that stood, so a dormant Doubt is always legitimate history — display dormancy handles the unmark-later case, the create gate closes the never-marked ambush.
- **At most ONE Doubt per (doubter, target, Prompt) — layered from UI to rules (findings 1 + round 2 finding 2).** The raise affordance disables synchronously on the first tap (a per-target in-flight ref) so a rapid double-tap raises exactly once, and stays disabled for a marker this Player already doubted — open or satisfied. `raiseDoubt` skips a raise when the passed-in open set already holds one (the subscription-echo backstop) and when the slot is already in the LOCAL cache (the `writeMomentOnce`-style pre-check, which also prevents the latency-compensation overwrite that would locally re-open a satisfied Doubt). The STRUCTURAL backstop is the deterministic slot id + the rules' id↔triple binding: a cross-client duplicate lands on the doc-exists `update` rule and is denied for the doubter (logged at debug — the designed path). The settled semantic follows — doubted → proved → settled — with retract-then-re-raise (owner delete + fresh create, both already allowed) as the structural escape, deliberately not surfaced in the UI this wave. A raise never blocks the UI: it returns the raise's settle promise only so the pending affordance can release.
- **Never an anonymous accusation (round 2 finding 3).** A Doubt stores the accuser's `fromDisplayName` PERMANENTLY (the rules allow no content rewrite), so the raise affordance is disabled until the viewer's saved identity is KNOWN — the same `identityKnown` tri-state `setMark` and the Moment broadcasts gate on — and `raiseDoubt` itself falls back to the CACHED player row's name (the same fallback `setMark` uses) before 'Anonymous' as defense-in-depth. Waiting beats guessing: the loading window is brief, and 'Anonymous' would otherwise be stamped into a public record forever.
- **Scoped, index-free proof reads (finding 4).** Doubt satisfaction never opens a Board-wide proof stream: the Square badge reads the viewer's OWN active Proofs (`useMyProofs`, `where uid == self`), and the Tally sheet reads ONE Prompt's active Proofs (`useProofsForItemText`, `where itemText ==`) only while open. Both are equality-only queries that ride the single-field indexes — no composite index, `firestore.indexes.json` untouched.
- **Glossary copy only:** the UI says "Doubt" / "pics or it didn't happen" — never callout, demand, or challenge.

## Claim → test

Every claim maps to a real assertion across the three layers; tests drive the real component/functions with only the SDK/data boundaries mocked (no vacuous mock of the thing under test).

### Rules — raise-on-another allowed, forgery/self-doubt denied, isolation from Board/player

Runner: `npm run test:rules` (Firestore emulator). Test: `tests/rules/w2-doubts.test.ts`.

- A signed-in Player may create a Doubt against ANOTHER Player's marked Prompt (`{ fromUid == self, targetUid != self, itemId, cellIndex, createdAt }`) at its bound slot id.
- A forged `fromUid` (raising as someone else) and a self-doubt (`targetUid == self`) are each denied (the fixtures seed every legitimately-doubted player — Alice included — as a Tally marker, so each denial turns on the rule under test, not a missing marker).
- The shape is enforced: a non-string `itemId`/`targetUid`, and a non-numeric `cellIndex`/`createdAt`, are each denied; a well-formed Doubt succeeds.
- **Standing-Mark binding (round 3 finding 1):** a Doubt against an unmarked PLAYER on a marked Prompt is denied, and a Doubt against a marked Player on a Prompt THEY have not marked is denied — the binding is per (target, Prompt); the same Doubt against a standing Mark is allowed.
- **Id↔triple binding (round 2 finding 2):** a well-formed Doubt at an arbitrary (auto-id style) doc id is denied, and a well-formed OWN payload parked at ANOTHER doubter's slot (the squat that would permanently deny that Player's raise) is denied; the same payload at its own slot is allowed.
- **Once-only:** a second create on the same slot (same triple, fresh `createdAt`) is a doc-exists update and is denied for the doubter — the cross-client duplicate backstop; the doubter may retract (delete) and re-create the same slot — the structural re-raise escape.
- `createdAt` is bounded near `request.time` (Codex P2, PR #106 finding 3): a near-now stamp is accepted, a far-future (+1h) and a far-past (-2d) stamp are each denied — the same +60s / -24h window proofs + moments use (each case on a DISTINCT slot so the once-only denial cannot mask the bound).
- **Isolation:** raising a Doubt lands in the `doubts` collection, but the same doubter can write NEITHER the target's `boards/{uid}` NOR their `players/{uid}` — so a Doubt can never block, unmark, or discount the Mark (ADR 0001).
- Reads are public; the doubted Player (or an admin) may mark it satisfied via `satisfiedAt`/`satisfiedProofId` ONLY (a non-target cannot, and even the target cannot change other content); the doubter (or an admin) may delete, the target may not.
- The cross-cutting `tests/rules/w0-firestore-rules.test.ts` doubt fixtures moved to canonical slot ids alongside this binding (exactly as its moments fixtures did for the #103/#105 binding), keeping their own-attribution assertions turning on ownership, not the id rule.

### Unit — `raiseDoubt` write shape + `demand_proof`; satisfied derivation truth table

Runner: `npm test` (Vitest, jsdom). Test: `src/data/w2-doubts.test.ts`.

- `raiseDoubt` writes ONE `setDoc` at the deterministic slot `doubts/{fromUid}_{targetUid}_{itemId}` (`doubtDocId`) carrying `{ itemId, cellIndex, fromUid, fromDisplayName, targetUid, targetDisplayName, createdAt }` — no `id`, no `satisfied*` — and fires `demand_proof` with `{ itemId }` from that single raise path; it is not a transaction and not a Feed post.
- **`demand_proof` counts persisted demands only (round 3 finding 2):** with the write still IN FLIGHT nothing is tracked; the event fires exactly once when the write settles successfully; a rejected write — the once-only duplicate denial or a genuine failure — fires nothing.
- Attribution uses the shared saved-player helper: an absent name falls back to `'Anonymous'`, an over-long name is bounded to 100 chars — and when no preferred name is passed, the CACHED player row's name wins over `'Anonymous'` (round 2 finding 3, the setMark fallback).
- A self-doubt is a no-op: no write, no analytics.
- Duplicate paths are no-ops (round 2 finding 2): a raise whose `currentlyOpen` already holds this doubter's Doubt skips write + analytics; a slot already in the LOCAL cache skips write + analytics (debug log — the `writeMomentOnce`-style pre-check); a `permission-denied` write rejection (the once-only server backstop) logs at DEBUG and resolves, while any other rejection keeps `console.error`.
- `isDoubtSatisfied` is true ONLY for the doubted Player's Proof for the same Prompt at or after the Doubt, within the rules' 60s skew (round 2 finding 1): an immediate Proof against a +59s fast-clock Doubt satisfies; a Proof from genuinely before the Doubt − 60s, by another Player, or for another Prompt does not answer it. `openDoubts` keeps only the unanswered Doubts; `doubtStatusFor` is `none`/`open`/`satisfied` per Player, `open` winning any mix. The satisfaction cutoff is CLAMPED to no-later-than the injected `now` (finding 3): a future-dated Doubt is answered by any Proof from `now − skew` on, while a normal past-dated Doubt is unaffected.

### Component — count on the Square, raise from another's Tally entry, distinct satisfied state

Runner: `npm test` (Vitest, jsdom). Test: `src/components/w2-doubts.test.tsx`.
