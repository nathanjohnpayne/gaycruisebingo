# w3-claim-modes — Claim Modes (honor / proof_required / admin_confirmed) end to end, plus the confirm-path Moment

**Claim Mode is the Event-wide setting for how much friction a Mark carries — a friction/vibe knob, NOT a trust level (ADR 0001).** It is a dispute/ceremony tool, never anti-cheat. There are three modes:

- **`honor`** (default, least friction) — a claim tap opens the ProofSheet, where the 🎖️ Cross My Heart pledge marks in ONE tap (issue #181, `specs/w4-honor-pledge.md`; before #181 the tap itself marked with no sheet). The pledge is the same bare `setMark` an honor tap always took: the square is `confirmed` and counts toward stats and wins the moment it is marked, and the data-layer semantics below are unchanged.
- **`proof_required`** — a Mark needs a Proof first: the ProofSheet opens, and on attach the square marks `confirmed` (it counts). The Proof is flavour, not enforcement (ADR 0001) — it enriches the Feed, it does not make the Mark more trustworthy.
- **`admin_confirmed`** (renamed from the misleading `verified`) — a Mark starts **pending**: the square is `marked` but `status: 'pending'`, excluded from the win mask (`game/logic`: `marked && status !== 'pending'`), so it does NOT count. A Claim is raised for an Admin to **confirm** (the square becomes `confirmed`, credited, and its pending Proof is published) or **reject** (the square is unmarked and its Tally marker removed; the Proof stays admin-only). Admin-confirmed is a dispute/ceremony tool, NOT anti-cheat — do not frame or build it as one.

Most of this shipped incrementally across #16 (the `verified → admin_confirmed` rename + read-migration), #32 (proof capture), #37 (admin console), and #75/#107. This ticket **verifies all three modes end to end** and closes the one genuinely-open piece: the **confirm-path Moment** deferred from #99 finding 6.

## The rename is complete (read-migration preserved)

`ClaimMode = 'honor' | 'proof_required' | 'admin_confirmed'` (`src/types.ts`). The type no longer admits `'verified'`; every write emits a current value. An Event doc persisted before the rename still storing `claimMode: 'verified'` is coerced to `admin_confirmed` on read by `migrateClaimMode` (`src/data/converters.ts`, via `eventConverter`) — a pre-rename Event behaves EXACTLY as `admin_confirmed`, pinned by `src/data/w0-type-contract.test.ts`. The Admin control reads Honor / Proof req. / Admin-confirmed — never "Verified" (`src/components/Admin.tsx`). Card intentionally omits the redundant mode label so the cruise title owns its full row and stays on one line at supported mobile widths; `claimMode` still governs mark behavior (`src/components/Board.tsx`). The last internal `firestore.rules` comments naming the old mode are renamed too; the compiled ruleset is unchanged.

## The confirm-path Moment (issue #41 — the deferred PR #99 finding 6)

**The gap.** PR #104/#110 moved the public Moment broadcast off snapshot diffing and onto the completing MARK's transition verdict (`setMark` / `attachProof`, drained through Board's module-scope pending queue). In `admin_confirmed` mode the attached cell is `pending` and excluded from the win mask, so `attachProof` structurally crosses NO transition and broadcasts nothing at attach time — a deliberate decision (`AttachProofResult` doc-comment): a pending Claim can be REJECTED and a Moment is IMMUTABLE, so an attach-time broadcast would leave a permanent win announcement for a claim an Admin then rejects. The win — and its Moment — materialize only when an Admin **confirms** the Claim (`resolve()`, `src/data/admin.ts`, flips `pending → confirmed` and recomputes stats). But that confirm can land while the winning Player is off the Card route (Board unmounted) or offline, so Board's route-scoped machinery never sees it: stats and the leaderboard update while the Feed silently misses the win.

**The mechanism chosen: a player-side confirm listener (`src/components/ConfirmWinMoments.tsx`), mounted once at the App shell OUTSIDE the tab Routes.** It watches the Player's OWN Claims (`useMyClaims`, a `where('uid','==',uid)`-scoped read the claims rule permits — a player is not an admin, so an unconstrained collection read would be denied), notices when one flips to `confirmed`, waits for the confirm's board write to reflect the flip (the cell is no longer `pending`), and emits the SAME Moment(s) the live edge would have — attributed to the WINNER (the claim owner), NEVER the confirming Admin. The emitter runs entirely as the winner (`isOwner` create), so it needs **no rules change**: the moments create rule stays own-uid-only and immutable (which #99 finding 4 deliberately tightened), rather than widening it to cross-uid admin creates — the surface the spec flagged as reopening the dedup/forgery risk. The emit decision is factored into the pure `planConfirmBroadcasts` (`src/data/moments.ts`) for direct unit-testing.

**Why not the alternatives.** (a) A rules evolution letting an Admin create a Moment for another uid re-opens exactly the forgery/dedup surface finding 4 closed. (b) Emitting from the `resolve()` transaction cannot attribute to the winner without that same cross-uid rules widening. The player-side listener keeps the write own-uid and the rules untouched.

### Hardened properties (Codex #116 review, rounds 1–4)

The listener MIRRORS the #110 live pipeline; each property below was a Codex #116 finding:

1. **Transition-gated (R1 finding 1).** `planConfirmBroadcasts` broadcasts only what the just-confirmed cells CROSSED, not what `hasBingo` reports on the current board. It recomputes the win against the board with those cells (`confirmedIndexes`) forced back to their pre-confirm PENDING state and emits a kind only on a genuine no-win → win transition. So a player who ALREADY holds a standing BINGO and has a DIFFERENT non-completing Claim confirmed posts nothing — matching the live path, which fires only on `bingoTransition` / `blackoutTransition`.
2. **Drain-until-empty (R1 finding 2).** The prior-win witness read is async; a second confirm arriving while it is in flight is enqueued (the `inFlight` guard skips it) and the drain re-runs on settle until the queue is empty, so a rapid multi-confirm flow never strands the later Moment.
3. **Fresh-confirm witness — server-backed only (R1 finding 3, tightened by R2 finding 2).** A confirm is fresh (emits) only if the listener observed the Claim PENDING while mounted (`seenPending`), and that pending observation must come from a **server-backed snapshot** (`!fromCache`), never a cache-only replay. On a fresh reload the persistent IndexedDB cache re-delivers the last-known rows; without the server-backed gate, a Claim that was pending at close and confirmed WHILE CLOSED would be recorded pending from that cache snapshot, making the subsequent server `confirmed` look like an in-session flip and posting a NEW Moment (with `Date.now()`) that pins an OLD admin confirmation to the Feed top. Gating on `!fromCache` keeps the fully-closed confirm baselined, while the **offline-but-open** case still fires: that case saw the Claim pending from a server snapshot in-session (the app was open and online) before going offline, so it seeds normally and its reconnect `confirmed` is a genuine fresh transition. **Explicit residual (the ONLY accepted loss):** a Claim whose pending→confirmed happened entirely while the app was FULLY CLOSED posts no Moment on the next open — the same class and narrowness as the live-edge reload residual.
4. **Uid-keyed persistence (R1 finding 4).** The working state (`seenPending` / `handled` / `awaiting` / `heldCeremony` / `inFlight`) lives in MODULE scope keyed by uid (`getConfirmState`, mirroring the #110 pending-Moment queue), so an account switch PARKS held work — including a roster-held `first_bingo` — and a switch-back resumes it for the original uid rather than baselining the now-confirmed Claim away.
5. **Held-ceremony fall-clearing (R2 finding 1).** A `first_bingo` candidate parked at the roster gate is identified with the ORIGINAL win. On EVERY board update while holding — not only at publish time, and REGARDLESS of roster/identity state — the listener re-checks that the held win still STANDS on an attributable board; the moment it does not (an observed bingo fall), the candidate is VOIDED. Without this, a slow-roster window let a player lose the held bingo then complete a DIFFERENT line before the roster confirmed, and the publish-time `hasBingo` check would fire the STALE held singleton against the REGAINED board. A later regained line is a NEW win, adjudicated by its own confirm transition — never this candidate. This mirrors the #110 round-4 fall clearing (drop the candidate on an observed bingo fall). A missing / non-attributable board is NOT a fall (it holds), so the uid-keyed park (property 4) is preserved across account switches.
6. **Cross-account claim guard (R3 finding 1).** `useColSub` clears a retired query's rows only in an effect, so on an account switch the listener can render once — or a queued stale listener callback can fire — with the PREVIOUS account's Claims under the NEW uid. Every Claim is therefore filtered by `c.uid === uid` at the point of use, before it can seed the witness or be enqueued/adjudicated. This is belt-and-braces beyond the effect-time clear — the same synchronous attribution guard #110/#106 apply — so a prior account's pending Claim can neither seed the new account's `seenPending` nor make it broadcast a Moment against its OWN board.
7. **Listener survives the attestation gate (R3 finding 2).** `AuthProvider` renders the attestation re-prompt INSTEAD of its children for a signed-in-but-unattested returning user, so a listener mounted in `App` (the children) would UNMOUNT during that prompt — and a confirm that lands while the player is attesting would then be seen only as already-confirmed and baselined as history. The listener is therefore mounted BESIDE the gate — `{user && <ConfirmWinMoments />}` in `AuthProvider`, rendered for ANY signed-in user regardless of attestation state — so it observes the Claim pending in-session and stays up across the prompt. Its uid-keyed module state (property 4) also carries any parked ceremony across a remount. The change is scoped to the mount location only; the attestation gate logic is untouched (that surface is #117's).
8. **Board-write-landed reflection (R3 finding 3).** A confirm is adjudicated (and CONSUMED) only once the resolve() board write has LANDED at its cell — the cell is MARKED and `status === 'confirmed'`, the exact state `confirmClaim` writes. The prior `status !== 'pending'` test wrongly treated a not-yet-arrived board (a stale / pre-claim cell whose `status` is `undefined`) as reflected, adjudicated the STALE board (no transition), and consumed the awaiting Claim — so the later board snapshot that actually completed the line never posted. Now a not-yet-arrived board leaves the Claim AWAITING, and only the snapshot that reflects the confirmed mark adjudicates and consumes it — the consume-on-decision correctness fix.
9. **Post-await reflected recompute — batch adjudication (R4 finding 1).** The reflected set is recomputed from the CURRENT `awaiting` + board AFTER the async witness read, not from a pre-await snapshot. A sibling confirm that lands while the witness read is in flight is included, and ALL currently-reflected confirms are adjudicated as ONE batch: `confirmedIndexes` holds every reflected cell, so `planConfirmBroadcasts` forces them all back to pending for the "before" state and measures the transition the BATCH crossed. Without this, a rapid admin multi-confirm where two claims each complete a line let the first plan see the LATER claim's cell as an already-standing bingo (cross nothing) and the rerun see the FIRST as already-standing (cross nothing) — both masked, the Moment lost. Batching makes concurrent multi-confirm deterministic: each crossing emits exactly once (deterministic-id dedup besides), none masks another, none doubles.
10. **Claim-specific reflection by proofId (R4 finding 2).** `confirmClaim` (`src/data/admin.ts`) resolves the board cell by `x.proofId === c.proofId` when the Claim has a proofId, so the confirmed cell carries THAT Claim's proofId. The listener therefore stores `{ cellIndex, proofId }` in `awaiting` and a cell is "reflected" for a Claim only when it is marked, `confirmed`, AND its `proofId` matches (falling back to index-only for legacy Claims with no proofId). Without this, a player with two pending submissions for the SAME square could have an OLDER Claim marked confirmed by proofId WITHOUT touching the current cell (which carries the newer proof), yet the index-only test would treat that cell as the older Claim's reflection and emit a Moment for a transition the older Claim did not cause. Matching the proofId adjudicates only the Claim the board cell actually reflects.

### Invariant satisfied

> A confirmed win must produce the SAME Moment the live edge would have — the same kind at the same deterministic id, attributed to the WINNER — **exactly once**, and NEVER a false singleton, whether the winner is off the Card tab or offline.

- **Exactly-once (across the live + confirm paths) is STRUCTURAL, not timing.** The writers' deterministic doc ids (`${uid}-bingo` / `${uid}-blackout` / the `first_bingo` singleton) are create-only and a Moment is fully immutable, so if a live edge ever also posts the same win (the Player was on the Card route and a live Mark completed it), the confirm-path emit is a denied/skipped no-op — the `writeMomentOnce` cache pre-check skips it, or the create-only rule denies it on a cold cache. This is the same backstop #99/#110 rely on; the confirm path composes with it rather than adding a second dedup. The transition gate (property 1) also keeps a non-completing confirm from ever attempting a redundant broadcast.
- **No false singleton under a confirm racing a live edge.** `planConfirmBroadcasts` applies the SAME first-BINGO gates the live drain does: the ceremonial `first_bingo` is raised only on a bingo TRANSITION, claimed only against a **server-confirmed roster** showing no OTHER Player already has a `firstBingoAt`, **held** (not guessed) while the roster is unconfirmed, and **suppressed on a regain** via the durable `hasPriorBingoWitness` — read ONCE at detection so a later roster-gate re-decision never re-reads it and self-suppresses against this pipeline's own just-posted `${uid}-bingo` (the #99 round-3 finding-A trap). An empty / non-attributable board emits nothing — the `cells.length > 0` guard keeps `isBlackout([])` (vacuously true) from firing.

**Documented residual (accepted, the ONLY loss — properties 3 + 5).** A Claim whose pending→confirmed happened entirely while the app was FULLY CLOSED is baselined as history on the next open: the listener never observed it pending from a **server-backed** snapshot (a cache-only replay does not seed the witness — R2 finding 2), so `seenPending` lacks it and no Moment posts — the same class and narrowness as the live-edge reload residual (`specs/w2-feed-moments.md`). A confirm while the app is merely backgrounded / offline-but-open still fires on reconnect, because the listener saw the Claim pending from a server snapshot while mounted and the pending→confirmed flip is then a fresh transition.

## Acceptance criteria → tests

The checker matches this spec's basename to any `*w3-claim-modes*.test.*`.

- **`src/data/w3-claim-modes.test.ts`** (data layer)
  - Honor marks instantly (`computeMark`: square `confirmed`, credited, line stands); `admin_confirmed` marks start `pending`, excluded from bingo credit until confirmed.
  - `confirmClaim` flips the pending cell to `confirmed`, credits the square, stamps `firstBingoAt` on a completed line, and publishes the pending Proof (`active`); `rejectClaim` unmarks and does not credit or publish.
  - `planConfirmBroadcasts` — the TRANSITION-gated emit decision (property 1): a confirm that completes the line emits bingo + ceremonial first; **the CORE finding-1 case — a standing-BINGO player confirming a different non-completing square emits nothing**; a full-card confirm emits blackout without re-emitting the already-standing bingo; no false singleton when another Player bingoed; held while the roster is unconfirmed; regain (witness) posts the plain bingo but never re-mints the singleton; empty board and no-line squares emit nothing.
- **`src/components/w3-claim-modes.test.tsx`** (component layer)
  - Admin Claim-Mode control: labels Honor / Proof req. / Admin-confirmed (never "Verified"), reflects the Event's active mode, writes the chosen mode; the pending-claims queue confirms/rejects as the acting Admin.
  - `ConfirmWinMoments`: a confirm completing a line off-route emits the winner-attributed bingo + first_bingo exactly once; a confirm racing another Player's live first-BINGO posts the bingo but not the ceremony (no false singleton); the ceremony holds while the roster is unconfirmed then fires on confirm; a win already standing at mount is baseline (not re-announced); a confirmed square completing no line emits nothing. Plus one regression per Codex #116 finding — round 1: **R1F1** standing-bingo + non-completing confirm → nothing; **R1F2** a second confirm arriving during the first witness read still drains (drain-until-empty) while the non-completing first emits nothing; **R1F3** an offline-but-open Claim observed pending from a SERVER snapshot then CONFIRMED on reconnect still emits; **R1F4** an account switch parks a roster-held first_bingo and a switch-back fires it once for the original uid — round 2: **R2F1** a held first_bingo whose ORIGINAL win falls is dropped and a later regained line does not fire it; **R2F2** a CACHE-only pending observation does not seed the witness, so a confirm that landed while the app was closed stays baselined — round 3: **R3F1** a prior account's stale Claim rendered under the new uid seeds and broadcasts nothing; **R3F3** a confirm reported before its board write lands HOLDS (no premature emit + delete) and fires once when the mark arrives — and round 4: **R4F1** two confirms each completing a different line, the second landing mid-witness-read, are batched so neither masks the other (exactly one BINGO); **R4F2** a Claim resolved by a proofId the board cell does not reflect stays awaiting and emits nothing. The listener-mount-survives-attestation regression (**R3F2**) lives in `src/components/w1-attestation.test.tsx` (it asserts the listener stays mounted behind the attestation re-prompt). Each was verified to FAIL against the corresponding pre-fix HEAD. Adjacent already-pinned coverage this ticket relies on: `admin_confirmed` `attachProof` → pending Proof + pending Claim (`src/data/w2-proof-capture.test.ts`); `rejectClaim`/`confirmClaim` Tally-marker symmetry (`src/data/w2-tally.test.ts`); the deterministic-id, create-only, immutable Moment writer + rules (`src/data/w2-feed-moments.test.ts`, `tests/rules/w2-feed-moments.test.ts`); the `verified → admin_confirmed` read-migration (`src/data/w0-type-contract.test.ts`).
