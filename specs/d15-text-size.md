# Text size S/M/L + auto-fit guard (d15-text-size)

Feature: `plans/daily-cards-spec.md` § "More menu" item 3, "Text size" — a Small / Medium / Large segmented control in the More menu, persisted per device (`gcb.textSize`), that scales the Square's base type and body text. The cell's auto-fit guard always has the last word: a long prompt steps its font down until it fits inside its Square, so Large is a ceiling on the starting size, never a guarantee of it or an overflow. No `fitText`-style measure-and-shrink utility existed anywhere in this codebase before this ticket (#215) — the guard below is a from-scratch design, not a port of an existing print-card routine (none exists yet).

## Contract

- `src/game/fitText.ts` (new) — `fitTextSize(text, box, options)`: a PURE, DOM-free estimator (greedy word-wrap over an average-glyph-width model) that returns the largest font size at or below `options.baseSize` that fits `text` inside `box`, floored at `options.minSize` (default 6px). Short text that already fits returns `baseSize` UNSHRUNK. Deliberately not a real canvas `measureText`/DOM-layout loop — `Board.tsx` supplies an already-measured box from the live DOM, so this file stays unit-testable in plain jsdom with no canvas dependency, and the estimate only needs to be MONOTONIC (a smaller box never yields a larger fitted size), not typographically exact.
- `src/hooks/useTextSize.ts` (new) — `useTextSize()`: a module-level shared store (mirrors `useInstallPrompt`'s pattern, not a Context/Provider — Text size is device-local only, unlike the theme pick's cross-device Firestore sync) subscribed to via `useSyncExternalStore`. Reads/writes `localStorage['gcb.textSize']` on the same defensive try/catch persistence pattern as `ThemeContext`'s `gcb.theme`; an explicit pick is the only thing ever persisted. Applies `data-text-size` to `<html>` synchronously inside the store's `setState`, ahead of notifying React subscribers, so every mount point (More's control, Board's fit guard) observes the SAME live pick with no remount required and the CSS custom property is already updated by the time any effect keyed on the returned value re-runs.
- `src/index.css` (modify) — `--text-scale` custom property on `:root`, overridden to `0.9` (small) / `1.15` (large) by `:root[data-text-size]`; medium needs no override (it is the existing unscaled baseline). `body`'s `font-size` reads `--text-scale` (`calc(100% * var(--text-scale, 1))`), so any body-copy text using relative (`em`) sizing inherits the scale from the root; badges, chips, day chips, and other chrome stay on their own fixed px sizes throughout `index.css` and are unaffected either way. **#325 split the Square ceiling onto its own multiplier:** `.cell` (and `.free-prompt`) read `--square-scale` — default `1`, `1.4` at large, deliberately NO small override — so the tiles render bigger at Large than the original `1.15` gave them and never render below the Medium baseline (Small used to put tiles at `0.9`), while every non-tile size is untouched by the tile bump (`--text-scale`'s values are unchanged). `.cell`'s font-size is `calc(clamp(8px, 2.3vw, 12px) * var(--square-scale, 1))` — the S/M/L-scaled, still viewport-clamped CEILING the auto-fit guard then measures against, so a bigger Large only enlarges prompts with room; long prompts keep shrinking to fit, never truncating. A `.text-size`/`.text-size-chip` block styles the segmented control, mirroring `.themes`/`.chip`.
- `src/components/Board.tsx` (modify) — `SquareText`: a small per-Square component that renders a non-free Square's prompt inside a `<span className="cell-text">`, measures the cell's live box (`getBoundingClientRect`, minus the cell's 4px padding) and the CSS-resolved ceiling (`getComputedStyle` on its own un-shrunk span), and applies `fitTextSize`'s result as an inline `font-size` — so the guard's shrink always wins over whatever the S/M/L pick would otherwise render, and never exceeds it. Re-measures when the prompt text changes OR the live `textSize` pick changes (read from `useTextSize()`, used only to retrigger the effect — the ceiling itself is always re-read fresh from the DOM, never hardcoded per size). A cell not yet laid out (`getBoundingClientRect` reporting 0×0, e.g. pre-first-paint) is left at the unshrunk ceiling, since `fitTextSize` itself treats a zero-area box as nothing to measure against yet.
- `src/components/More.tsx` (modify) — a new "Text size" `more-section` between Theme and Play (the slot #208 reserved for it), rendering a local `TextSizeSwitcher`: three `.text-size-chip` buttons (Small / Medium / Large), the active one highlighted via `useTextSize()`'s current value, firing `text_size_change` (`src/analytics.ts`, new GA4/PostHog event) on every pick alongside `setTextSize`.

## Acceptance criteria

- Given a Player picks Large, when a very long prompt lands on their card, then that Square's text shrinks below the Large base rather than overflowing or clipping.
- Given any pick, when a Square renders, then its ceiling never sits below the Medium baseline, and Large renders tiles larger than the pre-#325 `1.15` ceiling (`--square-scale`: 1 / 1 / 1.4) — while every non-tile text size renders exactly as it did before #325 (`--text-scale` values unchanged).
- Given a Player picks a size, when they reload, then the choice persists via `gcb.textSize` on that device.
- Given any text size, when badges/chips/chrome render, then their size is unaffected (they carry their own fixed px sizes throughout `index.css`, never `--text-scale`).
- S/M/L control added to the More menu, between Theme and Play.
- Fit guard implemented and always wins over the chosen base size — the guard only ever shrinks the CSS-resolved ceiling, never grows past it.
- `gcb.textSize` persists per device only (no Firestore write).
- A pick made in More is reflected on Board immediately (shared store, no remount required).

## Test coverage

- `src/game/fitText.test.ts` (new) — `fitTextSize`: a short string that already fits returns the base size unshrunk; an oversized prompt shrinks below the base without going under the configured floor; a pathological prompt bottoms out exactly at the floor; the result never exceeds `baseSize`; smaller boxes never yield a larger fitted size than bigger ones (monotonic); an unmeasured (zero-area) box and empty/whitespace-only text both return the base size unshrunk.
- `src/hooks/useTextSize.test.ts` (new) — defaults to medium without auto-persisting; a pick persists to `localStorage['gcb.textSize']` and applies `data-text-size` to `<html>`; the pick survives a remount (a fresh store snapshot reads the saved value back); two independent `useTextSize()` mount points (mirroring More's control and Board's fit guard) observe the SAME live pick with no remount.
- `src/components/Board.test.tsx` (extended) — with the cell's `getBoundingClientRect` stubbed to a realistic size (jsdom reports 0×0 by default) and a maximally long seeded prompt, the rendered `.cell-text`'s computed font size is strictly below the CSS-resolved ceiling for that same box — proof the guard actually shrinks rather than merely capping at the unshrunk base.
- `src/w2-ga4-events.test.tsx` (extended) — `GA4_EVENTS` includes `text_size_change` in its ordered catalog.
- `src/d15-square-scale.test.ts` (new, #325) — CSS-source pin of the tile-only split: `--square-scale` is 1 by default and 1.4 at large with NO small override; `--text-scale` stays 0.9 / 1 / 1.15; `.cell` and `.free-prompt` read `var(--square-scale)`; no other `font-size` in `index.css` does; `body` still reads `var(--text-scale)`.

## Out of scope

Cross-device sync of the text-size pick (a Firestore write, like the theme pick's `savePlayerTheme`) is deliberately excluded — the issue is explicit that this is device-local only, the same posture the theme pick had before Auto/cross-device sync existed. Scaling every individual body-copy element across the app (as opposed to the root `body` font-size that `em`-sized text inherits from) is out of scope for this ticket's size — the Square's prompt text is the one element this ticket's tests hold to a precise contract.
