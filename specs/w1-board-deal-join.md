# Board render + deal/freeze-at-join (w1-board-deal-join)

Feature: a Player's Board is dealt once, at join, from the active non-free Prompt pool and then frozen — 24 sampled Prompts plus the always-marked "Complain about Circuit Music" Free Space centre, rendered as a 5×5 card. There is NO re-deal and NO Square-swap (ADR 0003). When the active pool is too thin to fill a card (< 24 Prompts, ADR 0004) the deal guard is surfaced to the Player instead of a blank, forever-spinning card.

## Contract

- `src/game/logic.ts` — `dealBoard(pool, freeText, seed)` deals a frozen 5×5 Board deterministically from `seed`: 24 Prompts sampled from `pool` plus the free centre at index `CENTER` (12), which is `free: true` and `marked: true`. It throws when `pool.length < MIN_POOL` (24) rather than persisting a card with blank cells. `MIN_POOL` is the single source of truth for the 24-Prompt floor, shared by the deal and the Board render.
- `src/data/api.ts` — `joinAndDeal(u)` freezes the Board at join: it early-returns when a Board already exists for the uid (re-joining never re-deals), otherwise reads the active, non-free Prompt pool (`items` where `status == 'active'` and `!isFreeSpace`), deals via `dealBoard` seeded per-uid (`seedFromUid`), and batch-writes the `boards/{uid}` doc plus the Player row. The Player row denormalizes the Player's SAVED identity, not the raw Google one: it prefers the `users/{uid}` profile's `displayName`, and that profile's `photoURL` when it carries `customPhoto: true` (without the flag the profile photo is just a stale copy of the Google one, so the live auth photo wins). Because `users/{uid}` is self-writable (firestore.rules shape-checks only its `attestedAdultAt`), the saved fields are VALIDATED before denormalizing: a saved name counts only as a real, trimmed-non-empty string of ≤ 100 chars — `players/{uid}` writes themselves are deliberately unvalidated (self-writable by design, ADR 0001), so the guard adopts the 100-char `displayName` bound firestore.rules enforces on every other public displayName denormalization (markers, moments, proofs) — a saved photo counts only as a well-formed `https://` URL string (no other scheme, nothing unparseable), and the `customPhoto` opt-in counts only as EXACTLY boolean `true` (truthy junk like `'false'` or `1` never publishes the saved photo). Anything malformed falls back per-field to the auth values, exactly like a missing profile. The profile read is best-effort and join-path only — one extra read, fetched alongside the pool; a missing or unreadable profile falls back to the Firebase Auth values rather than blocking the deal. The `dealBoard` throw propagates (it is not caught here), so a thin pool never persists a broken Board.
- `src/components/Board.tsx` — renders the dealt Board's 25 Squares under the fixed `B I N G O` header: each cell shows its `text` verbatim (for the free centre, that is the seeded `FREE_TEXT`, "Complain about Circuit Music" — never the literal string "FREE") and reflects `marked` / `pending` / winning state. There is no re-deal, shuffle, or Square-swap control anywhere on the card. Before a Board exists, the component surfaces the ADR 0004 guard: once the active non-free pool has loaded and is `< MIN_POOL`, it shows the "not enough prompts" message (`role="alert"`) rather than the neutral "Dealing your card…" state, so a thin pool is never rendered as a blank card. The guard fires only on SERVER-CONFIRMED data: under the ADR 0006 persistent cache a cold or stale IndexedDB can resolve both listeners from cache with `board=null` and an empty pool, so `useDocSub`/`useColSub` expose a `hasServerData` latch (subscribed with `includeMetadataChanges`, latched by the first `!metadata.fromCache` snapshot, reset per subscription key) and the alert additionally requires it for both the board and the pool — cache-only data keeps the neutral state. Board subscribes to `useItems` only while there is no Board yet (that is what sizes the guard); once a Board exists, `useItems` is disabled (no live pool listener) — the pool is irrelevant to a Player whose card is already frozen.
- `src/components/SignIn.tsx` (`DealError`) — the retry surface App mounts while `dealError` is set. Recovery is deliberately MANUAL: the Retry button re-invokes the deal, and the `/items` Prompts tab stays reachable (the shell keeps rendering, per the w1-auth-google wiring) so a Player or Admin can add Prompts and come back. An automatic pool-recovery watcher was prototyped here during review and removed by human decision (PR #66 tiebreak): three review rounds showed it needs a deliberate context-level design (it misfires for non-pool deal failures because the pool subscription starts empty, and it unmounts when the Player navigates to `/items` — the exact recovery path). Tracked as a follow-up, not accreted onto this ticket.

## Acceptance criteria

- Given an active pool of ≥ 24 Prompts, when a User joins, then a frozen 5×5 Board is dealt once (24 Prompts + Free Space) and re-joining does not re-deal or swap any Square.
- Given an active pool of < 24 Prompts, when a User joins, then `dealBoard`/`joinAndDeal` throw rather than persisting a blank Board, and the Board render surfaces the guard to the Player (ADR 0004) instead of a blank card.
- Given a cold persistent cache (new device / cleared IndexedDB), when the listeners first resolve from cache with no board and an empty pool, then the thin-pool alert does NOT flash — the neutral "Dealing your card…" state holds until a server-confirmed snapshot arrives, and only a server-confirmed thin pool (or the server-delivered Board) replaces it.
- The Free Space centre reads "Complain about Circuit Music" (the seeded `FREE_TEXT`), is always marked, and counts toward completed lines.
- No re-deal / Square-swap affordance exists on the Board (ADR 0003).
- Given a deal error is up (App shows the `DealError` retry surface with the shell and `/items` still reachable), recovery is manual: the Retry button re-invokes the deal. There is no automatic pool-recovery retry (removed by the PR #66 human tiebreak; a context-level design is a tracked follow-up).
- Given a Player already has a Board, the pool subscription (`useItems`) is disabled — no live listener stays open on prompts that can no longer affect an already-frozen card.
- At join, the public Player row prefers the saved `users/{uid}` profile identity (custom `displayName`; `photoURL` when `customPhoto`) over the raw Google identity, and falls back to the auth values when no profile doc exists, the read fails, or a saved field is malformed (non-string / empty / over-100-char name; non-string or non-`https://` photo; `customPhoto` anything but boolean `true`).

## Test coverage

`src/components/w1-board-deal-join.test.tsx` (Vitest, RTL-jsdom + firestore-mocked unit):

- Board render — a real `dealBoard` result (from the seed pool) rendered through `Board`: exactly 25 Squares, one Free Space at the centre marked and reading the seeded `FREE_TEXT` ("Complain about Circuit Music", never the literal string "FREE"), and no re-deal / shuffle / swap / "new card" control present.
- Board deal guard — with no Board yet and the active non-free pool `< MIN_POOL`, `Board` shows the guard alert; with the pool `≥ MIN_POOL` (or still loading) it shows the neutral "Dealing…" state, proving the guard fires only on a genuinely thin pool, not on an in-flight deal.
- Board cold-cache suppression — with both subscriptions resolved but cache-only (`hasServerData: false`), an empty pool and missing board keep the neutral state (no alert); the alert appears only after the server confirms a thin pool, and a server-delivered Board renders the card with no alert flash.
- Board pool-listener gate — `useItems` is called with `enabled: true` while there is no Board, and `enabled: false` once a Board exists.
- `joinAndDeal` freeze-at-join — with an existing Board doc it early-returns without reading the pool, the profile, or writing (no re-deal, no extra read for returning Players); with no Board and a healthy pool it deals once and batch-writes a 25-cell Board whose centre is the marked `FREE_TEXT` ("Complain about Circuit Music"), excluding the Free Space item from the sampled Prompts.
- `joinAndDeal` guard propagation — with no Board and a pool of `< MIN_POOL` active Prompts, the deal throws and no Board doc is committed (the api-layer half of the "not a blank Board" criterion).
- `joinAndDeal` Player-row attribution — a saved profile with a custom name is written to the Player row instead of the Google name, its `photoURL` wins only under `customPhoto: true` (a stale non-custom profile photo loses to the live auth photo), a missing profile doc or a failed profile read falls back to the auth identity without blocking the deal, and malformed saved fields are ignored per-field in favour of the auth values: non-string, whitespace-only, and over-100-char names; non-string photos, non-`https` schemes (`http://` downgrade), and unparseable photo strings; truthy-junk `customPhoto` values (`'false'`, `1`) never publish the saved photo.

`src/hooks/useData.test.ts` (Vitest, firestore-mocked unit) — proves, against the real hooks: the `useItems` `enabled` gate (subscribes once by default and when re-enabled, zero `onSnapshot` calls while disabled) and the `hasServerData` latch (subscriptions pass `includeMetadataChanges: true`; the flag is false over a cache-only snapshot even though `loading` clears, latches on the first server-confirmed snapshot for both the collection and the doc sub, and stays latched through a later cache-served snapshot).

The pure `dealBoard` contract — deterministic sampling, 24 unique Prompts + null centre, and the `< 24` throw — is unit-tested in `src/game/logic.test.ts` (owned by the test-harness ticket) and not duplicated here.

The Player-facing sign-in-time surfacing of a failed deal — `dealError`/`dealing`/`retryDeal` state and App mounting `DealError` as the Card route — is wired in the auth/App join effect and owned by the auth ticket (`specs/w1-auth-google.md`). This spec owns the Board-side render guard that keeps a thin pool from showing as a blank card. Deal-failure recovery is manual (Retry button + the reachable `/items` tab); the automatic pool-recovery watcher was removed by the PR #66 human tiebreak and is a tracked follow-up.
