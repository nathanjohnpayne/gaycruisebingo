# Prompt pool reseed + stratified Board composition (`seed-and-composition`)

The Prompt pool is replaced with a canonical 87-entry set, and Board dealing changes from uniform-random sampling to a stratified spicy/tame sample, so a dealt card mixes both categories instead of possibly landing all one kind of joke. "Spicy" is the 🔞-tagged category, stored as a `spicy: boolean` on every Prompt — the emoji is a category tag applied at authoring/seeding time, never present in the Prompt's own display text. This spec supersedes `w1-event-seed`'s narrower ~30–50 "dense pool" band (that spec's other claims — the ADR 0001 Claim Mode naming, the ADR 0004 `blackoutEnabled` delete sentinel, the `ADMIN_UID` roster flow — are unchanged and still covered there).

## Contract

- `src/data/seed.ts` — `SEED_ITEMS: { text: string; spicy: boolean }[]` holds the canonical 87-entry pool (24 spicy, 63 tame), replacing the old 32-entry `string[]`. `FREE_TEXT` is `'Complain about circuit music'` (lowercased "circuit music", was "Circuit Music").
- `src/types.ts` — `ItemDoc.spicy: boolean` records the category on every Prompt document. `EventDoc['settings'].spicyRatio?: number` is the optional per-Event override of the target spicy share; it is read defensively (`typeof === 'number'`) at the one call site that needs it, the same way `settings.reportHideThreshold` already is, even though both are typed on `EventDoc`. Absent on an Event doc seeded before this field existed, in which case `dealBoard`'s own default applies.
- `src/game/logic.ts` — `DealItem` carries `spicy: boolean`. `dealBoard(pool, freeText, seed, spicyRatio = 0.4)` deals the frozen 5×5 Board via a stratified sample: it shuffles the spicy and tame subsets of `pool` independently (both with the same seeded PRNG instance, so the whole deal stays deterministic per seed), takes `round(24 * spicyRatio)` from spicy and the remainder from tame, backfills from whichever category has more remaining when the other runs short, and distributes the smaller selected category evenly across the chosen 24 so spicy and tame interleave rather than cluster together. The pre-existing `pool.length < MIN_POOL` (24) guard fires before any stratification and is unaffected by the ratio or the backfill.
- `src/data/api.ts` — `joinAndDeal` maps the fetched pool to `DealItem[]` including `spicy`, reads `settings.spicyRatio` off the same already-fetched event doc as `reportHideThreshold`, and passes it (or the 0.4 default when absent) as `dealBoard`'s 4th argument. `addItem(uid, text, spicy = false)` writes the `spicy` flag on a user-submitted Prompt; a Prompt added without checking the 🔞 toggle defaults to tame.
- `src/components/ItemPool.tsx` — a 🔞 checkbox next to the add-Prompt input lets a Player mark their submission as spicy; it resets to unchecked after a successful add, alongside the text field.
- `scripts/seed.mjs` — `EVENT_SEED.settings` carries both `reportHideThreshold: 4` and `spicyRatio: 0.4`. `ITEMS` holds the same 87-entry `{ text, spicy }` pool as `SEED_ITEMS`. The `seed()` function now replaces the seed-owned pool rather than appending: it deletes existing `events/{id}/items` docs whose `createdBy` is `'seed'` in the same batch that writes the 87 current seed prompts, so re-running the script never leaves stale seed Prompts from a prior pool mixed in with the current one and never erases live Player submissions. Doc ids stay content-hashed on `text` alone (unchanged from before), and each written doc now also carries `spicy`.

## Deploying a pool change (the drift check)

The player-facing pool is the Firestore `events/{id}/items` collection, not the JS bundle — the app renders whatever that collection holds. `SEED_ITEMS` / `ITEMS` are the canonical *source*, written to Firestore only when `scripts/seed.mjs` runs against the live project. Building and deploying the frontend (`npm run deploy:hosting`) ships client changes (e.g. the 🔞 toggle) but does **not** touch the pool, so a pool change that is merged and deployed without a reseed leaves players on the old pool. This is what reopened #129: the 87-entry pool merged and the bundle deployed, but the reseed step was skipped, so players kept seeing the pre-#135 32-entry pool.

- `scripts/seed.mjs` exports `verifySeedPool(existingDocs, pool = ITEMS)`, a pure drift check: it compares the live seed-owned docs (`createdBy === 'seed'`) against the canonical pool and returns `{ ok, expected, seedOwned, playerOwned, missing, mismatched, stale }`. `missing` = canonical prompts absent from the live pool; `stale` = seed-owned docs the canonical pool no longer contains; `mismatched` = same prompt with drifted seed-owned fields (`text`, strict-boolean `spicy`, `isFreeSpace: false`, `status: 'active'`, or a `reportCount` at/above the event's visibility threshold). Sub-threshold reports remain in the player-facing pool and do not count as drift. Player-submitted docs are ignored — they are never counted as drift.
- `node scripts/seed.mjs --verify` (npm: `verify:seed`) runs `verifySeedPool` against the live `events/{id}/items` read-only and exits 0 when in sync, 1 with an actionable report when it drifts. The npm command explicitly targets the production `gaycruisebingo` project; direct script invocations may select another project/event with the documented environment variables. It is the post-deploy smoke test for any pool change, and would have flagged the #129 gap immediately.
- `seed()` self-verifies after committing: it reads the collection back and runs `verifySeedPool`, exiting non-zero if a green-looking seed run somehow left drift (partial batch, wrong project, a stale doc a scoped delete missed).

## Acceptance criteria

- **Given** the seed script runs on a fresh Event, **when** it completes, **then** `events/{id}/items` holds exactly the 87 active seed Prompts with correct `spicy` flags, and the Free Space is not among them. **Given** the script re-runs later, **then** it replaces the seed-owned pool rather than duplicating it, while preserving Player-submitted Prompts in the same collection. (Tests: "exports the same 87-entry { text, spicy } pool as src/data/seed.ts SEED_ITEMS"; rules-emulator tests: "fresh seed writes exactly 87 active seed prompts with boolean spicy flags"; "running twice replaces seed-owned prompts without duplicating or deleting player submissions".)
- **Given** the full pool, **when** a Board is dealt, **then** it has 24 non-free Squares (10 spicy / 14 tame at the default 0.4 ratio) plus the marked Free Space center, deterministic per seed, with the spicy Squares interleaved (not clustered) across the grid. (Tests: "deals 25 cells: a marked free center + 24 non-free"; "deals exactly 10 spicy / 14 tame among the 24 non-free cells at the default ratio"; "is deterministic per seed and varies across seeds"; "interleaves spicy cells across at least 3 different grid rows (not clustered)".)
- **Given** a category is short, **then** backfill keeps the deal at 24 total; **given** the active non-free pool is below `MIN_POOL` (24), **then** `dealBoard` throws exactly like the pre-existing guard. (Tests: "backfills tame when spicy is short: 5 spicy + 40 tame -> 5 spicy + 19 tame, no throw"; "throws when the active pool has fewer than MIN_POOL (24) prompts, same as the pre-existing guard".)

## Test coverage

`src/data/seed-and-composition.test.ts` (Vitest, pure-logic unit — no Firebase, no emulator):

- Pool shape — `SEED_ITEMS` has exactly 87 entries (24 spicy, 63 tame), no duplicate `text`, no entry's text contains the 🔞 glyph, and `FREE_TEXT` is the lowercased value.
- `dealBoard` composition — the full 87-entry pool mapped to synthetic `DealItem[]`s deals 25 cells (marked free center + 24 non-free), exactly 10 spicy / 14 tame at the default ratio (looked up via a `Map<id, spicy>` built from the input pool, since `Cell` itself does not carry `spicy`), is deterministic for a repeated seed and differs for a different seed, and spans at least 3 distinct grid rows among its spicy cells for a fixed seed plus a regression seed that previously clustered all spicy cells into two rows.
- Backfill — a synthetic 5-spicy/40-tame pool deals 24 non-free cells as exactly 5 spicy + 19 tame with no throw; a 20-item pool (below `MIN_POOL`) throws the same guard message as before this ticket.
- Ratio config — the full pool dealt with `spicyRatio = 0.25` yields 6 spicy / 18 tame among the 24 non-free cells.
- `scripts/seed.mjs` replace-semantics payload — its exported `ITEMS` constant matches `SEED_ITEMS` content exactly, and `EVENT_SEED.settings.spicyRatio` is `0.4`. The rules-emulator test layer applies the same exported seed mutation helper twice and asserts a fresh Event has exactly 87 active seed prompts, stale seed-owned docs are removed on rerun, and Player-submitted Prompts in the same collection survive.
- `verifySeedPool` drift check (#129 reopened) — `src/test/w1-event-seed.test.ts` pins its verdicts as pure-function unit tests: `ok` for an in-sync live pool, player-submitted docs ignored, and each drift axis flagged (a canonical prompt never seeded → `missing`; a retired seed doc still present → `stale`; any drift in canonical content or player-facing visibility fields → `mismatched`). The reconcile-report test also pins the same-target command and explicitly cleared `ADMIN_UID`. One test reproduces the exact #129 production condition — an un-reseeded old pool — and asserts it drifts on both axes (86 `missing` + 3 `stale`). The rules-emulator layer runs the check end-to-end against Firestore: a fresh seed verifies `ok`, and a simulated un-reseeded old pool verifies `ok: false` and is then reconciled to `ok` by running the seed.

The pre-existing `dealBoard` contract (25 cells, free center, the `< MIN_POOL` throw) stays pinned in `src/game/logic.test.ts`, whose fixture pool is now all-tame (`spicy: false` on every entry) — stratified sampling backfills entirely from tame in that case, so its existing assertions (24 unique ids, determinism, the throw) hold unchanged. `w1-event-seed.md` keeps the remaining `scripts/seed.mjs` claims this spec does not touch (Claim Mode naming, the `blackoutEnabled` delete sentinel, the `ADMIN_UID` roster flow) and its own "prompt pool density" section now points at the 87-entry pool this spec establishes rather than the old ~30–50 band.
