# E2E happy-path (join ‚Üí Mark ‚Üí BINGO ‚Üí Leaderboard) + offline-Mark test against the emulator

Proves the PRD's headline metric ‚Äî a full round is playable end-to-end with zero coordination beyond the app's one shared link ‚Äî with a Playwright e2e suite against the Firebase Local Emulator Suite, and adds the [ADR 0006](../docs/adr/0006-offline-resilience.md) assertion that an offline Mark survives a reload (queued durably in IndexedDB) and syncs on reconnect. Runs on the Playwright harness `w0-test-harness` stood up (`playwright.config.ts`, `test:e2e`) and consumes the `firebase.json` `emulators` block that same ticket added.

## The suite

- `playwright.config.ts` ‚Äî extends the `w0-test-harness` smoke config with the app `webServer` + `baseURL` wiring it always said this ticket would add: a `vite build --mode e2e` + `vite preview` of the app (see "Emulator wiring" below for why a real build), never reusing an existing listener on the dedicated e2e port so the served bundle is always the one this run just built. The Firestore/Auth/Storage emulators are NOT a `webServer` entry: `npm run test:e2e` runs `scripts/test-e2e.sh`, which wraps `playwright test` in `firebase emulators:exec --only auth,firestore,storage --project demo-gaycruisebingo-e2e` (mirroring `test:rules`), only starting Playwright once every emulator is ready ‚Äî and shell-quotes any forwarded CLI arguments into the exec script, so a targeted local run like `npm run test:e2e -- tests/e2e/x-e2e-happy-path.spec.ts --grep "<title>"` reaches Playwright intact. The explicit `--project` pins the emulator to the SAME project id the browser bundle and seed helper use (`env.ts` `PROJECT_ID`), rather than the `.firebaserc` default (`gaycruisebingo`, a real project); a mismatch would evaluate the Auth-backed Firestore rules under a different project than the signed-in app writes as. So `npm run test:e2e` is the emulator-backed, self-contained entry point; a bare `npx playwright test` does not boot emulators and requires them to already be running (e.g. under `npm run emulator`), with the project id set.
- `tests/e2e/support/env.ts` ‚Äî the shared demo project id (`demo-gaycruisebingo-e2e`), a dedicated Event id (`e2e-happy-path`, never the real `med-2026`), the emulator ports `firebase.json` pins, and the dev-server URL ‚Äî imported by both the config and the test support so they cannot drift apart.
- `tests/e2e/support/seed.ts` ‚Äî CLEARS the Firestore emulator (`testEnv.clearFirestore()`, the `tests/rules` suites' pattern), then seeds the Event + the full `ITEMS` pool (32 Prompts, `scripts/seed.mjs` ‚Äî reused, not re-declared, so this suite deals boards from the exact real pool shape), with security rules disabled via `@firebase/rules-unit-testing` (already a devDependency the `tests/rules/` layer relies on) ‚Äî the same rules-bypassing posture `scripts/seed.mjs` gets from the Admin SDK, without adding `firebase-admin` as a dependency (it is deliberately absent from the app install; see that script's header comment). The wipe-before-seed makes every attempt start from a clean slate: players/boards rows from a failed attempt would otherwise persist in the same `emulators:exec` process, and a Playwright CI retry (fresh worker, so `beforeAll` re-seeds) would join a second generated user and permanently break the happy-path case's sole-Player rows assertion. The Auth emulator DOES accumulate one autogenerated account per join attempt and is deliberately left alone ‚Äî no assertion counts auth users (the board observer is uid-scoped and the leaderboard count reads the just-reset Firestore players rows), so auth accumulation is harmless.
- `tests/e2e/support/join.ts` ‚Äî the zero-coordination join: land on `/` (the shared link), accept the 18+ acknowledgement, click "Continue with Google", and drive the Firebase Auth Emulator's account-chooser popup that `signInWithPopup` opens (add a new auto-generated account and submit ‚Äî no real Google OAuth). Shared by both cases below so the join is asserted in exactly one place.
- `tests/e2e/support/board.ts` ‚Äî selects Squares by their dealt prompt TEXT (`src/components/Board.tsx` renders no `data-testid`/role on a cell), never CSS position, so the suite survives the Board gaining more per-cell chrome (the doubt affordance, tally badges) around the same 25-cell grid. Also `waitForBoardServerConfirmed` ‚Äî blocks on the `.grid[data-server-confirmed="true"]` latch (the one test-only affordance added to `Board.tsx`, mirroring `useBoard`'s `hasServerData`) so the winning line is only marked once the board is server-backed.
- `tests/e2e/x-e2e-happy-path.spec.ts` ‚Äî the two cases.

## Acceptance criteria ‚Üí test cases

- **Given** a seeded Event (dense pool) and the shared link, **when** a Player joins, Marks a winning line, and opens Ranks, **then** BINGO fires and the Player ranks correctly ‚Äî with no coordination beyond the link. ‚Üí `join -> Mark -> BINGO -> Leaderboard completes with only the shared link (zero admin action)`: joins, waits for the board's first SERVER-backed snapshot (the `data-server-confirmed="true"` latch `Board.tsx` sets from `useBoard`'s `hasServerData` ‚Äî marking the line while the board is still cache-only would leave it already in BINGO state when the Celebration baseline initializes, swallowing the animated edge and flaking the assertion), then taps the 4 non-free Squares of the middle row (index 12, the free centre, already counts ‚Äî the AC's "centre free space counts"), asserts the `BINGO!` celebration, opens Ranks, and asserts the sole Player's row shows rank 1 with `1 bingo` / `4 squares`. The test never navigates to `/admin` or calls anything in `src/components/Admin.tsx`.
- **Given** a Player offline, **when** they Mark a Square and reload, **then** the app cold-boots offline and RENDERS the cached Mark, and the Mark later syncs on reconnect (ADR 0006). ‚Üí `a Mark made offline survives a reload and syncs on reconnect (ADR 0006)`: first waits until the emulator shows this User's persisted 18+ attestation ‚Äî `signIn()` commits it via a Firestore transaction *after* the popup, transactions never queue offline, and the shell renders before it lands, so cutting the network earlier could kill it mid-flight and strand the User on the #23 re-prompt gate at the reload ‚Äî then `context.setOffline(true)`, Marks an arbitrary unmarked non-free Square, asserts it renders `marked` immediately (latency compensation), then reloads while STILL offline. The reload wipes memory and re-initializes the whole app from scratch, so anything that survives came from the durable IndexedDB queue ‚Äî the SW (real build, below) serves the shell + JS offline, so the Firestore SDK re-initializes and recovers the pending Mark. The suite then asserts the OFFLINE-reloaded page COLD-BOOTS and renders that recovered Mark (the Primary tab bar plus the `marked` cell), pinning the #115 offline cold-boot fix: `AuthContext` publishes the cached auth User and settles `loading: false` without awaiting the network-bound bootstrap, and settles the 18+ gate from the cached attestation ‚Äî so the cached Board paints offline instead of stalling on "Loading‚Ä¶". It then reconnects and polls an independent rules-disabled Firestore read of THIS Player's own board ‚Äî `boards/{uid}`, the uid captured from the page's Firebase Auth IndexedDB persistence right after join, never the reloaded tab's own cache and never an any-board scan (which could false-pass on a prompt-text collision with the happy-path Player's board in the same shared Event) ‚Äî until the emulator shows the Mark: since the Mark was made offline and never synced before the reload, its only path to the emulator is the durable queue surviving the reload and draining on reconnect ‚Äî the rigorous "not memory" proof, matching `tests/offline/w0-offline-persistence.test.ts`'s ordering discipline. The offline-render assertion is the end-to-end UI proof (the durable Mark reaches the UI too, not just storage); the emulator observer is the durability proof ‚Äî no fresh online reload needed, because the app cold-boots offline directly (#115).
- Dense pool ‚â• 24 active Prompts (ADR 0004's `dealBoard` guard, `src/game/logic.ts` `MIN_POOL`). ‚Üí asserted directly against the seeded `ITEMS` length in `beforeAll`.
- Zero-coordination (shared-link-only) path. ‚Üí structural: neither case performs any admin action or reads `Admin.tsx`; `join.ts` is the only entry point either case uses.

## Emulator wiring (delivered)

The originally-anticipated blocker (`@playwright/test` / the e2e harness itself) never applied: `package.json` already carries `@playwright/test` and `test:e2e` (`w0-test-harness` landed it). The one real gap was that `src/firebase.ts` exported its `auth`/`db`/`storage` singletons wired to production Firebase only ‚Äî no file under `src/**` called `connectAuthEmulator` / `connectFirestoreEmulator` / `connectStorageEmulator`, so the browser-served app could not reach the Local Emulator Suite this suite seeds. Every existing emulator-connected test (`tests/offline/**`, `tests/rules/**`) sidesteps this by constructing its OWN separate Firebase app instance directly in Node; none makes the real browser-served app talk to the emulator, and no in-scope test-only workaround could (Playwright cannot drive a real Google OAuth popup, and `firebase/auth` ships no `__FIREBASE_DEFAULTS__`/env shim that would redirect the singleton to the emulator from outside `src/`). An independent verifier confirmed the two cases failed at exactly `joinViaSharedLink`'s final assertion for this reason and nowhere else.

That branch is now delivered in `src/firebase.ts`, gated on `import.meta.env.MODE === 'e2e' && import.meta.env.VITE_FIREBASE_PROJECT_ID?.startsWith('demo-')`. It calls `connectAuthEmulator`/`connectFirestoreEmulator`/`connectStorageEmulator` against the ports `firebase.json` pins (auth 9099, firestore 8080, storage 9199). The `demo-` project-id check is belt-and-suspenders (the same emulator-only convention `tests/offline` and `tests/rules` use); `MODE === 'e2e'` is the load-bearing gate.

**Why `MODE === 'e2e'` and not `import.meta.env.DEV`.** The offline case reloads the page *while offline*, which only resolves when a precaching service worker serves the app shell ‚Äî and vite-plugin-pwa emits that SW only for a build, never for `vite dev`. So `playwright.config.ts`'s app webServer serves a real `vite build --mode e2e` + `vite preview`, where `import.meta.env.DEV` is `false`. `MODE` is a built-in Vite env var (unlike an unset `VITE_*` flag, which Vite does not statically replace, so it would NOT dead-code-eliminate), statically substituted at build time: `--mode e2e` bakes in `'e2e'`, so the branch is kept and runs; the real production build (`npm run build`, `MODE === 'production'`) folds the gate to `'production' === 'e2e'` ‚Üí `false` and drops the whole branch. `join.ts` then drives the Auth Emulator's account-chooser popup that `signInWithPopup` opens (add a new auto-generated account, submit ‚Äî no real Google OAuth; a brief retry absorbs the widget's handler-attachment delay), and both cases pass. The suite runs serially (`workers: 1`) because both cases join the same seeded Event and the happy-path case asserts it is the sole leaderboard Player.

**Production is unaffected.** Because `MODE === 'production'` in `npm run build`, `'production' === 'e2e'` folds to `false` and `false && ‚Ä¶` dead-code-eliminates the whole branch ‚Äî the shipped `dist/` bundle carries no `connectAuthEmulator`/`connectFirestoreEmulator`/`connectStorageEmulator` call and no emulator host string (`127.0.0.1:9099`/`:8080`/`:9199`), verified by grepping the built bundle after `npm run build`. `src/firebase.test.ts` (the ADR 0006 source guard) stays green: under `npm test` (Vitest, `MODE === 'test'`) `'test' === 'e2e'` is `false`, so the branch never runs.

**The e2e build is hermetic.** `playwright.config.ts`'s app webServer `env` explicitly BLANKS every optional analytics / App Check var ‚Äî `VITE_FIREBASE_MEASUREMENT_ID`, `VITE_POSTHOG_KEY`, `VITE_POSTHOG_HOST`, `VITE_RECAPTCHA_SITE_KEY` ‚Äî so an e2e build on a real developer checkout can never inherit a live key from `.env.local` or the parent env and initialize real GA4 / PostHog / App Check (which would emit synthetic login/mark/bingo/pageview events or load reCAPTCHA). An already-set (even empty) process-env var wins over `.env*` in Vite, and each initializer is gated on a truthy value, so blank means off ‚Äî confirmed by grepping the built e2e bundle for those keys.

`npm run test:e2e` (Playwright, real Chromium) remains a **local-only** layer ‚Äî `.github/workflows/app-ci.yml` says so explicitly ("intentionally not run here") ‚Äî so it does not run in `app-ci`; `tests/e2e/smoke.spec.ts` (`w0-test-harness`) still proves the runner launches Chromium independently.
